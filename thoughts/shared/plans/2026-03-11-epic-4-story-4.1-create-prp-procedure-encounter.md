# Epic 4 Story 4.1 — Create PRP Procedure Encounter: Implementation Plan

## Overview

Add the ability to record a PRP procedure encounter for a case. This covers the encounter-level data from Story 4.1: procedure date, injection site, laterality, ICD-10 diagnoses, consent, pain rating, procedure number in series, and vital signs. The existing `procedures` table will be extended with PRP-specific columns. Vital signs will live in a new `vital_signs` table linked to the procedure record for reuse across visit types.

---

## Current State Analysis

- `procedures` table exists at `supabase/migrations/002_case_dashboard_tables.sql:27-41` with only generic fields: `procedure_date`, `procedure_name`, `cpt_code`, `provider_id`, `notes`, `charge_amount`.
- `src/actions/procedures.ts` — read-only: `listProcedures`, `getProcedureCount`. No create/update.
- `src/components/procedures/procedure-table.tsx` — table display only. "Record Procedure" button is disabled with a "Coming Soon" tooltip.
- `src/app/(dashboard)/patients/[caseId]/procedures/page.tsx` — minimal page, just renders `ProcedureTable`.
- No `vital_signs` table exists anywhere in the schema.
- ICD-10 diagnoses live in PM extraction `diagnoses` JSONB arrays (`icd10_code` + `description` per item) — this is the source for the multi-select.
- Prior PRP encounter linking will be derived by querying the previous procedure for this case (ordered by `procedure_date` descending), not a stored FK.

## Desired End State

Provider can click "Record Procedure" on the Procedures page, fill out a dialog/form with all Story 4.1 fields, submit, and see the new procedure appear in the table. Vital signs are captured alongside and stored in a linked `vital_signs` row.

### Verification:
- Submitting the form creates a row in `procedures` (with PRP-specific columns populated) and a linked row in `vital_signs`.
- The procedure number in series is displayed correctly (e.g., "1st", "2nd").
- ICD-10 field allows selecting from PM extraction diagnoses AND free-text entry.
- The table reflects all new fields.

## What We're NOT Doing

- Story 4.2 fields (PRP prep, anesthesia, injection details) — separate story.
- Story 4.3 note generation — separate story.
- Vital signs for initial visit — the table is designed for it, but the initial visit UI hookup is out of scope here.
- Edit or delete of existing procedures.
- CPT code or billing fields — those exist already and are not PRP-specific.

---

## Implementation Approach

1. Migrate: extend `procedures` + create `vital_signs`
2. Server actions: `createPrpProcedure` (creates both rows in a transaction-like sequence)
3. Validation schema: Zod schema for the form
4. UI: Dialog with form triggered from the existing "Record Procedure" button

Form approach: single-step dialog (not a wizard) — the field count is manageable (~15 fields across two sections: Encounter + Vital Signs). Uses the standard `react-hook-form` + `zodResolver` pattern.

ICD-10 multi-select: a custom combobox that (a) fetches approved PM extraction diagnoses for the case and (b) allows free-text entry if the code isn't in the list. Stored as `jsonb` on `procedures`.

## Phase 1: Database Migration

### Overview
Extend `procedures` with PRP-specific columns. Create `vital_signs` table.

### Changes Required:

#### 1. New migration file
**File**: `supabase/migrations/013_prp_procedure_encounter.sql`

```sql
-- ============================================
-- EXTEND PROCEDURES TABLE WITH PRP FIELDS
-- ============================================
alter table public.procedures
  add column injection_site       text,
  add column laterality           text check (laterality in ('left', 'right', 'bilateral')),
  add column diagnoses            jsonb not null default '[]',
  add column consent_obtained     boolean,
  add column pain_rating          integer check (pain_rating >= 0 and pain_rating <= 10),
  add column procedure_number     integer;  -- 1st, 2nd, 3rd injection in series

-- GIN index for diagnoses search
create index idx_procedures_diagnoses on public.procedures using gin(diagnoses);

-- ============================================
-- VITAL SIGNS TABLE
-- ============================================
create table public.vital_signs (
  id                    uuid primary key default gen_random_uuid(),
  case_id               uuid not null references public.cases(id),
  procedure_id          uuid references public.procedures(id),
  recorded_at           timestamptz not null default now(),

  -- Vitals
  bp_systolic           integer,
  bp_diastolic          integer,
  heart_rate            integer,
  respiratory_rate      integer,
  temperature_f         numeric(4,1),
  spo2_percent          integer check (spo2_percent >= 0 and spo2_percent <= 100),

  -- Audit
  created_at            timestamptz not null default now(),
  updated_at            timestamptz not null default now(),
  deleted_at            timestamptz,
  created_by_user_id    uuid references public.users(id),
  updated_by_user_id    uuid references public.users(id)
);

create index idx_vital_signs_case_id on public.vital_signs(case_id);
create index idx_vital_signs_procedure_id on public.vital_signs(procedure_id);

create trigger set_updated_at before update on public.vital_signs
  for each row execute function update_updated_at();

alter table public.vital_signs enable row level security;

create policy "Authenticated users full access" on public.vital_signs
  for all using (auth.role() = 'authenticated');
```

```ts
const diagnosisSchema = z.object({
  icd10_code: z.string().min(1, 'ICD-10 code is required'),
  description: z.string().min(1, 'Description is required'),
})

const vitalSignsSchema = z.object({
  bp_systolic: z.number().int().min(1).max(300).nullable(),
  bp_diastolic: z.number().int().min(1).max(200).nullable(),
  heart_rate: z.number().int().min(1).max(300).nullable(),
  respiratory_rate: z.number().int().min(1).max(60).nullable(),
  temperature_f: z.number().min(90).max(110).nullable(),
  spo2_percent: z.number().int().min(0).max(100).nullable(),
})

export const prpProcedureFormSchema = z.object({
  procedure_date: z.string().min(1, 'Procedure date is required'),
  injection_site: z.string().min(1, 'Injection site is required'),
  laterality: z.enum(['left', 'right', 'bilateral'], {
    required_error: 'Laterality is required',
  }),
  diagnoses: z.array(diagnosisSchema).min(1, 'At least one diagnosis is required'),
  consent_obtained: z.boolean(),
  pain_rating: z.number().int().min(0).max(10).nullable(),
  vital_signs: vitalSignsSchema,
})

export type PrpProcedureFormValues = z.infer<typeof prpProcedureFormSchema>
export type PrpDiagnosis = z.infer<typeof diagnosisSchema>
export type PrpVitalSigns = z.infer<typeof vitalSignsSchema>
```

## Phase 3: Server Actions ✓

### Overview
Add `createPrpProcedure` server action. Also add a helper to fetch the case's approved PM diagnoses for the ICD-10 combobox, and a helper to get the prior procedure for the case.

### Changes Required:

#### 1. Extend `src/actions/procedures.ts`

```ts
'use server'

import { createClient } from '@/lib/supabase/server'
import { revalidatePath } from 'next/cache'
import type { PrpProcedureFormValues } from '@/lib/validations/prp-procedure'

// ... existing listProcedures, getProcedureCount ...

export async function createPrpProcedure(
  caseId: string,
  values: PrpProcedureFormValues
) {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return { error: 'Not authenticated' }

  // Derive procedure number in series
  const { count } = await supabase
    .from('procedures')
    .select('*', { count: 'exact', head: true })
    .eq('case_id', caseId)
    .is('deleted_at', null)

  const procedureNumber = (count ?? 0) + 1

  // Insert procedure record
  const { data: procedure, error: procError } = await supabase
    .from('procedures')
    .insert({
      case_id: caseId,
      procedure_date: values.procedure_date,
      procedure_name: 'PRP Injection',
      injection_site: values.injection_site,
      laterality: values.laterality,
      diagnoses: values.diagnoses,
      consent_obtained: values.consent_obtained,
      pain_rating: values.pain_rating,
      procedure_number: procedureNumber,
      created_by_user_id: user.id,
      updated_by_user_id: user.id,
    })
    .select()
    .single()

  if (procError || !procedure) return { error: procError?.message ?? 'Failed to create procedure' }

  // Insert vital signs record
  const vs = values.vital_signs
  const hasVitals = Object.values(vs).some((v) => v !== null && v !== undefined)
  if (hasVitals) {
    const { error: vsError } = await supabase
      .from('vital_signs')
      .insert({
        case_id: caseId,
        procedure_id: procedure.id,
        bp_systolic: vs.bp_systolic,
        bp_diastolic: vs.bp_diastolic,
        heart_rate: vs.heart_rate,
        respiratory_rate: vs.respiratory_rate,
        temperature_f: vs.temperature_f,
        spo2_percent: vs.spo2_percent,
        created_by_user_id: user.id,
        updated_by_user_id: user.id,
      })

    if (vsError) return { error: vsError.message }
  }

  revalidatePath(`/patients/${caseId}/procedures`)
  return { data: procedure }
}

// Fetch approved PM diagnoses for this case (ICD-10 combobox source)
export async function getCaseDiagnoses(caseId: string) {
  const supabase = await createClient()

  const { data, error } = await supabase
    .from('pain_management_extractions')
    .select('diagnoses')
    .eq('case_id', caseId)
    .eq('review_status', 'approved')
    .is('deleted_at', null)
    .order('created_at', { ascending: false })
    .limit(1)
    .single()

  if (error || !data) return { data: [] }

  const diagnoses = Array.isArray(data.diagnoses) ? data.diagnoses : []
  return {
    data: diagnoses as Array<{ icd10_code: string | null; description: string }>,
  }
}

// Get the most recent prior PRP procedure for comparison
export async function getPriorPrpProcedure(caseId: string) {
  const supabase = await createClient()

  const { data, error } = await supabase
    .from('procedures')
    .select('id, procedure_date, pain_rating, procedure_number')
    .eq('case_id', caseId)
    .is('deleted_at', null)
    .order('procedure_date', { ascending: false })
    .limit(1)
    .single()

  if (error || !data) return { data: null }
  return { data }
}
```

## Phase 4: Record Procedure Dialog + Form ✓

### Overview
Build the "Record Procedure" dialog triggered from the existing disabled button in `ProcedureTable`. The dialog has two sections: Encounter Details and Vital Signs. ICD-10 uses a combobox with PM extraction diagnoses pre-loaded, plus free-text add.

### Changes Required:

#### 1. New ICD-10 combobox component
**File**: `src/components/procedures/diagnosis-combobox.tsx`

A controlled multi-select combobox:
- Props: `value: PrpDiagnosis[]`, `onChange: (v: PrpDiagnosis[]) => void`, `suggestions: Array<{ icd10_code: string | null; description: string }>`
- Shows selected diagnoses as removable badges
- Dropdown lists PM extraction suggestions (filtered by typed text)
- If typed text matches nothing, shows "Add [code] — [description]" option that appends a free-text entry
- Free-text input accepts `[ICD10_CODE] description` format or just description

Pattern reference: Similar to the tag-style multi-select in `pt-extraction-form.tsx` but simpler since it's a controlled dialog field, not a useFieldArray context.

#### 2. New record procedure dialog
**File**: `src/components/procedures/record-procedure-dialog.tsx`

Form sections:
- **Encounter Details**: procedure_date (date input), injection_site (text), laterality (Select: Left/Right/Bilateral), diagnoses (DiagnosisCombobox), consent_obtained (Checkbox), pain_rating (number input 0–10)
- **Vital Signs**: bp_systolic + bp_diastolic (two side-by-side inputs), heart_rate, respiratory_rate, temperature_f, spo2_percent — all optional

Default values: all nullable vitals default to `null`; `consent_obtained` defaults to `false`; `procedure_date` defaults to today.

On submit: calls `createPrpProcedure(caseId, values)`. On success: closes dialog, shows `toast.success`. On error: `toast.error`.
