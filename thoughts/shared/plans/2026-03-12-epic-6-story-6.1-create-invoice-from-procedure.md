# Story 6.1 — Create Invoice from Procedure

## Overview

Implement the ability to create medical invoices from the Billing tab of a patient case. Two invoice types are supported: **Medical Invoice** (visit-level, multiple line items per service — initial exam, MRI review, PRP procedures, discharge) and **Medical Facility Invoice** (PRP procedures only, each described as "Medical site utilization" with same CPT codes 0232T/86999/76942). Invoices are auto-numbered, pre-populated with case/procedure data, and editable before saving.

## Current State Analysis

### What Exists:
- **DB schema**: `invoices`, `invoice_line_items`, `payments` tables in `002_case_dashboard_tables.sql`
- **Billing page**: read-only at `src/app/(dashboard)/patients/[caseId]/billing/page.tsx`
- **BillingTable**: sortable table with disabled "Create Invoice" button (`src/components/billing/billing-table.tsx`)
- **BillingSummaryCard**: displays total billed/paid/balance (`src/components/billing/billing-summary-card.tsx`)
- **Server actions**: `listInvoices` and `getBillingSummary` only (`src/actions/billing.ts`)
- **Cases table**: has `total_billed`, `total_paid`, `balance_due` columns (`001_initial_schema.sql:86-88`)
- **Procedures table**: has `charge_amount`, `cpt_code`, `diagnoses` (jsonb), `injection_site`, `laterality` columns
- **Clinic settings**: clinic name, address, phone, fax, logo (`007_clinic_provider_settings.sql`)
- **Provider profiles**: display_name, credentials, NPI (`007_clinic_provider_settings.sql`)

### What's Missing:
- No `invoice_type` column on invoices
- No `service_date` on `invoice_line_items` (needed — line items span different dates)
- No invoice number auto-generation (column exists but no trigger/sequence)
- No indication, claim_type, or diagnoses snapshot on invoices
- No payee fields on invoices
- No create/update/delete server actions for invoices
- No Zod validation schemas for invoices
- No invoice creation form/dialog UI
- No invoice detail view

### Key Discoveries:
- Diagnoses come from two sources: `initial_visit_notes.diagnoses` (free text) and `procedures.diagnoses` (jsonb array of `{icd10_code, description}`) — `src/lib/validations/prp-procedure.ts:3-6`
- Indication uses the same `formatReasonForVisit(accident_type)` helper from `src/lib/constants/clinical-note-header.ts` that Initial Visit Notes and Discharge Notes use — maps `cases.accident_type` to a standardized medical-legal etiology phrase (e.g., "Post-traumatic musculoskeletal pain following motor vehicle collision"). Consistency across all documents in the paper trail is critical for defensible PI paperwork.
- Clinic fax field exists: `008_add_fax_to_clinic_settings.sql`
- Invoice PDFs show: clinic header (logo, address, phone, fax), all info rendered as tables — patient/case info table, diagnoses table (ICD-10 | Description), attorney table (firm name + address), line items table (DATE | CPT | Description | QTY | Amount), total, payee line
- Check is always payable to the business (clinic), not the provider
- Two invoice types have different titles: "Medical Facility Invoice" vs "Medical Invoice"
- Existing patterns: server actions use `safeParse` + `revalidatePath`, forms use `react-hook-form` + `zodResolver`, dialogs use controlled `open`/`onOpenChange` props

## Desired End State

From the Billing tab, users can click "Create Invoice" and:
1. Select invoice type (Medical Invoice or Medical Facility Invoice)
2. See auto-populated patient info, provider info, diagnoses, indication, and attorney info
3. See pre-populated line items from recorded procedures (with CPT codes matching the PDF examples)
4. Edit any field including line items (add/remove/modify)
5. Save the invoice, which auto-generates an invoice number and updates case billing totals
6. View created invoices from the billing table (clickable rows leading to detail view)
7. Edit existing invoices

### Verification:
- Create both invoice types and verify all fields populate correctly
- Verify invoice numbers auto-increment (`INV-YYYY-NNNN`)
- Verify case `total_billed` / `balance_due` update after invoice creation
- Verify line items are editable and totals recalculate
- Verify existing billing table shows new invoices with correct status badges

## What We're NOT Doing

- PDF export (Story 6.3)
- Pricing catalog / admin pricing config (Story 6.2)
- Payment recording (separate feature)
- Invoice status workflow (draft → pending → paid transitions) — invoices save as `draft` for now
- Email/fax sending of invoices
- Multi-case invoices

## Implementation Approach

Build bottom-up: schema migration → server actions → validation → UI components. The invoice creation form is a dialog with two sections: header info (auto-populated, editable) and line items table (pre-populated from procedures, editable). Invoice detail is a new page matching the PDF layout.

  return new;
end;

-- ============================================
-- NOTE: This trigger ONLY fires on the invoices table.
-- The invoices row has `case_id` directly — no join needed.
create or replace function update_case_billing_totals()
returns trigger as $$
declare
  target_case_id uuid;
begin
  if TG_OP = 'DELETE' then
    target_case_id := OLD.case_id;
  else
    target_case_id := NEW.case_id;
  end if;

  -- Recalculate totals from all non-deleted invoices for this case
  update public.cases
  set
    total_billed = coalesce((
      select sum(total_amount) from public.invoices
      where case_id = target_case_id and deleted_at is null
    ), 0),
    total_paid = coalesce((
      select sum(paid_amount) from public.invoices
      where case_id = target_case_id and deleted_at is null
    ), 0),
    balance_due = coalesce((
      select sum(total_amount - paid_amount) from public.invoices
      where case_id = target_case_id and deleted_at is null
    ), 0)
  where id = target_case_id;

  return coalesce(NEW, OLD);
end;
$$ language plpgsql;

-- Trigger on invoice insert/update/delete
create trigger update_billing_totals_on_invoice
  after insert or update or delete on public.invoices
  for each row execute function update_case_billing_totals();

// Fetch all data needed for the invoice creation form
export async function getInvoiceFormData(caseId: string) {
  const supabase = await createClient()

  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return { error: 'Not authenticated', data: null }

  // Parallel fetch: case+patient+attorney, procedures, clinic, provider profile (current user), initial visit note
  // NOTE: Provider profile uses current user ID, NOT assigned_provider_id.
  // This matches the pattern in procedure-notes, discharge-notes, initial-visit-notes actions.
  // This is a single-provider clinic — the logged-in user IS the provider.
  const [caseResult, proceduresResult, clinicResult, providerProfileResult, initialVisitResult] = await Promise.all([
    supabase
      .from('cases')
      .select(`
        *,
        patient:patients(*),
        attorney:attorneys(*)
      `)
      .eq('id', caseId)
      .is('deleted_at', null)
      .single(),
    supabase
      .from('procedures')
      .select('*')
      .eq('case_id', caseId)
      .is('deleted_at', null)
      .order('procedure_date', { ascending: true }),
    supabase
      .from('clinic_settings')
      .select('*')
      .is('deleted_at', null)
      .maybeSingle(),
    supabase
      .from('provider_profiles')
      .select('*')
      .eq('user_id', user.id)
      .is('deleted_at', null)
      .maybeSingle(),
    supabase
      .from('initial_visit_notes')
      .select('chief_complaint, diagnoses, created_at, visit_date')
      .eq('case_id', caseId)
      .is('deleted_at', null)
      .order('created_at', { ascending: false })
      .limit(1)
      .maybeSingle(),
  ])

  if (caseResult.error) return { error: caseResult.error.message, data: null }

  const providerProfile = providerProfileResult.data

  // Derive diagnoses: prefer structured from procedures, fall back to initial visit text
  let diagnoses: Array<{ icd10_code: string | null; description: string }> = []
  const procedures = proceduresResult.data ?? []
  const procedureWithDiagnoses = procedures.find(
    (p: { diagnoses?: unknown }) => Array.isArray(p.diagnoses) && p.diagnoses.length > 0
  )
  if (procedureWithDiagnoses) {
    diagnoses = procedureWithDiagnoses.diagnoses as typeof diagnoses
  }

  // Derive indication: use formatReasonForVisit() — same medical-legal etiology phrase
  // used by Initial Visit Notes and Discharge Notes. Ensures the invoice's indication
  // matches the rest of the chart for defensible PI paperwork.
  const indication = formatReasonForVisit(caseResult.data.accident_type)

  // Build pre-populated line items from procedures
  const prePopulatedLineItems = procedures.map((proc: {
    id: string
    procedure_date: string
    cpt_code: string | null
    procedure_name: string
    injection_site?: string | null
    laterality?: string | null
    charge_amount: number | null
  }) => ({
    procedure_id: proc.id,
    service_date: proc.procedure_date,
    cpt_code: proc.cpt_code ?? '',
    description: proc.procedure_name + (proc.injection_site ? ` — ${proc.injection_site}` : '') + (proc.laterality ? ` (${proc.laterality})` : ''),
    quantity: 1,
    unit_price: Number(proc.charge_amount ?? 0),
    total_price: Number(proc.charge_amount ?? 0),
  }))

  return {
    data: {
      caseData: caseResult.data,
      procedures,
      clinic: clinicResult.data,
      providerProfile,
      diagnoses,
      indication,
      prePopulatedLineItems,
    },
  }
}

// Create a new invoice with line items
export async function createInvoice(caseId: string, values: CreateInvoiceFormValues) {
  const parsed = createInvoiceSchema.safeParse(values)
  if (!parsed.success) return { error: parsed.error.flatten().fieldErrors }

  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return { error: 'Not authenticated' }

  const { line_items, ...invoiceData } = parsed.data

  // Calculate total
  const totalAmount = line_items.reduce((sum, item) => sum + item.total_price, 0)

  // Insert invoice (invoice_number auto-generated by trigger)
  const { data: invoice, error: invoiceError } = await supabase
    .from('invoices')
    .insert({
      case_id: caseId,
      invoice_type: invoiceData.invoice_type,
      invoice_date: invoiceData.invoice_date,
      claim_type: invoiceData.claim_type,
      indication: invoiceData.indication || null,
      diagnoses_snapshot: invoiceData.diagnoses_snapshot,
      payee_name: invoiceData.payee_name || null,
      payee_address: invoiceData.payee_address || null,
      notes: invoiceData.notes || null,
      total_amount: totalAmount,
      status: 'draft',
      created_by_user_id: user.id,
      updated_by_user_id: user.id,
    })
    .select()
    .single()

  if (invoiceError || !invoice) return { error: invoiceError?.message ?? 'Failed to create invoice' }

  // Insert line items
  const lineItemRows = line_items.map((item) => ({
    invoice_id: invoice.id,
    procedure_id: item.procedure_id || null,
    service_date: item.service_date,
    cpt_code: item.cpt_code,
    description: item.description,
    quantity: item.quantity,
    unit_price: item.unit_price,
    total_price: item.total_price,
  }))

  const { error: lineItemsError } = await supabase
    .from('invoice_line_items')
    .insert(lineItemRows)

  if (lineItemsError) return { error: lineItemsError.message }

  revalidatePath(`/patients/${caseId}/billing`)
  return { data: invoice }
}

// Update an existing invoice
export async function updateInvoice(invoiceId: string, caseId: string, values: UpdateInvoiceFormValues) {
  const parsed = updateInvoiceSchema.safeParse(values)
  if (!parsed.success) return { error: parsed.error.flatten().fieldErrors }

  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return { error: 'Not authenticated' }

  const { line_items, ...invoiceData } = parsed.data
  const totalAmount = line_items.reduce((sum, item) => sum + item.total_price, 0)

  const { data: invoice, error: invoiceError } = await supabase
    .from('invoices')
    .update({
      ...invoiceData,
      indication: invoiceData.indication || null,
      payee_name: invoiceData.payee_name || null,
      payee_address: invoiceData.payee_address || null,
      notes: invoiceData.notes || null,
      total_amount: totalAmount,
      updated_by_user_id: user.id,
    })
    .eq('id', invoiceId)
    .select()
    .single()

  if (invoiceError || !invoice) return { error: invoiceError?.message ?? 'Failed to update invoice' }

  // Replace line items: delete existing, insert new
  await supabase
    .from('invoice_line_items')
    .delete()
    .eq('invoice_id', invoiceId)

  const lineItemRows = line_items.map((item) => ({
    invoice_id: invoiceId,
    procedure_id: item.procedure_id || null,
    service_date: item.service_date,
    cpt_code: item.cpt_code,
    description: item.description,
    quantity: item.quantity,
    unit_price: item.unit_price,
    total_price: item.total_price,
  }))

  const { error: lineItemsError } = await supabase
    .from('invoice_line_items')
    .insert(lineItemRows)

  if (lineItemsError) return { error: lineItemsError.message }

  revalidatePath(`/patients/${caseId}/billing`)
  return { data: invoice }
}

// Soft-delete an invoice
export async function deleteInvoice(invoiceId: string, caseId: string) {
  const supabase = await createClient()
  const { error } = await supabase
    .from('invoices')
    .update({ deleted_at: new Date().toISOString() })
    .eq('id', invoiceId)

  if (error) return { error: error.message }
  revalidatePath(`/patients/${caseId}/billing`)
  return { success: true }
}

// Fetch full invoice context for detail view (includes case, patient, attorney, clinic, provider)
export async function getInvoiceWithContext(invoiceId: string) {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return { error: 'Not authenticated', data: null }

  // Parallel fetch: invoice+case+patient+attorney, clinic, provider profile (current user)
  const [invoiceResult, clinicResult, providerProfileResult] = await Promise.all([
    supabase
      .from('invoices')
      .select(`
        *,
        line_items:invoice_line_items(*),
        case:cases(
          *,
          patient:patients(*),
          attorney:attorneys(*)
        )
      `)
      .eq('id', invoiceId)
      .is('deleted_at', null)
      .single(),
    supabase
      .from('clinic_settings')
      .select('*')
      .is('deleted_at', null)
      .maybeSingle(),
    // Use current user ID for provider profile (same pattern as getInvoiceFormData)
    supabase
      .from('provider_profiles')
      .select('*')
      .eq('user_id', user.id)
      .is('deleted_at', null)
      .maybeSingle(),
  ])

  if (invoiceResult.error) return { error: invoiceResult.error.message, data: null }

  return { data: { invoice: invoiceResult.data, clinic: clinicResult.data, providerProfile: providerProfileResult.data } }
}

#### 1a. Invoice Type Switches Line Items (IMPLEMENTED)
**File**: `src/components/billing/create-invoice-dialog.tsx`

**Requirement**: When the user switches invoice type, the pre-populated line items must update to match.

**Visit type** (current behavior — "Medical Invoice"):
All service items from the service catalog that match recorded clinical events:
| CPT | Description | Condition |
|-----|-------------|-----------|
| 99204 | Initial exam (45-60min) | `initial_visit_notes` row with `visit_type='initial_visit'` exists |
| 99204 | Pain evaluation visit (45-60min) | `initial_visit_notes` row with `visit_type='pain_evaluation_visit'` exists |
| 76140 | MRI review | Approved MRI extraction exists |
| 0232T / 86999 / 76942 | PRP preparation and injection with US guided | One per PRP procedure |
| 99213 | Follow up / Discharge visit | Discharge note exists |

**Facility type** ("Medical Facility Invoice"):
Only PRP procedure items, with a different description:
| CPT | Description | Condition |
|-----|-------------|-----------|
| 0232T / 86999 / 76942 | Medical site utilization | One per PRP procedure |

Same CPT codes and pricing (PRP bundle sum from service catalog), but:
- Description changes to **"Medical site utilization"**
