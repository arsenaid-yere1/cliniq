# Story 6.1 — Create Invoice from Procedure

## Overview

Implement the ability to create medical invoices from the Billing tab of a patient case. Two invoice types are supported: **Medical Invoice** (visit-level, multiple line items per service — initial exam, MRI review, PRP procedures, discharge) and **Medical Facility Invoice** (PRP procedures only, each described as "Medical site utilization" with same CPT codes 0232T/86999/76942). Invoices are auto-numbered, pre-populated with case/procedure data, and editable before saving.

## Current State Analysis

### What Exists:
- **DB schema**: `invoices`, `invoice_line_items`, `payments` tables in `002_case_dashboard_tables.sql`
- **Billing page**: read-only at `src/app/(dashboard)/patients/[caseId]/billing/page.tsx`
- **BillingTable**: sortable table with disabled "Create Invoice" button (`src/components/billing/billing-table.tsx`)
- **BillingSummaryCard**: displays total billed/paid/balance (`src/components/billing/billing-summary-card.tsx`)
- **Server actions**: `listInvoices` and `getBillingSummary` only (`src/actions/billing.ts`)
- **Cases table**: has `total_billed`, `total_paid`, `balance_due` columns (`001_initial_schema.sql:86-88`)
- **Procedures table**: has `cpt_code`, `diagnoses` (jsonb), `injection_site`, `laterality` columns (pricing comes from the product catalog, not the procedure row)
- **Clinic settings**: clinic name, address, phone, fax, logo (`007_clinic_provider_settings.sql`)
- **Provider profiles**: display_name, credentials, NPI (`007_clinic_provider_settings.sql`)

### What's Missing:
- No `invoice_type` column on invoices
- No `service_date` on `invoice_line_items` (needed — line items span different dates)
- No invoice number auto-generation (column exists but no trigger/sequence)
- No indication, claim_type, or diagnoses snapshot on invoices
- No payee fields on invoices
- No create/update/delete server actions for invoices
- No Zod validation schemas for invoices
- No invoice creation form/dialog UI
- No invoice detail view

### Key Discoveries:
- Diagnoses come from two sources: `initial_visit_notes.diagnoses` (free text) and `procedures.diagnoses` (jsonb array of `{icd10_code, description}`) — `src/lib/validations/prp-procedure.ts:3-6`
- Indication uses the same `formatReasonForVisit(accident_type)` helper from `src/lib/constants/clinical-note-header.ts` that Initial Visit Notes and Discharge Notes use — maps `cases.accident_type` to a standardized medical-legal etiology phrase (e.g., "Post-traumatic musculoskeletal pain following motor vehicle collision"). Consistency across all documents in the paper trail is critical for defensible PI paperwork.
- Clinic fax field exists: `008_add_fax_to_clinic_settings.sql`
- Invoice PDFs show: clinic header (logo, address, phone, fax), all info rendered as tables — patient/case info table, diagnoses table (ICD-10 | Description), attorney table (firm name + address), line items table (DATE | CPT | Description | QTY | Amount), total, payee line
- Check is always payable to the business (clinic), not the provider
- Two invoice types have different titles: "Medical Facility Invoice" vs "Medical Invoice"
- Existing patterns: server actions use `safeParse` + `revalidatePath`, forms use `react-hook-form` + `zodResolver`, dialogs use controlled `open`/`onOpenChange` props

## Desired End State

From the Billing tab, users can click "Create Invoice" and:
1. Select invoice type (Medical Invoice or Medical Facility Invoice)
2. See auto-populated patient info, provider info, diagnoses, indication, and attorney info
3. See pre-populated line items from recorded procedures (with CPT codes matching the PDF examples)
4. Edit any field including line items (add/remove/modify)
5. Save the invoice, which auto-generates an invoice number and updates case billing totals
6. View created invoices from the billing table (clickable rows leading to detail view)
7. Edit existing invoices

### Verification:

- Verify case `total_billed` / `balance_due` update after invoice creation
- Verify line items are editable and totals recalculate

## What We're NOT Doing

- PDF export (Story 6.3)
- Pricing catalog / admin pricing config (Story 6.2)
- Payment recording (separate feature)
- Invoice status workflow (draft → pending → paid transitions) — invoices save as `draft` for now
- Email/fax sending of invoices
- Multi-case invoices

## Implementation Approach

Build bottom-up: schema migration → server actions → validation → UI components. The invoice creation form is a dialog with two sections: header info (auto-populated, editable) and line items table (pre-populated from procedures, editable). Invoice detail is a new page matching the PDF layout.

## Phase 0.5: Per-Site Billing on Procedures (FIXED)

### Overview
A single PRP procedure record can cover multiple injection sites (e.g., cervical AND lumbar in the same encounter). Previously every procedure row billed as `quantity: 1`, under-billing multi-site encounters. Billing now derives the quantity automatically by parsing the existing `injection_site` free-text field — no schema change, no form change, no manual input.

**Scope**: Medical Invoice (visit type) only. Medical Facility Invoice continues to bill one line per procedure row at `quantity: 1`.

**Why parse instead of adding a column:** `injection_site` already captures the sites ("Cervical and Lumbar"). A separate `site_count` field was redundant — users would be entering the same information twice. Parsing at billing time keeps the UI unchanged and works retroactively for all existing procedures.

### Changes Required (IMPLEMENTED)

#### 1. `src/actions/billing.ts` — add `countInjectionSites()` and use it for visit PRP line items
Added a helper at the top of the file that splits `injection_site` on `,`, `;`, `/`, `&`, `+`, or the word "and" (case-insensitive, word boundary):

```typescript
function countInjectionSites(injectionSite: string | null | undefined): number {
  if (!injectionSite) return 1
  const parts = injectionSite
    .split(/,|;|\/|&|\+|\s+and\s+/i)
    .map((p) => p.trim())
    .filter((p) => p.length > 0)
  return Math.max(1, parts.length)
}
```

In the PRP procedure loop, replaced the hardcoded `quantity: 1` with:
```typescript
const quantity = countInjectionSites(typedProc.injection_site)
// total_price: unitPrice * quantity
```

Facility line items were left at `quantity: 1` (per user requirement — facility invoices bill one line per procedure regardless of site count).

#### 2. Migration — drop the short-lived `site_count` column
**File**: `supabase/migrations/20260420_drop_procedures_site_count.sql`

```sql
alter table public.procedures drop column if exists site_count;
```

(An earlier iteration added `site_count` with an explicit form field; that approach was abandoned in favor of the parser.)

  target_case_id uuid;

// Fetch all data needed for the invoice creation form
export async function getInvoiceFormData(caseId: string) {
  const supabase = await createClient()

  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return { error: 'Not authenticated', data: null }

  // Parallel fetch: case+patient+attorney, procedures, clinic, provider profile (current user), initial visit note
  // NOTE: Provider profile uses current user ID, NOT assigned_provider_id.
  // This matches the pattern in procedure-notes, discharge-notes, initial-visit-notes actions.
  // This is a single-provider clinic — the logged-in user IS the provider.
  const [caseResult, proceduresResult, clinicResult, providerProfileResult, initialVisitResult] = await Promise.all([
    supabase
      .from('cases')
      .select(`
        *,
        patient:patients(*),
        attorney:attorneys(*)
      `)
      .eq('id', caseId)
      .is('deleted_at', null)
      .single(),
    supabase
      .from('procedures')
      .select('*')
      .eq('case_id', caseId)
      .is('deleted_at', null)
      .order('procedure_date', { ascending: true }),
    supabase
      .from('clinic_settings')
      .select('*')
      .is('deleted_at', null)
      .maybeSingle(),
    supabase
      .from('provider_profiles')
      .select('*')
      .eq('user_id', user.id)
      .is('deleted_at', null)
      .maybeSingle(),
    supabase
      .from('initial_visit_notes')
      .select('chief_complaint, diagnoses, created_at, visit_date')
      .eq('case_id', caseId)
      .is('deleted_at', null)
      .order('created_at', { ascending: false })
      .limit(1)
      .maybeSingle(),
  ])

  if (caseResult.error) return { error: caseResult.error.message, data: null }

  const providerProfile = providerProfileResult.data

  // Derive diagnoses: prefer structured from procedures. If no procedure has structured
  // diagnoses AND the case has no discharge note yet (treatment in progress), fall back
  // to parsing ICD-10 codes from the Initial Visit Note's free-text `diagnoses` field.
  // Once a discharge note exists, we treat the case as closed enough that empty is the
  // intended state — don't surprise the user with old IVN codes on a post-discharge invoice.
  let diagnoses: Array<{ icd10_code: string | null; description: string }> = []
  const procedures = proceduresResult.data ?? []
  const procedureWithDiagnoses = procedures.find(
    (p: { diagnoses?: unknown }) => Array.isArray(p.diagnoses) && p.diagnoses.length > 0
  )
  if (procedureWithDiagnoses) {
    diagnoses = procedureWithDiagnoses.diagnoses as typeof diagnoses
  } else if (!dischargeNoteResult.data) {
    // Parse IVN diagnoses text — format: "• M54.5 — Low back pain" or "M54.5 - Low back pain" per line.
    // Prefer pain_evaluation_visit (imaging-confirmed codes) over initial_visit (clinical impression).
    // Same parsing logic as getCaseDiagnoses() in src/actions/procedures.ts.
    const ivnRows = visitNotes
    const preferredIvn =
      ivnRows.find((r) => r.visit_type === 'pain_evaluation_visit' && r.diagnoses)
      ?? ivnRows.find((r) => r.visit_type === 'initial_visit' && r.diagnoses)
      ?? null
    if (preferredIvn?.diagnoses) {
      for (const line of preferredIvn.diagnoses.split('\n')) {
        const match = line.match(/^[•\-\d.]*\s*([A-Z]\d{1,2}\.?\d{0,4}[A-Z]{0,2})\s*[—–\-]\s*(.+)$/i)
        if (match) {
          diagnoses.push({ icd10_code: match[1].trim(), description: match[2].trim() })
        }
      }
    }
  }

  // Derive indication: use formatReasonForVisit() — same medical-legal etiology phrase
  // used by Initial Visit Notes and Discharge Notes. Ensures the invoice's indication
  // matches the rest of the chart for defensible PI paperwork.
  const indication = formatReasonForVisit(caseResult.data.accident_type)

  // Build pre-populated line items from procedures. Pricing comes from the product catalog
  // (PRP bundle = 0232T + 86999 + 76942); procedures do not carry their own charge amount.
  const prePopulatedLineItems = procedures.map((proc: {
    id: string
    procedure_date: string
    cpt_code: string | null
    procedure_name: string
    injection_site?: string | null
    laterality?: string | null
  }) => ({
    procedure_id: proc.id,
    service_date: proc.procedure_date,
    cpt_code: proc.cpt_code ?? '',
    description: proc.procedure_name + (proc.injection_site ? ` — ${proc.injection_site}` : '') + (proc.laterality ? ` (${proc.laterality})` : ''),
    quantity: 1,
    unit_price: 0,
    total_price: 0,
  }))

  return {
    data: {
      caseData: caseResult.data,
      procedures,
      clinic: clinicResult.data,
      providerProfile,
      diagnoses,
      indication,
      prePopulatedLineItems,
    },
  }
}

// Create a new invoice with line items
export async function createInvoice(caseId: string, values: CreateInvoiceFormValues) {
  const parsed = createInvoiceSchema.safeParse(values)
  if (!parsed.success) return { error: parsed.error.flatten().fieldErrors }

  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return { error: 'Not authenticated' }

  const { line_items, ...invoiceData } = parsed.data

  // Calculate total
  const totalAmount = line_items.reduce((sum, item) => sum + item.total_price, 0)

  // Insert invoice (invoice_number auto-generated by trigger)
  const { data: invoice, error: invoiceError } = await supabase
    .from('invoices')
    .insert({
      case_id: caseId,
      invoice_type: invoiceData.invoice_type,
      invoice_date: invoiceData.invoice_date,
      claim_type: invoiceData.claim_type,
      indication: invoiceData.indication || null,
      diagnoses_snapshot: invoiceData.diagnoses_snapshot,
      payee_name: invoiceData.payee_name || null,
      payee_address: invoiceData.payee_address || null,
      notes: invoiceData.notes || null,
      total_amount: totalAmount,
      status: 'draft',
      created_by_user_id: user.id,
      updated_by_user_id: user.id,
    })
    .select()
    .single()

  if (invoiceError || !invoice) return { error: invoiceError?.message ?? 'Failed to create invoice' }

  // Insert line items
  const lineItemRows = line_items.map((item) => ({
    invoice_id: invoice.id,
    procedure_id: item.procedure_id || null,
    service_date: item.service_date,
    cpt_code: item.cpt_code,
    description: item.description,
    quantity: item.quantity,
    unit_price: item.unit_price,
    total_price: item.total_price,
  }))

  const { error: lineItemsError } = await supabase
    .from('invoice_line_items')
    .insert(lineItemRows)

  if (lineItemsError) return { error: lineItemsError.message }

  revalidatePath(`/patients/${caseId}/billing`)
  return { data: invoice }
}

// Update an existing invoice
export async function updateInvoice(invoiceId: string, caseId: string, values: UpdateInvoiceFormValues) {
  const parsed = updateInvoiceSchema.safeParse(values)
  if (!parsed.success) return { error: parsed.error.flatten().fieldErrors }

  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return { error: 'Not authenticated' }

  const { line_items, ...invoiceData } = parsed.data
  const totalAmount = line_items.reduce((sum, item) => sum + item.total_price, 0)

  const { data: invoice, error: invoiceError } = await supabase
    .from('invoices')
    .update({
      ...invoiceData,
      indication: invoiceData.indication || null,
      payee_name: invoiceData.payee_name || null,
      payee_address: invoiceData.payee_address || null,
      notes: invoiceData.notes || null,
      total_amount: totalAmount,
      updated_by_user_id: user.id,
    })
    .eq('id', invoiceId)
    .select()
    .single()

  if (invoiceError || !invoice) return { error: invoiceError?.message ?? 'Failed to update invoice' }

  // Replace line items: delete existing, insert new
  await supabase
    .from('invoice_line_items')
    .delete()
    .eq('invoice_id', invoiceId)

  const lineItemRows = line_items.map((item) => ({
    invoice_id: invoiceId,
    procedure_id: item.procedure_id || null,
    service_date: item.service_date,
    cpt_code: item.cpt_code,
    description: item.description,
    quantity: item.quantity,
    unit_price: item.unit_price,
    total_price: item.total_price,
  }))

// Soft-delete an invoice
export async function deleteInvoice(invoiceId: string, caseId: string) {

    .update({ deleted_at: new Date().toISOString() })

// Fetch full invoice context for detail view (includes case, patient, attorney, clinic, provider)
export async function getInvoiceWithContext(invoiceId: string) {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return { error: 'Not authenticated', data: null }

  // Parallel fetch: invoice+case+patient+attorney, clinic, provider profile (current user)
  const [invoiceResult, clinicResult, providerProfileResult] = await Promise.all([
    supabase
      .from('invoices')
      .select(`
        *,
        line_items:invoice_line_items(*),
        case:cases(
          *,
          patient:patients(*),
          attorney:attorneys(*)
        )
      `)
      .eq('id', invoiceId)
      .is('deleted_at', null)
      .single(),
    supabase
      .from('clinic_settings')
      .select('*')
      .is('deleted_at', null)
      .maybeSingle(),
    // Use current user ID for provider profile (same pattern as getInvoiceFormData)
    supabase
      .from('provider_profiles')
      .select('*')
      .eq('user_id', user.id)
      .is('deleted_at', null)
      .maybeSingle(),
  ])

  if (invoiceResult.error) return { error: invoiceResult.error.message, data: null }

  return { data: { invoice: invoiceResult.data, clinic: clinicResult.data, providerProfile: providerProfileResult.data } }

**Why a new column instead of relying on insert order**: Supabase/PostgREST returns nested rows in an unspecified order unless `.order()` is explicit. `created_at` on `invoice_line_items` (`002_case_dashboard_tables.sql`) would tie on a bulk insert (same-microsecond timestamps) and becomes meaningless on edit since `updateInvoice` deletes and re-inserts. A small `display_order` int column is the simplest durable signal.

## Phase 5: CPT Code Normalization (Prices Not Picked Up from Settings Price List)

### Overview

**Bug**: Invoice pre-population sometimes shows `$0.00` for line items even though the corresponding CPT code is configured with a non-zero `default_price` in the Settings → Pricing Catalog.

**Root cause**: String-key mismatch between how CPT codes are saved by the settings UI and how they're looked up during invoice pre-population.

- **Settings UI** ([src/components/settings/pricing-catalog-form.tsx:206-210](src/components/settings/pricing-catalog-form.tsx#L206-L210)) saves `cpt_code` as whatever the user typed — no `.trim()`, no case normalization. Zod schema ([src/lib/validations/service-catalog.ts:5](src/lib/validations/service-catalog.ts#L5)) is `z.string().min(1)` with no pattern.
- **Price map** ([src/actions/service-catalog.ts:103-105](src/actions/service-catalog.ts#L103-L105)) uses the raw DB string as the key: `priceMap[item.cpt_code] = Number(item.default_price)`.
- **Pre-population** ([src/actions/billing.ts:215](src/actions/billing.ts#L215), [src/actions/billing.ts:236](src/actions/billing.ts#L236), [src/actions/billing.ts:270](src/actions/billing.ts#L270), [src/actions/billing.ts:308](src/actions/billing.ts#L308)) looks up with hardcoded uppercase, whitespace-free keys: `priceMap['99204']`, `priceMap['76140']`, `priceMap['0232T']`, `priceMap['86999']`, `priceMap['76942']`, `priceMap['99213']`.

Any of these causes a silent $0 fallback:
- Trailing/leading whitespace (`"99204 "`)
- Lowercase T (`"0232t"`)
- User typed a bundled row (`"0232T+86999+76942"` or `"0232T, 86999, 76942"`) — none of the three individual lookups hit
- Row was deleted from catalog entirely

The line-item CPT picker in the dialog ([src/components/billing/cpt-code-combobox.tsx](src/components/billing/cpt-code-combobox.tsx)) is already populated from the catalog via `formData.catalogItems` — that part works. The bug is isolated to the **pre-population path** when the dialog first opens.

**Why fix at both ends (write + read)**: Normalizing only on write leaves existing rows broken. Normalizing only on read means Settings still accepts dirty input and the UI table will show `"0232t"` next to `"76140"` inconsistently. Do both.

### Changes Required

#### 1. Normalize on write — Settings form
**File**: [src/lib/validations/service-catalog.ts](src/lib/validations/service-catalog.ts)

Transform `cpt_code` in the Zod schema so it's impossible to save a code with surrounding whitespace or lowercase letters:

```typescript
import { z } from 'zod'

export const serviceCatalogItemSchema = z.object({
  id: z.string().uuid().optional(),
  cpt_code: z
    .string()
    .min(1, 'CPT code is required')
    .transform((s) => s.trim().toUpperCase())
    .refine((s) => /^[A-Z0-9]+$/.test(s), {
      message: 'CPT code must contain only letters and digits (no spaces, punctuation, or bundles). Enter each code as a separate row.',
    }),
  description: z.string().min(1, 'Description is required'),
  default_price: z.coerce.number().min(0, 'Price must be non-negative'),
  sort_order: z.coerce.number().int().optional(),
})

export type ServiceCatalogItemFormValues = z.infer<typeof serviceCatalogItemSchema>
```

The `.refine()` explicitly rejects user attempts to enter bundles like `"0232T+86999+76942"` — the refine message tells them to add each code as its own row. This matches how billing.ts looks them up individually.

No change needed in the form component itself — the schema transform runs during `safeParse` in `createServiceCatalogItem` / `updateServiceCatalogItem` ([src/actions/service-catalog.ts:23, 59](src/actions/service-catalog.ts)), so normalized values land in the DB.

#### 2. Normalize on read — Price map
**File**: [src/actions/service-catalog.ts](src/actions/service-catalog.ts)

Defend against any existing rows that slipped in before normalization was added:

```typescript
export async function getServiceCatalogPriceMap(): Promise<Record<string, number>> {
  const supabase = await createClient()
  const { data } = await supabase
    .from('service_catalog')
    .select('cpt_code, default_price')
    .is('deleted_at', null)

  const priceMap: Record<string, number> = {}
  for (const item of data ?? []) {
    const normalized = (item.cpt_code ?? '').trim().toUpperCase()
    if (!normalized) continue
    priceMap[normalized] = Number(item.default_price)
  }
  return priceMap
}
```

The billing.ts lookup keys (`'99204'`, `'0232T'`, etc.) are already uppercase/trimmed, so no changes needed there.

#### 3. Data fix for existing dirty rows
**File**: `supabase/migrations/NNN_normalize_service_catalog_cpt_codes.sql` (new)

One-time normalization of whatever is already in the DB:

```sql
update public.service_catalog
set cpt_code = upper(trim(cpt_code))
where cpt_code <> upper(trim(cpt_code));
```

This is safe to re-run — it's a no-op on rows that are already clean. It does NOT split bundle rows (e.g., `"0232T+86999+76942"`) — those are treated as an invalid configuration that the user must fix manually via Settings after the Zod refine lands. A separate admin step (not part of this migration) should notify the user if any post-trim row still fails the `^[A-Z0-9]+$/` check.

#### 4. Diagnostic: surface missing-price mismatches
**File**: [src/actions/billing.ts](src/actions/billing.ts)

Currently, a missing CPT silently falls back to `unit_price: 0`. This is how the bug stayed hidden. Add a lightweight server-side console warning (no user-facing change) when `getInvoiceFormData` builds a pre-populated line item with a `$0` price for a CPT that *should* have one:

```typescript
// After building priceMap, before the line-item loops:
const expectedCodes = ['99204', '76140', '0232T', '86999', '76942', '99213']
const missingCodes = expectedCodes.filter((code) => (priceMap[code] ?? 0) === 0)
if (missingCodes.length > 0) {
  console.warn(
    `[billing] Missing or zero-priced CPT codes in service_catalog: ${missingCodes.join(', ')} — check Settings → Pricing Catalog`,
  )
}
```

Low cost (runs once per dialog open), no UI change, makes the next instance of this bug immediately diagnosable in Vercel / Supabase logs. Leave as `console.warn` — not a toast or thrown error — because a clinic legitimately might choose to leave some codes at `$0` and override manually.

#### 5. Regenerate types (no-op if no columns changed)
```bash
npx supabase gen types typescript --linked > src/types/database.ts
```

Expected: no diff, since the migration only updates data, not the schema.
