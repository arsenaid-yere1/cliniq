# Epic 4 Story 4.2 — Capture PRP Procedure Details: Implementation Plan

## Overview

Extend the existing "Record Procedure" dialog (built in Story 4.1) with four new form sections capturing PRP-specific treatment data: PRP Preparation, Anesthesia, Injection, and Post-Procedure. All new fields are stored as new columns on the `procedures` table. The dialog becomes a single scrollable form with a sticky section nav for orientation.

Additionally, Pain is relocated from Encounter Details into the Vitals section and captured as a **range** (min/max on a 0–10 NRS) rather than a single value, using the existing `vital_signs.pain_score_min` / `vital_signs.pain_score_max` columns. All downstream document generators (Procedure Note, Discharge Note) are updated to consume the range.

---

## Current State Analysis

- `procedures` table has Story 4.1 PRP columns: `injection_site`, `laterality`, `diagnoses`, `consent_obtained`, `pain_rating`, `procedure_number` (`supabase/migrations/013_prp_procedure_encounter.sql`)
- `vital_signs` table already has `pain_score_min` / `pain_score_max` integer columns (0–10 check) from `supabase/migrations/028_vital_signs_pain_score.sql` — not yet wired into PRP form or server action
- `src/lib/validations/prp-procedure.ts` — `prpProcedureFormSchema` has `pain_rating` as sibling of `vital_signs`; `vitalSignsSchema` contains 6 fields (BP, HR, RR, temp, SpO2) — **no pain columns**
- `src/actions/procedures.ts` — `createPrpProcedure` writes `pain_rating` to `procedures` (line 112) and vitals to `vital_signs` (lines 142–162); `updatePrpProcedure` mirrors this; `getProcedureDefaults` reads `vital_signs.pain_score_max` from Initial Visit as the `pain_rating` default (line 435)
- `src/components/procedures/record-procedure-dialog.tsx` — pain_rating FormField lives in **Encounter Details** section (lines 378–399); Vital Signs section (lines 794–931) has no pain field
- Document generation touchpoints for pain/vitals:
  - `lib/claude/generate-procedure-note.ts` — `ProcedureNoteInputData.procedureRecord.pain_rating` (single int) + `vitalSigns` object (6 fields) + `priorProcedure.pain_rating` for comparison; system prompt renders as `"• Pain: {X}/10"` and compares prior visit
  - `src/actions/procedure-notes.ts` — `gatherProcedureNoteSourceData` reads `procedures.pain_rating` (line 157) and `vital_signs` (lines 43–55) separately
  - `lib/claude/generate-discharge-note.ts` — `DischargeNoteInputData.latestPainRating` (single int) + `latestVitals` object + per-procedure `pain_rating` in summary array
  - `src/actions/discharge-notes.ts` — `gatherDischargeNoteSourceData` selects `procedures.pain_rating` (line 52) and per-procedure latest vitals (lines 149–160)
  - PDF templates (`src/lib/pdf/procedure-note-template.tsx`, `src/lib/pdf/discharge-note-template.tsx`) render a single `objective_vitals` string — pain is embedded in the prose by Claude, not a separate template variable
- Story 4.3 will consume all 4.1 + 4.2 fields for AI note generation — every column name here becomes a variable in the note template

## Desired End State

Provider opens "Record Procedure", scrolls through 6 sections (Encounter Details → PRP Preparation → Anesthesia → Injection → Post-Procedure → Vital Signs), fills in required fields, submits. All new fields persist to `procedures`. The Vital Signs section includes a **Pain Range (min / max, 0–10)** pair that writes to `vital_signs.pain_score_min` / `pain_score_max`. Section nav pills at the top of the dialog jump to each section on click.

### Verification:
- Submitting the form creates a `procedures` row with all Story 4.2 columns populated
- Pain range min/max persist to `vital_signs.pain_score_min` / `pain_score_max` (no longer to `procedures.pain_rating`)
- Zod rejects submission when `pain_score_min > pain_score_max` (both present)
- Required fields (blood draw volume, injection volume, anesthetic agent, guidance method, complications) block submission when empty
- Complications "Other – specify" reveals textarea; selecting "None" hides it and clears the value
- Section nav pills scroll the dialog to the correct section
- Procedure Note and Discharge Note generate correctly with the pain range (e.g. "Pain: 3–6/10", or "Pain: 5/10" when min == max)
- Existing Story 4.1 behavior (injection site, laterality, diagnoses, consent) is unchanged

## What We're NOT Doing

- Story 4.3 note generation — separate story (but we DO update the existing note-generation action signatures + prompts to consume the pain range, since the old `pain_rating` single int is being retired from the write path)
- Edit/delete of existing procedure records
- Supplies used as a structured multi-select — plain textarea is sufficient
- Guidance method as free-text — enum only (ultrasound / fluoroscopy / landmark)
- Changing the dialog to a wizard
- Dropping `procedures.pain_rating` column — leave in place for historical rows; stop writing to it. A later cleanup migration can drop after backfill is verified.

**(a) New file**: `src/lib/icd10/parse-ivn-diagnoses.ts`

```ts
import { normalizeIcd10Code, validateIcd10Code } from './validation'

export type ParsedIvnDiagnosis = {
  icd10_code: string
  description: string
}

// Parse ICD-10 codes from Initial Visit Note `diagnoses` free-text.
// Format per line: "• M54.5 — Low back pain" or "M54.5 - Low back pain".
// Structurally invalid codes are skipped.
export function parseIvnDiagnoses(text: string | null | undefined): ParsedIvnDiagnosis[] {
  if (!text) return []
  const out: ParsedIvnDiagnosis[] = []
  for (const line of text.split('\n')) {
    const match = line.match(/^[•\-\d.]*\s*([A-Z]\d{1,2}\.?\d{0,4}[A-Z]{0,2})\s*[—–\-]\s*(.+)$/i)
    if (!match) continue
    const v = validateIcd10Code(match[1])
    if (!v.ok && v.reason === 'structure') continue
    out.push({
      icd10_code: normalizeIcd10Code(match[1]),
      description: match[2].trim(),
    })
  }
  return out
}
```

## Phase 6: Injection Site Default Robustness

### Overview
Providers enter `body_region` as free text on the Initial Visit intake ([initial-visit-editor.tsx:604](src/components/clinical/initial-visit-editor.tsx#L604), placeholder `"e.g., Neck, Lower Back"`). The current default parser in [procedures.ts:411-419](src/actions/procedures.ts#L411-L419) only recognizes `"left "` / `"right "` prefixes, so real-world values like `"L Knee"`, `"Lt Shoulder"`, `"R lower back"`, or `"Bilateral knees"` fall through with `laterality: null` and an unclean `injection_site`. Fix by expanding the parser to handle common abbreviations and normalize casing.

### Changes Required:

#### 1. Extract a pure helper in `src/actions/procedures.ts`

Replace the inline parser at [procedures.ts:404-420](src/actions/procedures.ts#L404-L420) with a call to a new exported helper:

```ts
export function parseBodyRegion(raw: string): {
  injection_site: string
  laterality: 'left' | 'right' | 'bilateral' | null
} {
  const region = (raw ?? '').trim()
  if (!region) return { injection_site: '', laterality: null }

  // Match leading laterality token, optionally followed by "." and whitespace.
  // Groups: left | right | bilateral (with aliases)
  const match = region.match(
    /^(left|lt|l|right|rt|r|bilateral|bilat|both)\.?\s+(.+)$/i,
  )

  if (!match) {
    return { injection_site: titleCase(region), laterality: null }
  }

  const token = match[1].toLowerCase()
  let rest = match[2].trim()
  let laterality: 'left' | 'right' | 'bilateral'

  if (token === 'left' || token === 'lt' || token === 'l') laterality = 'left'
  else if (token === 'right' || token === 'rt' || token === 'r') laterality = 'right'
  else laterality = 'bilateral'

  // For bilateral, strip trailing plural 's' so "Bilateral knees" → "Knee".
  if (laterality === 'bilateral' && /[a-z]s$/i.test(rest)) {
    rest = rest.slice(0, -1)
  }

  return { injection_site: titleCase(rest), laterality }
}

function titleCase(s: string): string {
  return s
    .toLowerCase()
    .replace(/\b\w/g, (c) => c.toUpperCase())
}
```

Then in `getProcedureDefaults`:

