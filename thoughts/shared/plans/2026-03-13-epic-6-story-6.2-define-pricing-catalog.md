# Story 6.2 — Define Pricing Catalog Implementation Plan

## Overview

Add a pricing catalog to the Settings page where an admin can configure default prices for all service types. These prices auto-populate into invoice line items when creating invoices, replacing the current hardcoded `0` values.

## Current State Analysis

### What Exists:
- **Invoice line item pre-population** in `src/actions/billing.ts:153-229` builds line items from procedures, initial visit notes, MRI extractions, and discharge notes — but hardcodes `unit_price: 0` and `total_price: 0` for everything except PRP procedures (which pull from `procedures.charge_amount`)
- **Settings page** at `src/app/(dashboard)/settings/page.tsx` with tabbed layout (`src/components/settings/settings-tabs.tsx`) — 5 existing tabs
- **No pricing/catalog table** in the database
- **Existing CPT codes** used in invoice generation:
  - `99204` — Initial exam (45-60min)
  - `76140` — MRI review
  - `0232T` / `86999` / `76942` — PRP preparation and injection with US guided
  - `99213` — Follow up / Discharge visit

### Key Discoveries:
- Settings page fetches data server-side and passes to a client `SettingsTabs` component — `src/app/(dashboard)/settings/page.tsx:5-8`
- Settings use an upsert pattern (check existence, then insert or update) — `src/actions/settings.ts:25-63`
- The `procedures.charge_amount` column (`numeric(10,2)`) exists but is the only place pricing lives — `supabase/migrations/002_case_dashboard_tables.sql:35`
- Invoice line items have `unit_price` and `total_price` columns (`numeric(10,2)`) — `supabase/migrations/002_case_dashboard_tables.sql:73-74`

## Desired End State

A "Pricing" tab in Settings where the admin can:
1. See all catalog entries in an inline-editable table
2. Edit CPT code, description, and default price directly in table cells
3. Add new catalog entries
4. Remove catalog entries (soft delete)
5. Save changes

When creating an invoice, `getInvoiceFormData` looks up default prices from the catalog by CPT code and pre-populates `unit_price` and `total_price` on line items instead of hardcoding `0`.

### Verification:
- Configure prices for all service types in Settings > Pricing
- Create an invoice — verify all line items have correct default prices from the catalog
- Edit a catalog price — create a new invoice — verify the new price appears
- Line items remain editable on the invoice (catalog prices are defaults, not locked)

## What We're NOT Doing

- Per-site pricing (different prices for different injection sites)
- Tiered pricing (discounts for multiple procedures)
- Historical pricing / price versioning (invoices snapshot the price at creation time, which they already do via `unit_price` on line items)
- CPT code validation against an external database
- Bulk import/export of pricing data

## Implementation Approach

Build bottom-up: DB migration → server actions → validation schema → UI component → wire into invoice generation. The catalog is a simple flat table with inline editing in a new Settings tab.

## Phase 1: Database Migration

### Overview
Create the `service_catalog` table and seed it with the known service types.

### Changes Required:

#### 1. New Migration File
**File**: `supabase/migrations/019_service_catalog.sql`

```sql
-- ============================================
-- SERVICE CATALOG TABLE
-- ============================================
create table public.service_catalog (
  id                    uuid primary key default gen_random_uuid(),
  cpt_code              text not null,
  description           text not null,
  default_price         numeric(10,2) not null default 0,
  sort_order            integer not null default 0,
  created_at            timestamptz not null default now(),
  updated_at            timestamptz not null default now(),
  deleted_at            timestamptz,
  created_by_user_id    uuid references public.users(id),
  updated_by_user_id    uuid references public.users(id)
);

create index idx_service_catalog_cpt_code on public.service_catalog(cpt_code);

create trigger set_updated_at before update on public.service_catalog
  for each row execute function update_updated_at();

alter table public.service_catalog enable row level security;

create policy "Authenticated users full access" on public.service_catalog
  for all using (auth.role() = 'authenticated');

-- ============================================
-- SEED DEFAULT CATALOG ENTRIES
-- ============================================
insert into public.service_catalog (cpt_code, description, default_price, sort_order) values
  ('99204', 'Initial exam (45-60min)', 0, 1),
  ('76140', 'MRI review', 0, 2),
  ('0232T', 'PRP preparation and injection', 0, 3),
  ('86999', 'Blood draw and centrifuge', 0, 4),
  ('76942', 'Ultrasound guidance', 0, 5),
  ('99213', 'Follow up / Discharge visit', 0, 6);
```

Notes:
- `sort_order` controls display order in the admin table and can be used for line item ordering
- `cpt_code` is not unique — allows multiple entries with the same CPT code if needed (e.g., different descriptions)
- Seed prices are `0` — admin sets real prices after migration
- Follows existing patterns: `gen_random_uuid()`, `update_updated_at()` trigger, soft delete, RLS, audit fields

## Phase 2: Validation Schema & Server Actions

### Overview
Create Zod schema for catalog entries and server actions for CRUD + listing.

### Changes Required:

#### 1. Validation Schema
**File**: `src/lib/validations/service-catalog.ts`

```typescript
import { z } from 'zod'

export const serviceCatalogItemSchema = z.object({
  id: z.string().uuid().optional(),
  cpt_code: z.string().min(1, 'CPT code is required'),
  description: z.string().min(1, 'Description is required'),
  default_price: z.coerce.number().min(0, 'Price must be non-negative'),
  sort_order: z.coerce.number().int().optional(),
})

export type ServiceCatalogItemFormValues = z.infer<typeof serviceCatalogItemSchema>
```

#### 2. Server Actions
**File**: `src/actions/service-catalog.ts`

```typescript
'use server'

import { revalidatePath } from 'next/cache'
import { createClient } from '@/lib/supabase/server'
import {
  serviceCatalogItemSchema,
  type ServiceCatalogItemFormValues,
} from '@/lib/validations/service-catalog'

export async function listServiceCatalog() {
  const supabase = await createClient()
  const { data, error } = await supabase
    .from('service_catalog')
    .select('*')
    .is('deleted_at', null)
    .order('sort_order', { ascending: true })

  if (error) return { error: error.message, data: [] }
  return { data: data ?? [] }
}

export async function createServiceCatalogItem(values: ServiceCatalogItemFormValues) {
  const parsed = serviceCatalogItemSchema.safeParse(values)
  if (!parsed.success) return { error: parsed.error.flatten().fieldErrors }

  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  // Get next sort_order
  const { data: maxRow } = await supabase
    .from('service_catalog')
    .select('sort_order')
    .is('deleted_at', null)
    .order('sort_order', { ascending: false })
    .limit(1)
    .maybeSingle()

  const nextSortOrder = (maxRow?.sort_order ?? 0) + 1

  const { data, error } = await supabase
    .from('service_catalog')
    .insert({
      cpt_code: parsed.data.cpt_code,
      description: parsed.data.description,
      default_price: parsed.data.default_price,
      sort_order: nextSortOrder,
      created_by_user_id: user?.id,
      updated_by_user_id: user?.id,
    })
    .select()
    .single()

  if (error) return { error: error.message }
  revalidatePath('/settings')
  return { data }
}

export async function updateServiceCatalogItem(id: string, values: ServiceCatalogItemFormValues) {
  const parsed = serviceCatalogItemSchema.safeParse(values)
  if (!parsed.success) return { error: parsed.error.flatten().fieldErrors }

  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  const { data, error } = await supabase
    .from('service_catalog')
    .update({
      cpt_code: parsed.data.cpt_code,
      description: parsed.data.description,
      default_price: parsed.data.default_price,
      updated_by_user_id: user?.id,
    })
    .eq('id', id)
    .select()
    .single()

  if (error) return { error: error.message }
  revalidatePath('/settings')
  return { data }
}

export async function deleteServiceCatalogItem(id: string) {
  const supabase = await createClient()
  const { error } = await supabase
    .from('service_catalog')
    .update({ deleted_at: new Date().toISOString() })
    .eq('id', id)

  if (error) return { error: error.message }
  revalidatePath('/settings')
  return { success: true }
}

// Used by getInvoiceFormData to look up default prices by CPT code
export async function getServiceCatalogPriceMap(): Promise<Record<string, number>> {
  const supabase = await createClient()
  const { data } = await supabase
    .from('service_catalog')
    .select('cpt_code, default_price')
    .is('deleted_at', null)

  const priceMap: Record<string, number> = {}
  for (const item of data ?? []) {
    // If multiple entries share a CPT code, last one wins (by default DB order)
    priceMap[item.cpt_code] = Number(item.default_price)
  }
  return priceMap
}
```

## Phase 3: Pricing Tab UI (Inline Editable Table)

### Overview
Add a "Pricing" tab to the Settings page with an inline-editable table for managing the service catalog.

### Changes Required:

#### 1. Pricing Catalog Form Component
**File**: `src/components/settings/pricing-catalog-form.tsx`

A `'use client'` component that renders an inline-editable table:

- **Table columns**: CPT Code | Description | Default Price ($) | Actions (delete button)
- **Inline editing**: Each cell is an `<Input>` field. Values are edited in-place.
- **Row state**: Track which rows are "dirty" (modified since last save). Use local component state with an array of catalog items.
- **Add row**: "Add Service" button appends a new empty row at the bottom
- **Delete row**: Trash icon per row. For new (unsaved) rows, just remove from local state. For existing rows, call `deleteServiceCatalogItem` with confirmation via `AlertDialog`.
- **Save**: A single "Save Changes" button at the bottom. On click:
  - For rows with an `id` that are dirty → call `updateServiceCatalogItem`
  - For rows without an `id` (new) → call `createServiceCatalogItem`
  - Use `Promise.all` to save all changes in parallel
  - Show toast on success/failure
- **Props**: `initialData: ServiceCatalogItem[]` — the current catalog rows from the server

```typescript
interface ServiceCatalogItem {
  id: string
  cpt_code: string
  description: string
  default_price: number
  sort_order: number

interface PricingCatalogFormProps {
  initialData: ServiceCatalogItem[]
}
```

Key implementation details:
- Use `useState` for the items array (not react-hook-form — simpler for inline table editing)

- Empty state: "No services configured. Add your first service to get started."
- Disable "Save Changes" when no rows are dirty and no new rows exist

### Edge Cases:
- Empty catalog (all entries deleted) → invoice line items should default to $0
- New CPT code added to catalog that doesn't match any auto-generated line item → no effect on invoices (catalog is a lookup, not a driver)
- Multiple catalog entries with same CPT code → last one wins in price map (acceptable for now)

## Performance Considerations

- `getServiceCatalogPriceMap` is a lightweight single-table query (< 20 rows expected) — negligible overhead added to `getInvoiceFormData`
- Settings page adds one additional query to its `Promise.all` — no perceptible impact

## Migration Notes

- Migration is additive only — new table, no changes to existing tables
- Seed data uses `$0` prices — admin must configure real prices after deployment
- Existing invoices are unaffected — they already have `unit_price` saved on each line item
- No data migration needed
