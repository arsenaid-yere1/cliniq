# Epic 2, Story 2.4 — Physical Therapy Report Extraction: Implementation Plan

## Overview

Add AI-powered extraction of structured clinical data from uploaded physical therapy (PT) evaluation reports. When a user uploads a PT PDF, the system automatically sends it to Claude API, extracts functional outcome measures, ROM data, muscle strength, special tests, treatment goals, and plan of care into a structured format. The provider reviews, edits, approves, or rejects the extraction via the existing clinical data split-view workflow.

## Current State Analysis

The codebase has three complete extraction pipelines (MRI, Chiro, Pain Management) that are structurally identical. PT follows the exact same pattern.

### PT vs Pain Management — Why a Separate Pipeline

PT evaluation reports have fundamentally different data than pain management reports:

| PT Report (function-centric) | Pain Management Report (diagnosis/intervention-centric) |
|---|---|
| Functional outcome measures (NDI, ODI, PSFS, LEFS) | Chief complaints with pain ratings |
| Muscle strength (MMT grades per muscle group) | Palpation findings |
| Treatment goals (short-term and long-term, SMART format) | Treatment plan with cost estimates |
| Plan of care (frequency, duration, modalities with CPT codes) | Injection/surgery recommendations |
| Gait and postural assessment | Neurological summary |
| Functional tests (TUG, sit-to-stand, single leg stance) | Diagnostic studies summary |

There is overlap in ROM and orthopedic tests, but the core value proposition of each report type is different. PT documents the **functional recovery timeline** (critical for attorney damages calculations); pain management establishes **medical causation and treatment necessity**.

### Sample PT Evaluation Report Structure (from research):
1. **Patient/referral info** — patient name, DOB, DOI, evaluation date, referring provider
2. **Subjective** — chief complaint, mechanism of injury, pain ratings (NPRS at rest/with activity), symptom characteristics, functional limitations, prior treatment, work status
3. **Objective — Observation** — postural assessment, gait analysis
4. **Objective — ROM** — per region, AROM/PROM in degrees, pain at end range
5. **Objective — Strength** — MMT grades (0-5) per muscle group
6. **Objective — Palpation** — tenderness (0-3+ scale), spasm, trigger points
7. **Objective — Special Tests** — orthopedic tests (positive/negative)
8. **Objective — Neurological** — DTRs, sensation, upper/lower motor neuron signs
9. **Objective — Functional Tests** — TUG, sit-to-stand, single leg stance
10. **Outcome Measures** — NDI, ODI, PSFS, LEFS, DASH (scored instruments)
11. **Assessment** — clinical impression, causation statement, prognosis
12. **Goals** — short-term (2-4 weeks) and long-term (6-12 weeks), SMART format
13. **Plan of Care** — frequency, duration, modalities, HEP, re-evaluation schedule
14. **Diagnoses** — ICD-10 codes

## Desired End State

After implementation:
1. `pt_report` is a valid document type in the upload selector
2. Uploading a PT PDF triggers automatic extraction via Claude API
3. Extraction results are stored in `pt_extractions` table
4. The clinical data page has a "Physical Therapy" tab showing extractions
5. Clicking an extraction opens the split-view: PDF on left, editable data on right
6. Provider can approve, edit + save, or reject each extraction
7. Extraction failures show a manual entry fallback

### How to verify:
1. Upload a PDF with document type "PT Report" → extraction runs automatically, toast shows success
2. Navigate to Clinical Data tab → "Physical Therapy" tab shows extraction with confidence badge
3. Click extraction → split-view shows PDF alongside extracted fields
4. Edit a field and save → `provider_overrides` contains the edit, `review_status` = `'edited'`
5. Approve → `review_status` = `'approved'`
6. Re-extract → old record soft-deleted, new extraction created

## What We're NOT Doing

- Extracting individual visit/progress notes (only initial evaluation reports)
- Tracking visit-by-visit outcome measure trends (would require a separate progress note pipeline)
- CPT code validation or billing integration
- Auto-linking PT goals to pain management diagnoses
- Discharge summary extraction (separate report type if needed later)
- Extracting from scanned/image-based reports (Claude handles PDFs natively)

## Implementation Approach

Follow the established per-type extraction pattern (same as MRI/Chiro/PM). Build bottom-up: database → schemas → Claude module → server actions → upload hook → UI.

**System prompt** — tailored to PT evaluation reports:
```
You are a medical data extraction assistant for a personal injury clinic.
You are extracting structured data from a physical therapy initial evaluation report.

RULES:
1. Extract pain ratings exactly as stated. NPRS at rest, with activity, worst, and best.
2. For range of motion, extract AROM/PROM distinction, degrees, and whether pain occurs at end range. Use standard normal values.
3. For muscle strength, extract MMT grades (0/5 to 5/5) per muscle group with side (left/right/bilateral).
4. Extract palpation findings with tenderness grade (0-3+), spasm presence, and trigger point presence per location.
5. Special/orthopedic tests: extract each test name, result (positive/negative), side, and any notes.
6. For standardized outcome measures (NDI, ODI, PSFS, LEFS, DASH, QuickDASH), extract the instrument name, numeric score, max score, percentage if stated, and interpretation.
7. Extract ALL treatment goals as written — both short-term (2-4 week) and long-term (6-12 week). Include timeframe, baseline value, and target value when stated.
8. For plan of care, extract frequency (e.g., "3x/week"), duration (e.g., "6 weeks"), modalities with CPT codes when listed, HEP prescribed (yes/no), and re-evaluation schedule.
9. Extract ALL diagnosis codes exactly as written, including the ICD-10 7th character.
10. Neurological screening: extract DTR grades per level with side, sensation summary, and motor notes.
11. Functional tests: extract test name (e.g., "Timed Up and Go"), measured value (e.g., "12.5 seconds"), and interpretation when provided.
12. Return null for any field that cannot be determined from the document.
13. Set confidence to "low" for poor quality scans, partial documents, or ambiguous content.
14. Add extraction_notes for anything ambiguous or noteworthy.
```

#### 2. Update `gatherSourceData`
**File**: `src/actions/case-summaries.ts` (modify lines 18-62)

Add `pain_management_extractions` and `pt_extractions` queries to the `Promise.all`:

```typescript
async function gatherSourceData(
  supabase: Awaited<ReturnType<typeof createClient>>,
  caseId: string,
): Promise<{ data: SummaryInputData | null; error: string | null }> {
  const [caseRes, mriRes, chiroRes, pmRes, ptRes] = await Promise.all([
    supabase
      .from('cases')
      .select('accident_type, accident_date, accident_description')
      .eq('id', caseId)
      .is('deleted_at', null)
      .single(),
    supabase
      .from('mri_extractions')
      .select('body_region, mri_date, findings, impression_summary, provider_overrides')
      .eq('case_id', caseId)
      .is('deleted_at', null)
      .in('review_status', ['approved', 'edited']),
    supabase
      .from('chiro_extractions')
      .select('report_type, report_date, treatment_dates, diagnoses, treatment_modalities, functional_outcomes, provider_overrides')
      .eq('case_id', caseId)
      .is('deleted_at', null)
      .in('review_status', ['approved', 'edited']),
    supabase
      .from('pain_management_extractions')
      .select('report_date, examining_provider, chief_complaints, physical_exam, diagnoses, treatment_plan, diagnostic_studies_summary, provider_overrides')
      .eq('case_id', caseId)
      .is('deleted_at', null)
      .in('review_status', ['approved', 'edited']),
    supabase
      .from('pt_extractions')
      .select('evaluation_date, evaluating_therapist, pain_ratings, range_of_motion, muscle_strength, special_tests, outcome_measures, short_term_goals, long_term_goals, plan_of_care, diagnoses, clinical_impression, causation_statement, prognosis, provider_overrides')
      .eq('case_id', caseId)
      .is('deleted_at', null)
      .in('review_status', ['approved', 'edited']),
  ])

  if (caseRes.error || !caseRes.data) {
    return { data: null, error: 'Failed to fetch case details' }
  }

  const mriExtractions = mriRes.data || []
  const chiroExtractions = chiroRes.data || []
  const pmExtractions = pmRes.data || []
  const ptExtractions = ptRes.data || []

  if (mriExtractions.length === 0 && chiroExtractions.length === 0 && pmExtractions.length === 0 && ptExtractions.length === 0) {
    return { data: null, error: 'No approved extractions found. Approve at least one extraction first.' }
  }

  return {
    data: {
      caseDetails: caseRes.data,
      mriExtractions,
      chiroExtractions,
      pmExtractions,
      ptExtractions,
    },
    error: null,
  }
}
```

Key changes from current implementation:
- 5 parallel queries instead of 3
- Error message no longer says "MRI or chiro" — now accepts any approved extraction type
- Returns all four extraction arrays in `SummaryInputData`
