# Story 3.1 — Initial Visit Note Generation: Implementation Plan

## Overview

Implement AI-powered Initial Visit note generation that produces a clinical document with 16 sections (introduction, history of accident, post-accident history, chief complaint, past medical history, social history, review of systems, physical exam, imaging findings, diagnoses, medical necessity, treatment plan, patient education, prognosis, time and complexity attestation, clinician disclaimer) from existing case data. The provider can edit each section independently before finalizing. Finalization locks the note and creates a `documents` table entry.

## Current State Analysis

### What Exists
- **AI generation pattern**: `src/lib/claude/generate-summary.ts` — forced tool-use, Zod validation, null normalization, retry logic
- **Server action orchestration**: `src/actions/case-summaries.ts` — gather data → insert processing row → call Claude → write results → revalidate
- **Case summaries**: Approved case summaries provide synthesized data (chief complaint, imaging findings, prior treatment, symptoms timeline, diagnoses) — **required prerequisite** for note generation
- **Clinic/provider branding**: `clinic_settings` (name, address, phone, fax, logo) and `provider_profiles` (display name, credentials, NPI, signature) via `src/actions/settings.ts`
- **Documents table**: `document_type` CHECK constraint already includes `'generated'` — ready for finalized notes
- **Route structure**: `src/app/(dashboard)/patients/[caseId]/` with layout + sidebar

### What's Missing
1. `initial_visit_notes` database table
2. AI generation module for clinical narrative prose
3. Server actions for note CRUD + generation + finalization
4. Validation schemas for AI output and provider edit form
5. UI: page route, editor component, finalized view
6. Sidebar navigation entry
7. `'generated'` value in Zod document type enum (exists in DB constraint but missing from `src/lib/validations/document.ts`)

### Key Discoveries
- Route group is `(dashboard)`, not `(authenticated)` — `src/app/(dashboard)/patients/[caseId]/layout.tsx`
- Case sidebar nav items are a static array at `src/components/patients/case-sidebar.tsx:41-48`
- Next migration number is `010`
- `saveDocumentMetadata` at `src/actions/documents.ts:82` handles document row insertion — needs `caseId`, `documentType`, `fileName`, `filePath`, `fileSizeBytes`, `mimeType`
- The `documents` table requires a `file_path` (storage path) — for generated notes, we'll store the finalized note content as a JSON file in Supabase Storage

## Desired End State

After implementation:
1. Provider navigates to `/patients/[caseId]/initial-visit` via case sidebar
2. If no note exists and an approved case summary exists → "Generate Initial Visit Note" button is available
3. Clicking generate creates a draft note with 16 AI-populated sections
4. Provider edits any section via Textarea fields, can regenerate individual sections
5. Provider can edit the "Date of Visit" (defaults to today) and clicks "Save Draft" to persist edits, "Finalize" to lock the note
6. Finalized note displays as a read-only document with clinic header, provider signature, the provider-chosen visit date, and finalization date
7. Finalization also creates a `documents` row with `document_type: 'generated'`
8. Provider can "Unfinalize" to re-open for editing

### Verification
- Migration applies cleanly
- TypeScript compiles without errors
- Note generation produces valid 16-section content from approved case summary data (neurological findings included in Physical Examination)
- Provider can edit all 16 sections and save drafts
- Finalization locks the note and creates a documents entry
- Per-section regeneration works without losing edits to other sections
- Sidebar shows "Initial Visit" link with correct active state

## What We're NOT Doing

- Rich text editing (Tiptap/Quill) — plain Textarea for MVP
- Auto-save / debounced save — explicit "Save Draft" button only
- Multiple visit note types (follow-up, discharge) — only Initial Visit
- Note versioning / revision history
- Clinic-specific physical exam templates stored in settings

## Phase 1: Database Migration

### Overview
Create the `initial_visit_notes` table with text columns for 16 narrative sections, status tracking, and finalization metadata.

### Changes Required

#### 1. Migration File
**File**: `supabase/migrations/010_initial_visit_notes.sql` (new)

```sql
-- Initial visit notes: AI-generated clinical notes with provider editing
-- Follows case_summaries pattern (006) but uses TEXT columns for narrative prose

create table public.initial_visit_notes (
  id uuid primary key default uuid_generate_v4(),
  case_id uuid not null references public.cases(id),

  -- Section text fields (narrative prose)
  patient_info text,
  post_accident_history text,
  chief_complaint text,
  history_of_present_illness text,
  imaging_review text,
  prior_treatment_summary text,
  physical_exam text,
  assessment text,
  treatment_plan text,

  -- AI metadata
  ai_model text,
  raw_ai_response jsonb,

  -- Note status: generating → draft → finalized
  status text not null default 'draft'
    check (status in ('generating', 'draft', 'finalized', 'failed')),
  generation_error text,
  generation_attempts integer not null default 0,
  source_data_hash text,

  -- Provider finalization
  finalized_by_user_id uuid references public.users(id),
  finalized_at timestamptz,

  -- Link to documents table entry (created on finalization)
  document_id uuid references public.documents(id),

  -- Standard audit fields
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  created_by_user_id uuid references public.users(id),
  updated_by_user_id uuid references public.users(id),
  deleted_at timestamptz
);

-- Indexes
create index idx_initial_visit_notes_case_id on public.initial_visit_notes(case_id);
create index idx_initial_visit_notes_status on public.initial_visit_notes(status);

-- One active note per case (soft-delete aware)
create unique index idx_initial_visit_notes_case_active
  on public.initial_visit_notes(case_id) where deleted_at is null;

-- Updated_at trigger (reuse existing function from 001)
create trigger set_updated_at before update on public.initial_visit_notes
  for each row execute function update_updated_at();

-- RLS (same pattern as all other tables)
alter table public.initial_visit_notes enable row level security;

create policy "Authenticated users full access" on public.initial_visit_notes
  for all using (auth.role() = 'authenticated');
```

2. When generating, `gatherSourceData` fetches ROM from the active note row
3. ROM data is passed to Claude as part of `InitialVisitInputData`

- In `generateInitialVisitNote`: before soft-deleting the old note, read its `rom_data`. After inserting the new "generating" row, write that `rom_data` onto it.

**Recommended approach** (minimal changes):
1. `saveInitialVisitRom` saves `rom_data` on the active note row. If no note exists, it creates a new note row with `status: 'draft'` and only `rom_data` populated.
2. `generateInitialVisitNote` checks for an existing draft note. If one exists (with ROM data), it updates that row to `status: 'generating'` instead of soft-deleting and re-creating. This preserves `rom_data`.
3. `gatherSourceData` accepts the note ID and reads `rom_data` from the note row directly.
4. After generation, ROM data is preserved on the note row alongside the generated sections.

## Phase 8: Editable Visit Date

### Overview
Allow the provider to edit the "Date of Visit" on an Initial Visit Note instead of it being implicitly pinned to `finalized_at` (in the UI) or `new Date()` at render time (in the PDF). A new `visit_date date` column on `initial_visit_notes` stores the provider's chosen date, defaulting to today when a note is first created. This also fixes a latent bug where the PDF's "Date of Visit" silently drifts to the current date every time the PDF is re-rendered.

### Edge Cases
- Case with no clinic settings or provider profile → note should still generate (sections may lack branding data)
- Case summary with sparse data → AI should still produce reasonable content
- Very long section content → Textarea should handle gracefully
- Double-click prevention on Generate/Finalize buttons → `isPending` state should disable buttons
- Partial vitals (e.g., only BP and pain score entered, no HR) → AI should use real values for BP and pain score, `[XX]` for missing fields
- Vitals saved for procedures (with `procedure_id`) should NOT be fetched for initial visit — only vitals with `procedure_id IS NULL`
- ROM never added (provider skips ROM entirely) → AI generation omits ROM from physical exam entirely (no RANGE OF MOTION sub-heading), no ROM card shown
- Partial ROM (e.g., only cervical entered, lumbar all nulls) → AI uses real values for cervical, `[XX]` for lumbar movements with null actuals
- No orthopedic testing in physical exam → AI should never include Spurling's, Kemp's, straight leg raise, or similar tests
- ROM with custom added regions/movements → AI should handle arbitrary region names gracefully
- ROM data preserved across regeneration — saving ROM then regenerating the full note should not lose ROM data
- Provider adds ROM, saves, then removes ROM before generating → generation should not include ROM data
- Tone hint with very long text (500+ chars) → should still work; the user message to Claude can handle it
- Tone hint with adversarial content (e.g., "ignore all instructions") → Claude's system prompt should override; the hint is appended to the user message, not the system prompt, so it has lower priority
- Reset on a note that was just generated (no edits) → should work the same as reset after edits
- Reset button should not appear on finalized notes (must unfinalize first, then reset if desired)
- Double-click prevention on Reset button → `isPending` state should disable it
- First-visit with non-standard body region (e.g., "Wrist", "Ankle") → AI should select the most appropriate strain/sprain code for that anatomical region even though it's not in the explicit menu
- PRP evaluation with `suggested_diagnoses` containing low-confidence codes → AI should not blindly include them; should verify against imaging findings
- PRP evaluation with empty `suggested_diagnoses` array → AI should still generate appropriate codes from `imaging_findings` alone
- First-visit with headache but no accident link (accident_type is null) → should use R51.9 (unspecified headache) instead of G44.309 (post-traumatic)
- Case with both cervical and thoracic complaints → diagnoses should include codes for BOTH regions, not just the more common one
- Visit date in a different timezone: `visit_date` is stored as a `date` (no time component) — must be parsed as local time (`YYYY-MM-DDT00:00:00`) in both the editor and the PDF to avoid UTC-boundary off-by-one (e.g., "2026-04-11" rendering as "04/10/2026" for providers west of UTC)
- Future visit dates: the date input does not restrict to the past — providers can enter a future date if needed (e.g., backdating a note drafted ahead of a scheduled visit)
- Legacy notes created before this phase have `visit_date = null` → UI/PDF must gracefully fall back to `finalized_at`, and editing such a note pre-fills the input with today (not `finalized_at`) so the provider actively confirms the date
