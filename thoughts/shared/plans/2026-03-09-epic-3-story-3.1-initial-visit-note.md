# Story 3.1 — Initial Visit Note Generation: Implementation Plan

## Overview

Implement AI-powered Initial Visit note generation that produces a clinical document with 8 sections (patient info, chief complaint, history of present illness, imaging review, prior treatment summary, physical exam template, assessment, treatment plan) from existing case data. The provider can edit each section independently before finalizing. Finalization locks the note and creates a `documents` table entry.

## Current State Analysis

### What Exists
- **AI generation pattern**: `src/lib/claude/generate-summary.ts` — forced tool-use, Zod validation, null normalization, retry logic
- **Server action orchestration**: `src/actions/case-summaries.ts` — gather data → insert processing row → call Claude → write results → revalidate
- **Case summaries**: Approved case summaries provide synthesized data (chief complaint, imaging findings, prior treatment, symptoms timeline, diagnoses) — **required prerequisite** for note generation
- **Clinic/provider branding**: `clinic_settings` (name, address, phone, fax, logo) and `provider_profiles` (display name, credentials, NPI, signature) via `src/actions/settings.ts`
- **Documents table**: `document_type` CHECK constraint already includes `'generated'` — ready for finalized notes
- **Route structure**: `src/app/(dashboard)/patients/[caseId]/` with layout + sidebar

### What's Missing
1. `initial_visit_notes` database table
2. AI generation module for clinical narrative prose
3. Server actions for note CRUD + generation + finalization
4. Validation schemas for AI output and provider edit form
5. UI: page route, editor component, finalized view
6. Sidebar navigation entry
7. `'generated'` value in Zod document type enum (exists in DB constraint but missing from `src/lib/validations/document.ts`)

### Key Discoveries
- Route group is `(dashboard)`, not `(authenticated)` — `src/app/(dashboard)/patients/[caseId]/layout.tsx`
- Case sidebar nav items are a static array at `src/components/patients/case-sidebar.tsx:41-48`
- Next migration number is `010`
- `saveDocumentMetadata` at `src/actions/documents.ts:82` handles document row insertion — needs `caseId`, `documentType`, `fileName`, `filePath`, `fileSizeBytes`, `mimeType`
- The `documents` table requires a `file_path` (storage path) — for generated notes, we'll store the finalized note content as a JSON file in Supabase Storage

## Desired End State

After implementation:
1. Provider navigates to `/patients/[caseId]/initial-visit` via case sidebar
2. If no note exists and an approved case summary exists → "Generate Initial Visit Note" button is available
3. Clicking generate creates a draft note with 8 AI-populated sections
4. Provider edits any section via Textarea fields, can regenerate individual sections
5. Provider clicks "Save Draft" to persist edits, "Finalize" to lock the note
6. Finalized note displays as a read-only document with clinic header, provider signature, and finalization date
7. Finalization also creates a `documents` row with `document_type: 'generated'`
8. Provider can "Unfinalize" to re-open for editing

### Verification
- Migration applies cleanly
- TypeScript compiles without errors
- Note generation produces valid 8-section content from approved case summary data
- Provider can edit all 8 sections and save drafts
- Finalization locks the note and creates a documents entry
- Per-section regeneration works without losing edits to other sections
- Sidebar shows "Initial Visit" link with correct active state

## What We're NOT Doing

- PDF export or print formatting (future story)
- Rich text editing (Tiptap/Quill) — plain Textarea for MVP
- Auto-save / debounced save — explicit "Save Draft" button only
- Multiple visit note types (follow-up, discharge) — only Initial Visit
- Note versioning / revision history
- Clinic-specific physical exam templates stored in settings

## Phase 1: Database Migration

### Overview
Create the `initial_visit_notes` table with text columns for 8 narrative sections, status tracking, and finalization metadata.

### Changes Required

#### 1. Migration File
**File**: `supabase/migrations/010_initial_visit_notes.sql` (new)

```sql
-- Initial visit notes: AI-generated clinical notes with provider editing
-- Follows case_summaries pattern (006) but uses TEXT columns for narrative prose

create table public.initial_visit_notes (
  id uuid primary key default uuid_generate_v4(),
  case_id uuid not null references public.cases(id),

  -- Section text fields (narrative prose)
  patient_info text,
  chief_complaint text,
  history_of_present_illness text,
  imaging_review text,
  prior_treatment_summary text,
  physical_exam text,
  assessment text,
  treatment_plan text,

  -- AI metadata
  ai_model text,
  raw_ai_response jsonb,

  -- Note status: generating → draft → finalized
  status text not null default 'draft'
    check (status in ('generating', 'draft', 'finalized', 'failed')),
  generation_error text,
  generation_attempts integer not null default 0,
  source_data_hash text,

  -- Provider finalization
  finalized_by_user_id uuid references public.users(id),
  finalized_at timestamptz,

  -- Link to documents table entry (created on finalization)
  document_id uuid references public.documents(id),

  -- Standard audit fields
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  created_by_user_id uuid references public.users(id),
  updated_by_user_id uuid references public.users(id),
  deleted_at timestamptz
);

-- Indexes
create index idx_initial_visit_notes_case_id on public.initial_visit_notes(case_id);
create index idx_initial_visit_notes_status on public.initial_visit_notes(status);

-- One active note per case (soft-delete aware)
create unique index idx_initial_visit_notes_case_active
  on public.initial_visit_notes(case_id) where deleted_at is null;

-- Updated_at trigger (reuse existing function from 001)
create trigger set_updated_at before update on public.initial_visit_notes
  for each row execute function update_updated_at();

-- RLS (same pattern as all other tables)
alter table public.initial_visit_notes enable row level security;

create policy "Authenticated users full access" on public.initial_visit_notes
  for all using (auth.role() = 'authenticated');
```

## Phase 3: AI Generation Module

### Overview
Create the Claude integration for generating Initial Visit note prose from case summary data, patient demographics, and clinic/provider info.

### Changes Required

#### 1. Generation Module
**File**: `src/lib/claude/generate-initial-visit.ts` (new)

**System Prompt** — instructs Claude to:
- Write in clinical Initial Visit note format appropriate for personal injury documentation
- Use precise medical terminology suitable for legal proceedings
- Generate narrative prose (not structured data) for each section
- For physical exam: generate a body-region-specific exam template based on imaging findings
- For treatment plan: generate a preliminary plan based on diagnoses and prior treatment

**Tool Definition** — `generate_initial_visit_note` tool with 8 required string properties (one per section). No nullable fields — all sections must be populated.

**Input Data Interface** — `InitialVisitInputData`:
```typescript
export interface InitialVisitInputData {
  // From patients + cases tables
  patientInfo: {
    first_name: string
    last_name: string
    date_of_birth: string | null
    gender: string | null
  }
  caseDetails: {
    case_number: string
    accident_type: string | null
    accident_date: string | null
    accident_description: string | null
  }
  // From approved case_summaries (required prerequisite)
  caseSummary: {
    chief_complaint: string | null
    imaging_findings: unknown
    prior_treatment: unknown
    symptoms_timeline: unknown
    suggested_diagnoses: unknown
  }
  // From clinic_settings
  clinicInfo: {
    clinic_name: string | null
    address_line1: string | null
    city: string | null
    state: string | null
    zip_code: string | null
    phone: string | null
    fax: string | null
  }
  // From provider_profiles
  providerInfo: {
    display_name: string | null
    credentials: string | null
    npi_number: string | null
  }
}
```

**Function** — `generateInitialVisitFromData(inputData: InitialVisitInputData)`:
- Same pattern as `generateCaseSummaryFromData`
- Uses `claude-sonnet-4-6`, `max_tokens: 8192` (larger than summary since we're generating prose)
- Forced tool-use: `tool_choice: { type: 'tool', name: 'generate_initial_visit_note' }`
- No null normalization needed (all sections are required strings)
- Validates with `initialVisitNoteResultSchema.safeParse()`
- Returns `{ data?: InitialVisitNoteResult; rawResponse?: unknown; error?: string }`

**Per-Section Regeneration Function** — `regenerateSection(inputData: InitialVisitInputData, section: InitialVisitSection, currentContent: string)`:
- Takes existing input data, the target section key, and the current content of that section
- Uses a modified system prompt that instructs Claude to regenerate ONLY the specified section
- Uses a simpler tool with a single string output field
- Returns `{ data?: string; error?: string }`

## Phase 4: Server Actions

### Overview
Create server actions for note generation, retrieval, editing, finalization, and per-section regeneration.

### Changes Required

#### 1. Server Actions File
**File**: `src/actions/initial-visit-notes.ts` (new)

```
'use server'
```

**Private helper: `gatherSourceData(caseId: string)`**
- Parallel queries for:
  1. `cases` table: `case_number`, `accident_type`, `accident_date`, `accident_description` + joined `patients` data (`first_name`, `last_name`, `date_of_birth`, `gender`)
  2. `case_summaries` table: the active approved summary (`review_status IN ('approved', 'edited')`, `generation_status = 'completed'`, `deleted_at IS NULL`)
  3. `clinic_settings` table: `clinic_name`, `address_line1`, `city`, `state`, `zip_code`, `phone`, `fax`
  4. `provider_profiles` table: `display_name`, `credentials`, `npi_number` (filtered by current user)
- Returns error if no approved case summary exists: `'An approved case summary is required before generating an Initial Visit note.'`
- Returns assembled `InitialVisitInputData` on success

**Private helper: `computeSourceHash(inputData: InitialVisitInputData)`**
- Same SHA-256 pattern as `case-summaries.ts`

**Exported actions:**

| Action | Signature | Purpose |
|---|---|---|
| `generateInitialVisitNote` | `(caseId: string)` | Gather data → soft-delete existing → insert `status: 'generating'` → call Claude → write sections → set `status: 'draft'`. One retry on failure. |
| `getInitialVisitNote` | `(caseId: string)` | Fetch the active (non-deleted) note for a case. Returns `null` if none exists. |
| `saveInitialVisitNote` | `(caseId: string, values: InitialVisitNoteEditValues)` | Validate with `initialVisitNoteEditSchema` → update section text columns → set `updated_by_user_id`. |
| `finalizeInitialVisitNote` | `(caseId: string)` | Set `status: 'finalized'`, `finalized_by_user_id`, `finalized_at`. Store note content as JSON in Supabase Storage (`case-documents` bucket) → create `documents` row with `document_type: 'generated'` → save `document_id` on the note. |
| `unfinalizeInitialVisitNote` | `(caseId: string)` | Set `status: 'draft'`, clear `finalized_by_user_id` and `finalized_at`. Does NOT delete the documents entry (keep audit trail). |
| `regenerateNoteSection` | `(caseId: string, section: InitialVisitSection)` | Gather fresh source data → call `regenerateSection()` → update only the specified section column. |
| `checkNotePrerequisites` | `(caseId: string)` | Check if an approved case summary exists. Returns `{ canGenerate: boolean; reason?: string }`. |

**Finalization document creation detail:**

When finalizing, the action:
1. Assembles the full note content as a JSON object
2. Uploads to Supabase Storage at `cases/{caseId}/initial-visit-note-{timestamp}.json`
3. Calls `saveDocumentMetadata` (or inserts directly) with:
   - `document_type: 'generated'`
   - `file_name: 'Initial Visit Note'`
   - `file_path: <storage path>`
   - `mime_type: 'application/json'`
   - `status: 'reviewed'` (finalized notes are already reviewed)
4. Updates the note row with the `document_id`

All mutating actions call `revalidatePath(`/patients/${caseId}`)`.

## Phase 5: UI — Page & Editor Component

### Overview
Create the Initial Visit page and editor component with all states: empty (with prerequisite check), generating, draft editing, finalized view, and failed.

### Changes Required

#### 1. Page Route
**File**: `src/app/(dashboard)/patients/[caseId]/initial-visit/page.tsx` (new)

Server component that:
- Awaits `params` to get `caseId`
- Calls `getInitialVisitNote(caseId)`, `checkNotePrerequisites(caseId)`, `getClinicSettings()`, `getProviderProfile()`, `getClinicLogoUrl()`, `getProviderSignatureUrl()` in parallel
- Passes all data to `InitialVisitEditor` client component

```typescript
import { getInitialVisitNote, checkNotePrerequisites } from '@/actions/initial-visit-notes'
import { getClinicSettings, getProviderProfile, getClinicLogoUrl, getProviderSignatureUrl } from '@/actions/settings'
import { InitialVisitEditor } from '@/components/clinical/initial-visit-editor'

export default async function InitialVisitPage({ params }: { params: Promise<{ caseId: string }> }) {
  const { caseId } = await params

  const [noteResult, prereqResult, clinicResult, providerResult, logoResult, signatureResult] = await Promise.all([
    getInitialVisitNote(caseId),
    checkNotePrerequisites(caseId),
    getClinicSettings(),
    getProviderProfile(),
    getClinicLogoUrl(),
    getProviderSignatureUrl(),
  ])

  return (
    <InitialVisitEditor
      caseId={caseId}
      note={noteResult.data ?? null}
      canGenerate={prereqResult.data?.canGenerate ?? false}
      prerequisiteReason={prereqResult.data?.reason}
      clinicSettings={clinicResult.data ?? null}
      providerProfile={providerResult.data ?? null}
      clinicLogoUrl={logoResult.url ?? null}
      providerSignatureUrl={signatureResult.url ?? null}
    />
  )
}
```

#### 2. Editor Component
**File**: `src/components/clinical/initial-visit-editor.tsx` (new)

Client component (`'use client'`) with the following states:

**State 1: No note exists**
- If `canGenerate` is true: "Generate Initial Visit Note" button
- If `canGenerate` is false: disabled button + message showing `prerequisiteReason` (e.g., "An approved case summary is required")

**State 2: Generating (`status === 'generating'`)**
- 8 skeleton sections with labels
- "Generating Initial Visit Note..." message

**State 3: Draft (`status === 'draft'`)**
- Page header: "Initial Visit Note" + status badge + "Save Draft" and "Finalize" buttons
- Form using `react-hook-form` + `zodResolver(initialVisitNoteEditSchema)`
- For each of the 8 sections:
  - Section label from `sectionLabels`
  - `<Textarea>` with auto-resize, pre-populated with AI-generated content
  - "Regenerate" icon button (RefreshCw icon) — calls `regenerateNoteSection` for that section
- Form `defaultValues` populated from the note's section columns
- "Save Draft" calls `saveInitialVisitNote`
- "Finalize" calls `finalizeInitialVisitNote` (with confirmation dialog: "Finalizing will lock this note. Continue?")

**State 4: Finalized (`status === 'finalized'`)**
- Read-only document view with:
  - **Clinic header**: logo (if available), clinic name, address, phone/fax
  - **Patient info section** rendered as text
  - All 8 sections rendered as formatted text with section headings
  - **Provider signature block**: provider name, credentials, signature image (if available), finalization date
- "Edit" button → calls `unfinalizeInitialVisitNote` to re-open as draft

**State 5: Failed (`status === 'failed'`)**
- Red alert with `generation_error`
- "Retry" button → calls `generateInitialVisitNote`

**Key implementation details:**
- Uses `useTransition` for all server action calls (consistent with case-summary-card pattern)
- Toast notifications on success/error
- `useForm` with `zodResolver` for draft editing
- Each Textarea should have `rows={6}` minimum with CSS `resize: vertical`
- Per-section regenerate uses `startTransition` and shows a spinner on the specific section's regenerate button

#### 3. Section Regenerate Confirmation
When provider clicks "Regenerate" on a section:
- Show inline confirmation: "This will replace the current content. Continue?"
- On confirm: call `regenerateNoteSection(caseId, sectionKey)`
- On success: update the form field value with the new content via `form.setValue(sectionKey, newContent)`

## Testing Strategy

### Manual Testing Steps
1. **Prerequisites check**: Navigate to Initial Visit for a case without an approved summary → verify button is disabled with explanation
2. **Generation flow**: Approve a case summary → navigate to Initial Visit → click Generate → verify 8 sections populated
3. **Editing flow**: Edit content in multiple sections → Save Draft → refresh page → verify edits persisted
4. **Section regeneration**: Click regenerate on one section → verify only that section changes, others are preserved
5. **Finalization flow**: Click Finalize → verify read-only view with clinic header and provider signature → check Documents tab for new entry
6. **Unfinalization**: Click Edit on finalized note → verify note returns to draft editing mode
7. **Error handling**: (simulate) If generation fails → verify error message + retry button works
8. **Navigation**: Verify sidebar link, active state, and all page transitions work smoothly

### Edge Cases
- Case with no clinic settings or provider profile → note should still generate (sections may lack branding data)
- Case summary with sparse data → AI should still produce reasonable content
- Very long section content → Textarea should handle gracefully
- Double-click prevention on Generate/Finalize buttons → `isPending` state should disable buttons

## Performance Considerations

- The 8-section form is a single form instance (not 8 separate forms) — efficient re-renders
- AI generation uses `claude-sonnet-4-6` with `max_tokens: 8192` — may take 10-15 seconds; skeleton UI keeps the user informed
- Per-section regeneration is fast (single section output) — ~2-3 seconds
- Finalization storage upload is a small JSON file — negligible latency

## Migration Notes

- No data migration needed (new table, no existing data)
- The `'generated'` document type already exists in the DB constraint — only the Zod enum needs updating
- No breaking changes to existing features
