# Story 3.1 — Initial Visit Note Generation: Implementation Plan

## Overview

Implement AI-powered Initial Visit note generation that produces a clinical document with 8 sections (patient info, chief complaint, history of present illness, imaging review, prior treatment summary, physical exam template, assessment, treatment plan) from existing case data. The provider can edit each section independently before finalizing. Finalization locks the note and creates a `documents` table entry.

## Current State Analysis

### What Exists
- **AI generation pattern**: `src/lib/claude/generate-summary.ts` — forced tool-use, Zod validation, null normalization, retry logic
- **Server action orchestration**: `src/actions/case-summaries.ts` — gather data → insert processing row → call Claude → write results → revalidate
- **Case summaries**: Approved case summaries provide synthesized data (chief complaint, imaging findings, prior treatment, symptoms timeline, diagnoses) — **required prerequisite** for note generation
- **Clinic/provider branding**: `clinic_settings` (name, address, phone, fax, logo) and `provider_profiles` (display name, credentials, NPI, signature) via `src/actions/settings.ts`
- **Documents table**: `document_type` CHECK constraint already includes `'generated'` — ready for finalized notes
- **Route structure**: `src/app/(dashboard)/patients/[caseId]/` with layout + sidebar

### What's Missing
1. `initial_visit_notes` database table
2. AI generation module for clinical narrative prose
3. Server actions for note CRUD + generation + finalization
4. Validation schemas for AI output and provider edit form
5. UI: page route, editor component, finalized view
6. Sidebar navigation entry
7. `'generated'` value in Zod document type enum (exists in DB constraint but missing from `src/lib/validations/document.ts`)

### Key Discoveries
- Route group is `(dashboard)`, not `(authenticated)` — `src/app/(dashboard)/patients/[caseId]/layout.tsx`
- Case sidebar nav items are a static array at `src/components/patients/case-sidebar.tsx:41-48`
- Next migration number is `010`
- `saveDocumentMetadata` at `src/actions/documents.ts:82` handles document row insertion — needs `caseId`, `documentType`, `fileName`, `filePath`, `fileSizeBytes`, `mimeType`
- The `documents` table requires a `file_path` (storage path) — for generated notes, we'll store the finalized note content as a JSON file in Supabase Storage

## Desired End State

After implementation:
1. Provider navigates to `/patients/[caseId]/initial-visit` via case sidebar
2. If no note exists and an approved case summary exists → "Generate Initial Visit Note" button is available
3. Clicking generate creates a draft note with 8 AI-populated sections
4. Provider edits any section via Textarea fields, can regenerate individual sections
5. Provider clicks "Save Draft" to persist edits, "Finalize" to lock the note
6. Finalized note displays as a read-only document with clinic header, provider signature, and finalization date
7. Finalization also creates a `documents` row with `document_type: 'generated'`
8. Provider can "Unfinalize" to re-open for editing

### Verification
- Migration applies cleanly
- TypeScript compiles without errors
- Note generation produces valid 8-section content from approved case summary data
- Provider can edit all 8 sections and save drafts
- Finalization locks the note and creates a documents entry
- Per-section regeneration works without losing edits to other sections
- Sidebar shows "Initial Visit" link with correct active state

## What We're NOT Doing

- Rich text editing (Tiptap/Quill) — plain Textarea for MVP
- Auto-save / debounced save — explicit "Save Draft" button only
- Multiple visit note types (follow-up, discharge) — only Initial Visit
- Note versioning / revision history
- Clinic-specific physical exam templates stored in settings

## Phase 1: Database Migration

### Overview
Create the `initial_visit_notes` table with text columns for 8 narrative sections, status tracking, and finalization metadata.

### Changes Required

#### 1. Migration File
**File**: `supabase/migrations/010_initial_visit_notes.sql` (new)

```sql
-- Initial visit notes: AI-generated clinical notes with provider editing
-- Follows case_summaries pattern (006) but uses TEXT columns for narrative prose

create table public.initial_visit_notes (
  id uuid primary key default uuid_generate_v4(),
  case_id uuid not null references public.cases(id),

  -- Section text fields (narrative prose)
  patient_info text,
  chief_complaint text,
  history_of_present_illness text,
  imaging_review text,
  prior_treatment_summary text,
  physical_exam text,
  assessment text,
  treatment_plan text,

  -- AI metadata
  ai_model text,
  raw_ai_response jsonb,

  -- Note status: generating → draft → finalized
  status text not null default 'draft'
    check (status in ('generating', 'draft', 'finalized', 'failed')),
  generation_error text,
  generation_attempts integer not null default 0,
  source_data_hash text,

  -- Provider finalization
  finalized_by_user_id uuid references public.users(id),
  finalized_at timestamptz,

  -- Link to documents table entry (created on finalization)
  document_id uuid references public.documents(id),

  -- Standard audit fields
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  created_by_user_id uuid references public.users(id),
  updated_by_user_id uuid references public.users(id),
  deleted_at timestamptz
);

-- Indexes
create index idx_initial_visit_notes_case_id on public.initial_visit_notes(case_id);
create index idx_initial_visit_notes_status on public.initial_visit_notes(status);

-- One active note per case (soft-delete aware)
create unique index idx_initial_visit_notes_case_active
  on public.initial_visit_notes(case_id) where deleted_at is null;

-- Updated_at trigger (reuse existing function from 001)
create trigger set_updated_at before update on public.initial_visit_notes
  for each row execute function update_updated_at();

-- RLS (same pattern as all other tables)
alter table public.initial_visit_notes enable row level security;

create policy "Authenticated users full access" on public.initial_visit_notes
  for all using (auth.role() = 'authenticated');
```

## Phase 4: Server Actions

### Overview
Create server actions for note generation, retrieval, editing, finalization, per-section regeneration, and vital signs CRUD.

### Changes Required

#### 1. Server Actions File
**File**: `src/actions/initial-visit-notes.ts` (existing — update)

```
'use server'
```

**Private helper: `gatherSourceData(caseId: string)`**
- Parallel queries for:
  1. `cases` table: `case_number`, `accident_type`, `accident_date`, `accident_description` + joined `patients` data (`first_name`, `last_name`, `date_of_birth`, `gender`)
  2. `case_summaries` table: the active approved summary (`review_status IN ('approved', 'edited')`, `generation_status = 'completed'`, `deleted_at IS NULL`)
  3. `clinic_settings` table: `clinic_name`, `address_line1`, `city`, `state`, `zip_code`, `phone`, `fax`
  4. `provider_profiles` table: `display_name`, `credentials`, `npi_number` (filtered by current user)
  5. **NEW**: `vital_signs` table: fetch the most recent vitals for the case where `procedure_id IS NULL` (initial visit vitals, not procedure vitals). Select `bp_systolic, bp_diastolic, heart_rate, respiratory_rate, temperature_f, spo2_percent`. Order by `recorded_at DESC`, limit 1, `maybeSingle()`.
- Returns error if no approved case summary exists
- Returns assembled `InitialVisitInputData` on success, with `vitalSigns` set to the fetched row or `null`

**Private helper: `computeSourceHash(inputData: InitialVisitInputData)`**
- Same SHA-256 pattern as `case-summaries.ts`

**Exported actions:**

| Action | Signature | Purpose |
|---|---|---|
| `generateInitialVisitNote` | `(caseId: string)` | Gather data (including vitals) → soft-delete existing → insert `status: 'generating'` → call Claude → write sections → set `status: 'draft'`. One retry on failure. |
| `getInitialVisitNote` | `(caseId: string)` | Fetch the active (non-deleted) note for a case. Returns `null` if none exists. |
| `saveInitialVisitNote` | `(caseId: string, values: InitialVisitNoteEditValues)` | Validate with `initialVisitNoteEditSchema` → update section text columns → set `updated_by_user_id`. |
| `finalizeInitialVisitNote` | `(caseId: string)` | Set `status: 'finalized'`, `finalized_by_user_id`, `finalized_at`. Store note content as JSON in Supabase Storage (`case-documents` bucket) → create `documents` row with `document_type: 'generated'` → save `document_id` on the note. |
| `unfinalizeInitialVisitNote` | `(caseId: string)` | Set `status: 'draft'`, clear `finalized_by_user_id` and `finalized_at`. Does NOT delete the documents entry (keep audit trail). |
| `regenerateNoteSection` | `(caseId: string, section: InitialVisitSection)` | Gather fresh source data (including latest vitals) → call `regenerateSection()` → update only the specified section column. |
| `checkNotePrerequisites` | `(caseId: string)` | Check if an approved case summary exists. Returns `{ canGenerate: boolean; reason?: string }`. |
| **NEW** `getInitialVisitVitals` | `(caseId: string)` | Fetch the most recent vitals for this case where `procedure_id IS NULL`. Returns `{ data: VitalSignsRow \| null }`. |
| **NEW** `saveInitialVisitVitals` | `(caseId: string, vitals: InitialVisitVitalsValues)` | Upsert vitals for the initial visit. If a row exists with `case_id` and `procedure_id IS NULL`, update it. Otherwise insert a new row with `procedure_id: null`. Validates with `initialVisitVitalsSchema`. |

**Vitals storage detail:**
- Vitals for the initial visit are stored in the existing `vital_signs` table with `procedure_id = NULL` (the column is already nullable)
- This distinguishes initial visit vitals from procedure vitals (which have `procedure_id` set)
- No new migration needed

**Finalization document creation detail:**

When finalizing, the action:
1. Assembles the full note content as a JSON object
2. Uploads to Supabase Storage at `cases/{caseId}/initial-visit-note-{timestamp}.json`
3. Calls `saveDocumentMetadata` (or inserts directly) with:
   - `document_type: 'generated'`
   - `file_name: 'Initial Visit Note'`
   - `file_path: <storage path>`
   - `mime_type: 'application/json'`
   - `status: 'reviewed'` (finalized notes are already reviewed)
4. Updates the note row with the `document_id`

All mutating actions call `revalidatePath(`/patients/${caseId}`)`.

## Phase 7: Range of Motion (ROM) Inputs

### Overview
Add structured ROM data entry so providers can record range of motion measurements during the initial visit. The AI generator will use actual ROM values instead of `[XX]` placeholders when ROM data is available. ROM data is stored as a `rom_data jsonb` column on the `initial_visit_notes` table.

### ROM Data Structure

Region-grouped, matching the PM extraction ROM pattern (`src/lib/validations/pain-management-extraction.ts:16-21`):

```typescript
interface RomRegion {
  region: string  // e.g., "Cervical Spine", "Lumbar Spine"
  movements: RomMovement[]
}

interface RomMovement {
  movement: string       // e.g., "Flexion", "Extension"
  normal: number | null  // Normal ROM in degrees
  actual: number | null  // Measured ROM in degrees
  pain: boolean          // Pain at end range
}
```

**Default pre-populated regions/movements** (provider can add/remove):

- **Cervical Spine**: Flexion (normal 60°), Extension (normal 75°), Left Lateral Flexion (normal 45°), Right Lateral Flexion (normal 45°), Left Rotation (normal 80°), Right Rotation (normal 80°)
- **Lumbar Spine**: Flexion (normal 60°), Extension (normal 25°), Left Lateral Flexion (normal 25°), Right Lateral Flexion (normal 25°), Left Rotation (normal 30°), Right Rotation (normal 30°)

#### 4. Server Actions Update
**File**: `src/actions/initial-visit-notes.ts` (existing — update)

**Update `gatherSourceData`:**
- Add a 6th parallel query to fetch `rom_data` from the active `initial_visit_notes` row for this case (if one exists). This handles the case where ROM was saved before generation.
- Alternatively: fetch ROM from the note row being generated. Since ROM is saved directly on the note row, read it from there.

**Actually simpler approach**: ROM is stored on the note row itself. In `generateInitialVisitNote`, after the "generating" row is inserted, fetch any previously saved ROM data from the old (soft-deleted) note and copy it to the new row. Then in `gatherSourceData`, read `rom_data` from the current note row.

**Better approach**: ROM is saved independently before/during generation. The flow:
1. Provider saves ROM data on the note row (or a pre-note state)
2. When generating, `gatherSourceData` fetches ROM from the active note row
3. ROM data is passed to Claude as part of `InitialVisitInputData`

Since ROM lives on `initial_visit_notes.rom_data`, and the generation flow soft-deletes the old note before creating a new "generating" row, we need to **carry ROM data forward**:
- In `generateInitialVisitNote`: before soft-deleting the old note, read its `rom_data`. After inserting the new "generating" row, write that `rom_data` onto it.
- In `gatherSourceData`: also read `rom_data` from the current note row (requires the note ID to be passed in, or a separate query).

**Simpler alternative**: Store ROM independently in the note row. When there's no note yet (pre-generation), the provider saves ROM which creates a "draft" note with only `rom_data` populated and all sections null. Generation then updates this existing row instead of soft-deleting and re-creating.

**Recommended approach** (minimal changes):
1. `saveInitialVisitRom` saves `rom_data` on the active note row. If no note exists, it creates a new note row with `status: 'draft'` and only `rom_data` populated.
2. `generateInitialVisitNote` checks for an existing draft note. If one exists (with ROM data), it updates that row to `status: 'generating'` instead of soft-deleting and re-creating. This preserves `rom_data`.
3. `gatherSourceData` accepts the note ID and reads `rom_data` from the note row directly.
4. After generation, ROM data is preserved on the note row alongside the generated sections.

### Manual Testing Steps

5. **Generation without vitals/ROM**: On a case with no saved vitals or ROM, generate → verify physical exam section uses `[XX]` placeholders for both
6. **Editing flow**: Edit content in multiple sections → Save Draft → refresh page → verify edits persisted
7. **Vitals update + regenerate**: Update vitals in draft mode → Save Vitals → click Regenerate on Physical Examination section → verify new vitals values appear
8. **ROM update + regenerate**: Update ROM actuals in draft mode → Save ROM → click Regenerate on Physical Examination section → verify new ROM values appear in regenerated section
9. **Section regeneration**: Click regenerate on one section → verify only that section changes, others are preserved
10. **Finalization flow**: Click Finalize → verify read-only view with clinic header and provider signature → check Documents tab for new entry
11. **Unfinalization**: Click Edit on finalized note → verify note returns to draft editing mode
12. **Error handling**: (simulate) If generation fails → verify error message + retry button works
13. **Navigation**: Verify sidebar link, active state, and all page transitions work smoothly

### Edge Cases
- Case with no clinic settings or provider profile → note should still generate (sections may lack branding data)
- Case summary with sparse data → AI should still produce reasonable content
- Very long section content → Textarea should handle gracefully
- Double-click prevention on Generate/Finalize buttons → `isPending` state should disable buttons
- Partial vitals (e.g., only BP entered, no HR) → AI should use real values for BP, `[XX]` for missing fields
- Vitals saved for procedures (with `procedure_id`) should NOT be fetched for initial visit — only vitals with `procedure_id IS NULL`
- Partial ROM (e.g., only cervical entered, lumbar all nulls) → AI uses real values for cervical, `[XX]` for lumbar movements with null actuals
- ROM with custom added regions/movements → AI should handle arbitrary region names gracefully
- ROM data preserved across regeneration — saving ROM then regenerating the full note should not lose ROM data

## Performance Considerations

- The 8-section form is a single form instance (not 8 separate forms) — efficient re-renders
- AI generation uses `claude-sonnet-4-6` with `max_tokens: 8192` — may take 10-15 seconds; skeleton UI keeps the user informed
- Per-section regeneration is fast (single section output) — ~2-3 seconds
- Finalization storage upload is a small JSON file — negligible latency
