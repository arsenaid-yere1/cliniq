# Epic 1 — Story 1.2: View Case Dashboard — Implementation Plan

## Overview

Implement Story 1.2: activate the case dashboard tabs (Documents, Procedures, Billing, Timeline) and enhance the Overview tab with live stat cards and recent activity. This fills in the shell built by Story 1.1 with real content pages, backed by new database tables for documents, procedures, invoices, and payments.

## Current State Analysis

### What Exists
- Case dashboard layout (`src/app/(dashboard)/patients/[caseId]/layout.tsx`) with `CaseSidebar` + children
- Case sidebar (`src/components/patients/case-sidebar.tsx`) with 5 nav items — only "Overview" is enabled
- Case overview (`src/components/patients/case-overview.tsx`) with hardcoded "0" stats and disabled quick actions
- Single page route: `[caseId]/page.tsx` rendering `CaseOverview`
- DB tables: `patients`, `cases`, `attorneys`, `case_status_history`, `audit_logs`, `users`
- No document, procedure, invoice, or payment tables
- No generated TypeScript types — all types are inline interfaces or Zod-inferred
- Established patterns: server actions with Zod safeParse, TanStack table, react-hook-form, shadcn/ui Dialog/AlertDialog

### Key Discoveries
- Both layout and page independently call `getPatientCase(caseId)` — no shared context
- `cases` table already has denormalized `total_billed`, `total_paid`, `balance_due` fields
- `case_status_history` exists and can feed the timeline
- `Sheet` component is installed but unused — available for document preview
- `use-debounce` not installed yet (needed for document search)
- No `src/types/` directory — types are scattered as local interfaces

## Desired End State

After this plan is complete:
- Navigating to `/patients/[caseId]/documents` shows a searchable, filterable document list (card-based)
- Navigating to `/patients/[caseId]/procedures` shows a TanStack table of procedures
- Navigating to `/patients/[caseId]/billing` shows a billing summary card + TanStack table of invoices
- Navigating to `/patients/[caseId]/timeline` shows a vertical timeline merging events from status history, documents, procedures, and invoices
- The Overview tab shows live stat cards and a recent activity preview
- Sidebar nav links are active (except Clinical Data — stays disabled)
- "Invoices" renamed to "Billing" in sidebar
- All new tables have RLS, indexes, soft delete support, and audit fields
- TypeScript types are generated from the Supabase schema

### Verification
- All tabs render without errors and show empty states when no data exists
- Sidebar navigation highlights the active tab correctly
- Document search filters results by keyword, type, and status
- Timeline displays events in reverse chronological order with correct icons/colors
- Overview stat cards show accurate counts from the database
- `npm run build` passes with no type errors
- `npm run lint` passes

## What We're NOT Doing

- **Document file upload** — no Supabase Storage integration; Documents tab is display-only (file upload is Epic 2)
- **Create/edit forms** — no forms for adding documents, procedures, or invoices; action buttons shown as disabled "Coming Soon"
- **Clinical Data tab** — remains disabled in sidebar; deferred to Epic 2 (AI extraction)
- **Server-side pagination** — per-case data volumes are small enough for client-side filtering
- **Seed data scripts** — not included; testing uses manually inserted records or the Supabase dashboard
- **Encounters/appointments table** — not needed yet; timeline builds from existing tables

## Implementation Approach

Build bottom-up: database first, then server actions, then UI components. Each phase is independently testable. The sidebar update is early (Phase 2) so all subsequent phases can be navigated to immediately, even if they initially show empty states.

```tsx
'use server'

import { createClient } from '@/lib/supabase/server'

export async function listDocuments(caseId: string, filters?: {
  search?: string
  documentType?: string
  status?: string
}) {
  const supabase = await createClient()

  let query = supabase
    .from('documents')
    .select('*, uploaded_by:users!uploaded_by_user_id(full_name), reviewed_by:users!reviewed_by_user_id(full_name)')
    .eq('case_id', caseId)
    .is('deleted_at', null)
    .order('created_at', { ascending: false })

  if (filters?.documentType && filters.documentType !== 'all') {
    query = query.eq('document_type', filters.documentType)
  }

  if (filters?.status && filters.status !== 'all') {
    query = query.eq('status', filters.status)
  }

  if (filters?.search) {
    query = query.or(`file_name.ilike.%${filters.search}%,notes.ilike.%${filters.search}%`)
  }

  const { data, error } = await query

  if (error) return { error: error.message, data: [] }
  return { data: data ?? [] }
}

export async function getDocumentCount(caseId: string) {
  const supabase = await createClient()

  const { count, error } = await supabase
    .from('documents')
    .select('*', { count: 'exact', head: true })
    .eq('case_id', caseId)
    .is('deleted_at', null)

  if (error) return { error: error.message, count: 0 }
  return { count: count ?? 0 }
}
```

#### 4. Timeline action
**File**: `src/actions/timeline.ts` (new)

```tsx
'use server'

import { createClient } from '@/lib/supabase/server'

export type TimelineEventType = 'status_change' | 'document_added' | 'procedure' | 'invoice_created'

export interface TimelineEvent {
  id: string
  type: TimelineEventType
  date: string
  title: string
  description: string | null
  metadata?: Record<string, unknown>
}

export async function getTimelineEvents(caseId: string): Promise<{ data: TimelineEvent[], error?: string }> {
  const supabase = await createClient()

  // Fetch all event sources in parallel
  const [statusRes, docRes, procRes, invRes] = await Promise.all([
    supabase
      .from('case_status_history')
      .select('id, previous_status, new_status, changed_at, notes, changed_by:users!changed_by_user_id(full_name)')
      .eq('case_id', caseId)
      .order('changed_at', { ascending: false }),
    supabase
      .from('documents')
      .select('id, file_name, document_type, created_at')
      .eq('case_id', caseId)
      .is('deleted_at', null)
      .order('created_at', { ascending: false }),
    supabase
      .from('procedures')
      .select('id, procedure_name, procedure_date, cpt_code, provider:users!provider_id(full_name)')
      .eq('case_id', caseId)
      .is('deleted_at', null)
      .order('procedure_date', { ascending: false }),
    supabase
      .from('invoices')
      .select('id, invoice_number, invoice_date, total_amount, status')
      .eq('case_id', caseId)
      .is('deleted_at', null)
      .order('invoice_date', { ascending: false }),
  ])

  const events: TimelineEvent[] = []

  // Map status changes
  for (const s of statusRes.data ?? []) {
    const changedBy = Array.isArray(s.changed_by) ? s.changed_by[0] : s.changed_by
    events.push({
      id: `status-${s.id}`,
      type: 'status_change',
      date: s.changed_at,
      title: s.previous_status
        ? `Status changed from ${formatStatus(s.previous_status)} to ${formatStatus(s.new_status)}`
        : `Case opened as ${formatStatus(s.new_status)}`,
      description: s.notes ?? (changedBy?.full_name ? `By ${changedBy.full_name}` : null),
    })
  }

  // Map documents
  for (const d of docRes.data ?? []) {
    events.push({
      id: `doc-${d.id}`,
      type: 'document_added',
      date: d.created_at,
      title: `Document added: ${d.file_name}`,
      description: formatDocType(d.document_type),
    })
  }

  // Map procedures
  for (const p of procRes.data ?? []) {
    const provider = Array.isArray(p.provider) ? p.provider[0] : p.provider
    events.push({
      id: `proc-${p.id}`,
      type: 'procedure',
      date: p.procedure_date + 'T00:00:00',
      title: p.procedure_name,
      description: [p.cpt_code, provider?.full_name].filter(Boolean).join(' - ') || null,
    })
  }

  // Map invoices
  for (const i of invRes.data ?? []) {
    events.push({
      id: `inv-${i.id}`,
      type: 'invoice_created',
      date: i.invoice_date + 'T00:00:00',
      title: `Invoice ${i.invoice_number}`,
      description: `$${Number(i.total_amount).toFixed(2)} - ${formatInvoiceStatus(i.status)}`,
    })
  }

  // Sort by date descending
  events.sort((a, b) => new Date(b.date).getTime() - new Date(a.date).getTime())

  return { data: events }
}

function formatStatus(status: string): string {
  const labels: Record<string, string> = {
    intake: 'Intake',
    active: 'Active',
    pending_settlement: 'Pending Settlement',
    closed: 'Closed',
    archived: 'Archived',
  }
  return labels[status] ?? status
}

function formatDocType(type: string): string {
  const labels: Record<string, string> = {
    mri_report: 'MRI Report',
    chiro_report: 'Chiropractor Report',
    generated: 'Generated Document',
    other: 'Other',
  }
  return labels[type] ?? type
}

function formatInvoiceStatus(status: string): string {
  const labels: Record<string, string> = {
    draft: 'Draft',
    pending: 'Pending',
    paid: 'Paid',
    partial: 'Partial',
    denied: 'Denied',
    overdue: 'Overdue',
  }
  return labels[status] ?? status
}
```

```tsx
import { createClient } from '@/lib/supabase/server'

export interface CaseDashboardStats {
  documentCount: number
  procedureCount: number
  totalBilled: number
  totalPaid: number
  balanceDue: number
  lastProcedureDate: string | null
}

export async function getCaseDashboardStats(caseId: string): Promise<{ data: CaseDashboardStats, error?: string }> {
  const supabase = await createClient()

  const [docRes, procRes, caseRes, lastProcRes] = await Promise.all([
    supabase
      .from('documents')
      .select('*', { count: 'exact', head: true })
      .eq('case_id', caseId)
      .is('deleted_at', null),
    supabase
      .from('procedures')
      .select('*', { count: 'exact', head: true })
      .eq('case_id', caseId)
      .is('deleted_at', null),
    supabase
      .from('cases')
      .select('total_billed, total_paid, balance_due')
      .eq('id', caseId)
      .single(),
    supabase
      .from('procedures')
      .select('procedure_date')
      .eq('case_id', caseId)
      .is('deleted_at', null)
      .order('procedure_date', { ascending: false })
      .limit(1)
      .maybeSingle(),
  ])

  return {
    data: {
      documentCount: docRes.count ?? 0,
      procedureCount: procRes.count ?? 0,
      totalBilled: Number(caseRes.data?.total_billed ?? 0),
      totalPaid: Number(caseRes.data?.total_paid ?? 0),
      balanceDue: Number(caseRes.data?.balance_due ?? 0),
      lastProcedureDate: lastProcRes.data?.procedure_date ?? null,
    },
  }
}
```

```tsx
import { notFound } from 'next/navigation'
import { getPatientCase } from '@/actions/patients'
import { getCaseDashboardStats } from '@/actions/dashboard'
```

## Testing Strategy

### Manual Testing Steps
1. Navigate to an existing case → Overview should show stat cards with 0 values
2. Insert a document row via Supabase dashboard → Documents tab shows the card, Overview document count updates to 1, Timeline shows "Document added" event
3. Insert a procedure row → Procedures tab shows the table row, Overview procedure count updates, Timeline shows procedure event
4. Insert an invoice row → Billing tab shows summary card + table row, Timeline shows invoice event
5. Test document search: insert 3 documents with different types/names, verify search and filters work
6. Verify all sidebar links highlight correctly when navigating between tabs
7. Verify Timeline shows events in correct chronological order with correct icons

### Edge Cases
- Case with no data in any table → all tabs show empty states
- Very long document names → should truncate gracefully
- Very long procedure notes → truncated in table with tooltip
- Null/missing provider on procedure → shows `—`
- All invoice statuses → each renders with correct badge color

## Performance Considerations

- All tab data is fetched server-side (RSC) — no client-side loading spinners needed
- Overview page uses `Promise.all` for parallel data fetching (stats + case + timeline)
- Timeline action uses `Promise.all` for 4 parallel Supabase queries
- Client-side filtering on Documents tab avoids re-fetching on filter change
- Per-case data volumes are expected to be small (5-30 documents, 10-50 procedures) — no pagination needed for MVP
