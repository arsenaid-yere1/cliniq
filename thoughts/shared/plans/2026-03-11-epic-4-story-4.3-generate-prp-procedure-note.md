# PRP Procedure Note Generation — Implementation Plan

## Overview

Generate an AI-powered PRP procedure note from structured data captured in Stories 4.1 and 4.2, allow provider editing, then finalize to a signed PDF stored as a case document. This mirrors the Initial Visit Note workflow exactly — same status machine, same editor pattern, same PDF pipeline — but with PRP-specific sections and data sources.

## Current State Analysis

**Implemented (Stories 4.1 + 4.2):**
- `procedures` table: all PRP fields (injection site, laterality, diagnoses JSONB, consent, pain rating, procedure number in series, PRP prep, anesthesia, injection, post-procedure)
- `vital_signs` table: linked to `procedure_id`
- `getProcedureById` server action: returns procedure + nested `_vitals`
- `getPriorPrpProcedure`: returns prior procedure's date, pain_rating, procedure_number

**Existing AI/PDF infrastructure:**
- Anthropic SDK with tool_use pattern (`src/lib/claude/generate-initial-visit.ts`)
- Per-section regeneration pattern (`regenerateSection`)
- `@react-pdf/renderer` with `SectionBody` text parser, clinic header, signature block (`src/lib/pdf/initial-visit-template.tsx`)
- Finalization flow: render PDF → upload to `case-documents` bucket → insert `documents` row → link `document_id` to note (`src/actions/initial-visit-notes.ts`)

**Data sources for note generation:**
- `procedures` + `vital_signs` — the current procedure being documented
- `cases` + `patients` — patient header data
- `pain_management_extractions` (approved) — chief complaints, physical exam, diagnoses, treatment plan (note: no separate medication/allergy columns — those come from initial visit note)
- `mri_extractions` (approved/edited) — imaging findings
- `initial_visit_notes` (finalized, **required**) — `past_medical_history` (a structured text blob containing: Medical Problems, Surgeries, Medications Prior to Visit, Allergies — each as a labeled bullet), `social_history`; also signals case is fully evaluated before procedure note is generated
- Prior `procedures` record — prior pain rating for comparison (already fetched via `getPriorPrpProcedure`)
- `clinic_settings` + `provider_profiles` — letterhead and signature

**Missing:**
- `procedure_notes` table
- `generate-procedure-note.ts` AI function
- Zod schemas for note sections
- Server actions for procedure note lifecycle
- PDF template for procedure note
- UI editor + page
- Navigation from procedure list

## Desired End State

A "Generate Note" action in the procedure table row opens the note page at `/patients/[caseId]/procedures/[procedureId]/note`. The provider clicks "Generate", Claude produces all sections, the provider edits any section (with per-section regeneration), then finalizes — rendering a signed PDF and creating a case document record.

### Verification:
- Migration applies cleanly
- TypeScript builds without errors (`npm run build`)
- Note generates, saves, finalizes, and produces a downloadable PDF
- PDF contains clinic header, patient header, all sections, provider signature
- Finalized PDF appears in the case Documents tab

## Prerequisites

Generation requires a **finalized Initial Visit Note** for the case. This gate ensures:
1. The `past_medical_history` narrative (containing Medical Problems, Surgeries, Medications Prior to Visit, and Allergies as labeled bullets) and `social_history` are available in the `initial_visit_notes` record
2. The case has been fully clinically evaluated before a procedure note is generated
3. Consistent with the clinical workflow — Initial Visit precedes any injections

The `generateProcedureNote` server action and the UI generate button both enforce this gate. The empty state message clearly tells the provider what's missing.

## What We're NOT Doing

- PMH / medications / allergies / social history **input forms** — these come from the finalized Initial Visit Note's `past_medical_history` structured text (labeled bullets: Medical Problems, Surgeries, Medications Prior to Visit, Allergies) and `social_history`; we only read them
- Prior PT/chiro sections in the note — the ticket mentions "prior PT/chiro findings" as source data for the AI; we feed it as context but do not add dedicated note sections for them
- A separate `procedure_note_sections` table — sections are flat text columns on `procedure_notes`, same as `initial_visit_notes`
- A separate `patient_header` note section — patient identification is the opening sentence of the Subjective section; the structured header block in the PDF already contains patient name, DOB, indication, etc.
- Note versioning — same soft-delete + re-generate pattern as Initial Visit
- Medication-specific post-procedure ordering UI — handled as free text in the AI-generated section

## Phase 0: Auto-Populate ICD-10 Diagnoses from Initial Visit Note

### Overview
When recording a new procedure, auto-populate the diagnoses array with ICD-10 codes from the finalized Initial Visit Note. Currently `getCaseDiagnoses` only reads from `pain_management_extractions`; it must also parse ICD-10 codes from the `initial_visit_notes.diagnoses` text column and merge them. The procedure form should default to these diagnoses for new procedures (not edits).

### Changes Required:

#### 1. Enhance `getCaseDiagnoses` to include Initial Visit Note diagnoses
**File**: `src/actions/procedures.ts` — `getCaseDiagnoses` function (line 149)

Add a parallel fetch for the finalized initial visit note's `diagnoses` text column:

```typescript
export async function getCaseDiagnoses(caseId: string) {
  const supabase = await createClient()

  // Fetch PM extraction diagnoses and Initial Visit Note diagnoses in parallel
  const [pmRes, ivnRes] = await Promise.all([
    supabase
      .from('pain_management_extractions')
      .select('diagnoses')
      .eq('case_id', caseId)
      .eq('review_status', 'approved')
      .is('deleted_at', null)
      .order('created_at', { ascending: false })
      .limit(1)
      .maybeSingle(),
    supabase
      .from('initial_visit_notes')
      .select('diagnoses')
      .eq('case_id', caseId)
      .eq('status', 'finalized')
      .is('deleted_at', null)
      .maybeSingle(),
  ])

  const pmDiagnoses: Array<{ icd10_code: string | null; description: string }> =
    Array.isArray(pmRes.data?.diagnoses) ? pmRes.data.diagnoses : []

  // Parse ICD-10 codes from initial visit note diagnoses text
  // Format: "• M54.5 — Low back pain" or "M54.5 — Low back pain" per line
  const ivnDiagnoses: Array<{ icd10_code: string; description: string }> = []
  if (ivnRes.data?.diagnoses) {
    const lines = (ivnRes.data.diagnoses as string).split('\n')
    for (const line of lines) {
      // Match patterns like "• M54.5 — Description" or "M54.5 — Description" or "M54.5 - Description"
      const match = line.match(/^[•\-\d.]*\s*([A-Z]\d{1,2}\.?\d{0,4}[A-Z]{0,2})\s*[—–\-]\s*(.+)$/i)
      if (match) {
        ivnDiagnoses.push({ icd10_code: match[1].trim(), description: match[2].trim() })
      }
    }
  }

  // Merge: PM extraction first, then IVN codes not already present (dedup by icd10_code)
  const seen = new Set(pmDiagnoses.map((d) => d.icd10_code?.toUpperCase()))
  const merged = [
    ...pmDiagnoses,
    ...ivnDiagnoses.filter((d) => !seen.has(d.icd10_code.toUpperCase())),
  ]

  return { data: merged }
}
```

Global rules:
- PDF-safe formatting: unicode bullets (`• `), ALL CAPS sub-headings with colon, no markdown
- Clinical formal prose, no filler, no repetition across sections, ~6 pages target
- Do not repeat clinic name/address/provider credentials (rendered in PDF header/signature)
