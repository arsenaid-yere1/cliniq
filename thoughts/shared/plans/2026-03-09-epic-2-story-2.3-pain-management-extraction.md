# Epic 2, Story 2.3 — Pain Management Report Extraction: Implementation Plan

## Overview

Add AI-powered extraction of structured clinical data from uploaded pain management reports. When a user uploads a pain management PDF, the system automatically sends it to Claude API, extracts chief complaints, physical exam findings, diagnoses (with ICD-10 codes), range of motion data, and treatment plans into a structured format. The provider reviews, edits, approves, or rejects the extraction via the existing clinical data split-view workflow.

## Current State Analysis

The codebase already has two complete extraction pipelines (MRI + Chiro) that are structurally identical. Pain management follows the exact same pattern.

| Component | Location |
|-----------|----------|
| MRI extraction pipeline | `src/lib/claude/extract-mri.ts`, `src/actions/mri-extractions.ts` |
| Chiro extraction pipeline | `src/lib/claude/extract-chiro.ts`, `src/actions/chiro-extractions.ts` |

**What does NOT exist:**
- `pain_management` document type value
- `pain_management_extractions` database table
- Pain management extraction schemas, Claude module, server actions, or UI components

### Key Discoveries:
- The document type DB constraint is a `CHECK` in `002_case_dashboard_tables.sql:7` — needs an `ALTER TABLE` migration to add the new value
- The upload sheet selector at `upload-sheet.tsx:279-282` only shows MRI, Chiro, Other — needs a new `SelectItem`
- The clinical data page uses a `Tabs` component at `page.tsx:21-32` — needs a new "Pain Management" tab
- All extraction pipelines use tool-use API (not `response_format`) with `tool_choice: { type: 'tool', name: '...' }`
- The chiro extraction uses `max_tokens: 4096` due to larger schema — pain management will likely need the same or more given ROM tables
- The `normalizeNullString` / `normalizeNullStringsInArray` helpers in `extract-chiro.ts:155-174` handle Claude returning `"null"` strings

### Sample Document Structure (from real pain management report):
1. **Report metadata** — patient name, DOB, date of injury, exam date, provider
2. **History of the accident** — mechanism of injury narrative
3. **Chief complaints** — numbered list: location, pain rating (X/10), radiation, aggravating/alleviating factors
4. **Medical history** — past medical, surgeries, allergies, medications
5. **Physical examination** — per body region:
   - Palpation findings (tenderness grade, trigger points, spasms)
   - Range of motion table (movement, normal, actual, pain Y/N)
   - Orthopedic tests (name, positive/negative)
   - Neurological testing (motor grades, sensory status)
6. **Diagnostic studies** — referenced MRI/imaging findings
7. **Diagnoses** — ICD-10 codes with descriptions
8. **Treatment plan** — numbered recommendations with cost estimates

## Desired End State

After implementation:
1. `pain_management` is a valid document type in the upload selector
2. Uploading a pain management PDF triggers automatic extraction via Claude API
3. Extraction results are stored in `pain_management_extractions` table
4. The clinical data page has a "Pain Management" tab showing extractions
5. Clicking an extraction opens the split-view: PDF on left, editable data on right
6. Provider can approve, edit + save, or reject each extraction
7. Extraction failures show a manual entry fallback

### How to verify:
1. Upload a PDF with document type "Pain Management" → extraction runs automatically, toast shows success
2. Navigate to Clinical Data tab → "Pain Management" tab shows extraction with confidence badge
3. Click extraction → split-view shows PDF alongside extracted fields
4. Edit a field and save → `provider_overrides` contains the edit, `review_status` = `'edited'`
5. Approve → `review_status` = `'approved'`
6. Re-extract → old record soft-deleted, new extraction created

## What We're NOT Doing

- Extracting from scanned/image-based reports (OCR) — Claude handles PDFs natively
- Linking extracted diagnoses to chiro/MRI extraction diagnoses (cross-referencing)
- Auto-populating initial visit notes from extraction data
- Vitals extraction (blood pressure, heart rate, etc.)
- Detailed neurological exam extraction (motor/sensory tables) as structured tables — extracting as text summary instead, since these are typically normal/intact and the key value is in the chief complaints, diagnoses, and treatment plan

## Implementation Approach

Follow the established per-type extraction pattern (same as MRI/Chiro). Build bottom-up: database → schemas → Claude module → server actions → upload hook → UI.

    description: z.string().min(1, 'Description is required'),
  })),
  treatment_plan: z.array(treatmentPlanItemSchema.extend({
    description: z.string().min(1, 'Description is required'),

**File**: `src/lib/claude/extract-pain-management.ts` (new)

Follow the pattern from `src/lib/claude/extract-chiro.ts`. Key differences:

**System prompt** — tailored to pain management reports:
```
You are a medical data extraction assistant for a personal injury clinic.
You are extracting structured data from a pain management evaluation report.

RULES:
1. Extract ALL chief complaints with exact pain ratings as stated (e.g., "4-6/10" → min: 4, max: 6).
2. For physical examination, extract each body region separately with palpation findings, range of motion measurements, orthopedic tests, and neurological summary.
3. Range of motion: extract the Normal, Actual, and Pain columns exactly as shown in the report tables.
4. Orthopedic tests: extract each test name and whether it was positive or negative.
5. Extract ALL diagnosis codes exactly as written, including the ICD-10 7th character.
6. For treatment plan items, extract estimated costs when stated (e.g., "$3,000-10,500" → min: 3000, max: 10500).
7. Classify treatment types: "injection" for blocks/epidurals/PRP, "therapy" for physical/chiro/IDD, "medication" for prescriptions/OTC, "continuation" for continuing existing treatment, "monitoring" for re-evaluate/follow-up, "surgery" for surgical options, "alternative" for alternative approaches.
8. For diagnostic studies, summarize referenced imaging findings — do not re-extract MRI data in detail (that's handled by MRI extraction).
9. Return null for any field that cannot be determined from the document.
10. Set confidence to "low" for poor quality scans, partial documents, or ambiguous content.
11. Add extraction_notes for anything ambiguous or noteworthy.
```

**Tool schema** — `extract_pain_management_data` with properties matching `painManagementExtractionResultSchema`.

**Extraction function** — `extractPainManagementFromPdf(pdfBase64: string)` returning `{ data?: PainManagementExtractionResult; rawResponse?: unknown; error?: string }`.

Use `max_tokens: 4096` (pain management reports are detailed with multiple exam regions and ROM tables).

Normalization — use `normalizeNullString` and `normalizeNullStringsInArray` for nullable fields (same pattern as chiro). Nullable fields needing normalization:
- Top-level: `report_date`, `date_of_injury`, `examining_provider`, `diagnostic_studies_summary`, `extraction_notes`
- In `chief_complaints[]`: `radiation`
- In `physical_exam[]`: `palpation_findings`, `neurological_summary`
- In `diagnoses[]`: `icd10_code`
- In `treatment_plan[]`: `type`, `body_region`

## Testing Strategy

### Manual Testing Steps:
1. Upload a pain management report PDF → verify extraction triggers and completes
2. Upload a non-pain-management document → verify NO extraction triggers
3. Navigate to Clinical Data → verify "Pain Management" tab with extraction list
4. Open extraction → verify split-view with PDF and form
5. Verify chief complaints show correct pain ratings and factors
6. Verify physical exam regions show ROM tables with normal/actual/pain columns
7. Verify diagnoses show ICD-10 codes
8. Verify treatment plan shows recommendations with cost estimates
9. Approve without edits → verify status change
10. Edit fields and save → verify `provider_overrides`
11. Reject with reason → verify status and reason stored
12. Re-extract → verify old record soft-deleted
13. Manual entry on failed extraction → verify form saves correctly

### Edge Cases to Test:
- Pain management report with only 1 body region examined
- Report with no cost estimates in treatment plan
- Report referencing MRI findings but no standalone diagnostic studies section
- Report with multiple providers listed
- Very long treatment plan with many recommendations

## Performance Considerations

- Pain management reports are typically 5-7 pages — extraction should complete in 10-20 seconds
- ROM tables increase token count vs MRI reports — `max_tokens: 4096` should be sufficient
- The physical exam section is the densest part — the collapsible accordion UI keeps the review form manageable
