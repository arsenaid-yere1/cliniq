# Research: Create an Existing Case with Manual Documents, and Status Changes Unrestricted by Workflow State

**Date**: 2026-06-15T07:54:29-0700
**Researcher**: arsenaid
**Git Commit**: 01f671993dc9425d8782c4e9a25a7f569f746b82
**Branch**: main
**Repository**: cliniq

## Research Question
Add the ability to create an existing case with documents that were generated manually. Status changes should not be restricted by workflow state.

This document maps the codebase *as it exists today* across three areas relevant to that goal: (1) how cases are created, (2) how documents are created/stored, and (3) how case status and workflow-state restrictions are currently enforced. No recommendations — only the current state.

## Summary

- **Case creation** runs through one server action, `createPatientCase` ([src/actions/patients.ts:36](../../../src/actions/patients.ts)), reached from a 3-step UI wizard at route `/patients/new`. It supports two modes via a discriminated union: `new_patient` and `existing_patient`. In both modes the new case's `case_status` is **hardcoded to `'intake'`** and an initial `case_status_history` row is written. The caller cannot set status, case number, billing totals, or open/close dates at creation.

- **Documents** are tracked in one central `documents` table. There are two write paths: a **manual upload** path (`getUploadSession` → TUS upload → `saveDocumentMetadata`, [src/actions/documents.ts](../../../src/actions/documents.ts)) writing `status: 'pending_review'`, and **system-generated** PDF finalization paths (one per note type) writing `document_type: 'generated'`, `status: 'reviewed'`. A documents row is linked to a case solely by `case_id`; the minimal required fields are `case_id`, `file_path`, `document_type`.

- **Workflow-state restrictions** are enforced in several places: (a) a transition map `CASE_STATUS_TRANSITIONS` validated inside `updateCaseStatus`; (b) `LOCKED_STATUSES` checked by `assertCaseNotClosed`, which gates nearly every write action including both document write paths; (c) per-note generation/finalize guards and a `generation-lock`; (d) prerequisite checks (e.g. a medical invoice required before `pending_settlement`/`closed`). These are the constraints a "status changes unrestricted by workflow state" change would interact with.

## Detailed Findings

### Area 1 — Case Creation

**Data model.** `public.cases` is defined in [supabase/migrations/001_initial_schema.sql:73-94](../../../supabase/migrations/001_initial_schema.sql). Relevant columns: `case_number` (unique, auto-generated by trigger `generate_case_number()` at lines 99–110 producing `PI-YYYY-NNNN`), `patient_id`, `attorney_id`, `accident_date/description/type`, `case_status` (CHECK enum `intake|active|pending_settlement|closed|archived`, default `intake`, line 81), `case_open_date`, `case_close_date`, `lien_on_file`, `assigned_provider_id`, billing totals, audit columns. Audit table `case_status_history` at lines 115–123.

**Validation.** [src/lib/validations/patient.ts](../../../src/lib/validations/patient.ts) defines:
- `createPatientCaseSchema` (line 27) — new patient (identity + details)
- `createCaseForExistingPatientSchema` (line 29) — `details + patient_id`
- `createPatientCaseInputSchema` (line 33) — discriminated union on `mode: 'new_patient' | 'existing_patient'`

**Server action — `createPatientCase`** ([src/actions/patients.ts:36-126](../../../src/actions/patients.ts)):
- Parses via `createPatientCaseInputSchema.safeParse` (line 37); returns field errors on failure.
- `existing_patient` branch (lines 47–58): looks up patient by `patient_id` with `deleted_at IS NULL`; errors `'Patient not found'` if missing; sets `patientId = existing.id`. **No new patient insert.**
- `new_patient` branch (lines 59–88): inserts `public.patients`; `created_by_user_id`/`updated_by_user_id` hardcoded to `user?.id`.
- **Cases insert (lines 90–115).** Caller-settable: `accident_date`, `accident_type`, `accident_description`, `attorney_id`, `assigned_provider_id`, `lien_on_file`. **Hardcoded:** `patient_id` (derived), `case_status: 'intake'` (line 106), `created_by_user_id`, `updated_by_user_id`. `case_number` filled by DB trigger.
- **History insert (lines 118–122):** `case_id`, `new_status: 'intake'` (hardcoded), `changed_by_user_id`. No `previous_status`.
- Returns `{ data: caseRecord }`; `revalidatePath('/patients')`.

**UI / routes / entry points.**
- Page: [src/app/(dashboard)/patients/new/page.tsx](../../../src/app/(dashboard)/patients/new/page.tsx) — accepts `?patientId=` to pre-fill an existing patient (calls `getPatientForNewCase`), renders `<PatientWizard>`. Title `"New Case"` when existing patient.
- Wizard: [src/components/patients/patient-wizard.tsx](../../../src/components/patients/patient-wizard.tsx) — step Identity / Details / Review; calls `createPatientCase` at line 109; sets `mode` based on presence of an existing patient (lines 110–112); on success navigates to `/patients/${result.data.id}`.
- Entry buttons → `/patients/new`: people-list-page-client.tsx:112, patient-list-page-client.tsx:65, patient-detail.tsx:166 (`?patientId=`), case-overview.tsx:206 (`?patientId=`).

**Tests.** [src/actions/__tests__/patients.test.ts](../../../src/actions/__tests__/patients.test.ts) (createPatientCase both modes), [src/lib/validations/__tests__/patient.test.ts](../../../src/lib/validations/__tests__/patient.test.ts).

> Note re. the query phrase "create an existing case": today's `existing_patient` mode creates a *new case for an existing patient*. There is no current path that imports/back-dates a case into a non-`intake` status, nor one that sets case status at creation time.

### Area 2 — Document Creation & Storage

**Central table.** `documents` — [supabase/migrations/002_case_dashboard_tables.sql](../../../supabase/migrations/002_case_dashboard_tables.sql) (line 4), extended in `003_document_storage.sql`. `document_type` enum values (validation): `mri_report, chiro_report, pain_management, pt_report, orthopedic_report, ct_scan, x_ray, generated, lien_agreement, procedure_consent, other` ([src/lib/validations/document.ts:13](../../../src/lib/validations/document.ts)). `file_path` is a path within the private `case-documents` bucket.

**Storage buckets.** `case-documents` (private; `003_document_storage.sql`; path pattern `cases/{caseId}/...`) and `clinic-assets` (logo/signatures; `009_clinic_assets_storage.sql`).

**Minimal valid documents row** (from `saveDocumentMetadata` insert, [src/actions/documents.ts:148-159](../../../src/actions/documents.ts)): required to link to case = `case_id` + `file_path` + `document_type`. In practice both paths also set `file_name`, `file_size_bytes`, `mime_type`, `status`, and the three user-id audit columns. There is no FK from `documents` to a note/procedure — the reverse link is `note.document_id → documents.id`.

**Manual upload path:**
- `getUploadSession` ([documents.ts:96-127](../../../src/actions/documents.ts)): validates `documentUploadMetaSchema` (caseId, fileName, fileSize ≤50MB, mimeType ∈ ALLOWED_MIME_TYPES, documentType); auth; case-exists check; **`assertCaseNotClosed` guard (line 115)**; sanitizes filename; returns storage path `cases/{caseId}/{Date.now()}-{name}`. No row inserted here.
- `saveDocumentMetadata` ([documents.ts:129-176](../../../src/actions/documents.ts)): auth; **`assertCaseNotClosed` (line 141)**; `autoAdvanceFromIntake` (line 144); inserts row with `status: 'pending_review'`; if `lien_agreement` sets `cases.lien_on_file = true`; revalidates docs path. Note: `documentType` is **not** re-validated against the enum at this step.

**System-generated (finalize) path** — pattern shown by `finalizeInitialVisitNote` ([src/actions/initial-visit-notes.ts:652-731](../../../src/actions/initial-visit-notes.ts)):
- Guards: auth; **`assertCaseNotClosed` (line 657)**; requires an existing `initial_visit_notes` row with `status='draft'` (lines 661–668) else `'No draft note found to finalize'`.
- Soft-deletes any prior linked document (`finalize-document.ts`), renders PDF, uploads to `case-documents` (`upsert:false`), inserts documents row `document_type:'generated'`, `status:'reviewed'`, then updates note `status: draft → finalized`, sets `finalized_by/at`, `document_id`.
- Equivalent finalize actions exist for procedure-notes.ts (line 855), discharge-notes.ts (line 914), clinical-orders.ts (line 282); plus non-Claude generators lien.ts and procedure-consents.ts that also write documents rows.

**AI generation functions** live in [src/lib/claude/](../../../src/lib/claude/) (`generate-initial-visit.ts`, `generate-procedure-note.ts`, `generate-discharge-note.ts`, `generate-summary.ts`, `generate-clinical-orders.ts`, `generate-quality-review.ts`). PDF renderers in [src/lib/pdf/](../../../src/lib/pdf/). AI-note content tables: `initial_visit_notes` (010), `procedure_notes` (015), `discharge_notes` (016), `case_summaries` (006), `clinical_orders`, `case_quality_reviews` — each carries `status`, `raw_ai_response jsonb`, `source_data_hash`, `document_id`.

**Listing/filtering.** `listDocuments` ([documents.ts:8-81](../../../src/actions/documents.ts)) filters by `document_type`/`status`/search; for `generated` rows it enriches with `content_date`/`procedure_number` via parallel lookups against the four note tables.

### Area 3 — Status / Workflow-State Restrictions

**Case status constants** — [src/lib/constants/case-status.ts](../../../src/lib/constants/case-status.ts):
- `CASE_STATUSES = [intake, active, pending_settlement, closed, archived]` (line 1).
- `CASE_STATUS_TRANSITIONS` (lines 16–22) — allowed next states:
  | From | Allowed targets |
  |---|---|
  | intake | active, closed |
  | active | pending_settlement, closed |
  | pending_settlement | closed, active |
  | closed | active, archived |
  | archived | closed |

  No path returns to `intake`; `archived` cannot jump directly to `active`.
- `LOCKED_STATUSES = [pending_settlement, closed, archived]` (line 24).
- `CASE_STATUS_CONFIG` label/color map (lines 4–14).

**Enforcement — [src/actions/case-status.ts](../../../src/actions/case-status.ts):**
- `assertCaseNotClosed` (lines 9–25): fetches `case_status`; if in `LOCKED_STATUSES` returns a lock error. **Called as a write guard across the app** (documents.ts, billing.ts lines 398/467/538, and the note generate/finalize actions).
- `updateCaseStatus` (lines 29–104): auth; same-status guard; **transition validated against `CASE_STATUS_TRANSITIONS` (lines 51–54)**; **prerequisite (lines 57–70): moving to `pending_settlement` or `closed` requires ≥1 non-void `invoice_type='visit'` invoice**; sets `case_close_date` for closed/archived; updates case; inserts `case_status_history`.
- `autoAdvanceFromIntake` (lines 108–134): if status is exactly `intake`, directly updates to `active` (**bypasses** `updateCaseStatus` validation and prerequisites), writes history `'Auto-advanced: first clinical activity'`. Called from documents.ts, procedures.ts, and the three note actions on first write.
- `closeCase` (138) / `reopenCase` (142): thin wrappers over `updateCaseStatus` (so both pass through full transition validation).

**Other workflow guards:**
- Invoice state machine: [src/lib/constants/invoice-status.ts](../../../src/lib/constants/invoice-status.ts) (`ALLOWED_TRANSITIONS`, `TERMINAL_STATUSES`, `canTransitionTo`) enforced in [src/actions/invoice-status.ts](../../../src/actions/invoice-status.ts) `transitionInvoiceStatus`. Only `draft` invoices editable/deletable (billing.ts:471/542).
- Note generation lock: [src/lib/supabase/generation-lock.ts](../../../src/lib/supabase/generation-lock.ts) `acquireGenerationLock` — CAS on `status`, 5-min stale reclaim; lifecycle `draft|failed → generating → draft|finalized|failed`.
- Per-note prerequisites: procedure regen blocked if `finalized` (procedure-notes.ts:598); discharge generation requires a finalized initial-visit note (discharge-notes.ts:606–613); discharge finalize requires provider vitals (931–933); procedure finalize requires plan-deviation acknowledgment (884).
- QC step model: `qcStepValues` and `FIXABLE_STEPS` ([src/lib/validations/case-quality-review.ts:7-15,148-193](../../../src/lib/validations/case-quality-review.ts)).

**DB-level constraints** mirror the app enums: `cases.case_status` CHECK (001_initial_schema.sql:81); note tables CHECK `('generating','draft','finalized','failed')` (010/015/016); invoice CHECK (020).

## Architecture Documentation
- **Single creation entry point**: all case creation flows through one server action and one wizard; mode is selected by presence of an existing patient. Status is server-controlled, never client-set.
- **Documents are case-scoped records**: one `documents` table for both uploaded and generated artifacts, distinguished by `document_type` (`'generated'` vs the upload types) and `status` (`'reviewed'` vs `'pending_review'`). Generated PDFs are produced by a render layer and back-linked from note tables via `document_id`.
- **Workflow state is enforced in three layers**: app constants (transition maps), server-action guards (`assertCaseNotClosed` as a near-universal write gate; transition validation + prerequisites in `updateCaseStatus`), and DB CHECK constraints. `autoAdvanceFromIntake` is the one place that intentionally bypasses `updateCaseStatus` validation.

## Follow-up Research 2026-06-15T08:10 — deep-dive + recommended approach

### New findings

**A role/permission system already exists.**
- `src/lib/auth/require-role.ts` — role enum `'admin' | 'provider' | 'staff'`; `requireAdmin()` (lines 30–36) re-fetches the caller's `users.role` and throws/returns error if not admin. `users` table has `role text not null default 'staff' check (role in ('admin','provider','staff'))` (001_initial_schema.sql:11) and `is_active` (20260520_user_management.sql:3).
- `src/actions/users.ts` is the existing template for admin-gated server actions: `listUsers`/`inviteUser`/`updateUserRole`/`setUserActive` all start by calling the role guard (lines 20, 36, 88, 106).

**The DB does NOT enforce the workflow at all — restrictions live only in TS server actions.**
- `cases` RLS is a single `FOR ALL USING (auth.role() = 'authenticated')` policy (001_initial_schema.sql:184) — no `case_status` condition, no INSERT/UPDATE split, no role split. Same for `documents` (002:121) and the `case-documents` bucket (003:17–35).
- No trigger forces initial `case_status='intake'`. `case_status` is settable to any of the 5 enum values on INSERT; only `case_number` is locked (BEFORE INSERT trigger, cannot override). Minimal `cases` insert = `patient_id` only.
- Minimal `documents` insert = `case_id`, `document_type`, `file_name`, `file_path`. `status` defaults `'pending_review'`; `documents.status` CHECK is only `('pending_review','reviewed')`. No tie to parent `case_status`.

**Back-dating is already unrestricted** — `accident_date`, `case_open_date`, note `visit_date` have format-only validation, no temporal bounds.

**No import/seed mechanism exists** — no `scripts/`, no seed files, no bulk-insert action. Case creation only via the wizard + `createPatientCase`.

**Status-change UI is fully data-driven by the constraint maps.**
- `src/components/patients/status-change-dropdown.tsx` builds its menu from `CASE_STATUS_TRANSITIONS[currentStatus]` (line 36); returns `null` if empty (line 37). Server `updateCaseStatus` re-validates the same map (case-status.ts:50–54).
- `case-overview.tsx:79` computes `isLocked` from `LOCKED_STATUSES`; locked → quick-action buttons disabled + banner (157–178). `document-list.tsx:64` does the same via `useCaseStatus()` context to disable upload.

### Interpretation of the request

Mapping the three open questions to what exists:
1. **"create an existing case"** = create a case that is *already past intake* (e.g. an `active`/`closed` historical case being entered into the system after the fact), for a new or existing patient — not just the current always-`intake` flow.
2. **"with documents generated manually"** = attach documents to that case via the existing manual **upload** path (`document_type` ∈ upload types, `status:'pending_review'`), rather than the AI generate→finalize chain. The upload path already exists; the only thing blocking it on a back-entered non-active case is `assertCaseNotClosed`.
3. **"status changes not restricted by workflow state"** = allow setting/changing status outside `CASE_STATUS_TRANSITIONS` and bypassing the `LOCKED_STATUSES` write-gate and invoice prerequisite.

### Recommended approach (an admin "manual/override" capability, additive and non-breaking)

Rationale for the shape: every restriction is in TS, the DB is already permissive, an admin-role guard pattern already exists, and the UI is data-driven — so the cleanest change is an **admin-gated override that loosens the TS guards without touching the DB schema, RLS, or the default flow**. This matches the existing `requireAdmin` pattern and the project's preference for additive/non-breaking changes ([[feedback_phased_nonbreaking_ui]]).

Four pieces:

1. **Allow status at creation.** Extend `createPatientCaseInputSchema` with an optional `case_status` (default `'intake'`). In `createPatientCase` (patients.ts:106), use the supplied status instead of the hardcoded literal; write `case_status_history` with that status. Gate any non-`intake` value behind `requireAdmin()` so the normal flow is unchanged. No migration needed (column + CHECK already accept all 5 values).

2. **Allow arbitrary status transitions for admins.** In `updateCaseStatus` (case-status.ts), when the caller is admin and an `override` flag is passed, skip the `CASE_STATUS_TRANSITIONS` check (50–54) and the invoice prerequisite (57–70). Keep the history insert (audit trail preserved). Non-admin / non-override path unchanged. Optionally record `notes: 'admin override'`.

3. **Allow document upload on locked/back-entered cases for admins.** `assertCaseNotClosed` gates `getUploadSession` (documents.ts:115) and `saveDocumentMetadata` (141). Add an admin-override bypass at these two call sites (or thread an `allowLocked` param) so manual documents can be attached to a `closed`/`archived`/back-entered case. The upload mechanism itself needs no change.

4. **UI surface (admin-only).** Add an "Advanced / admin" affordance: free-status select on the new-case wizard, and an override toggle on `status-change-dropdown.tsx` that offers all 5 statuses (not just `CASE_STATUS_TRANSITIONS`) and disables the lock banner's button-disabling when override is on. Render these only when the session user is admin.

**What this deliberately does not do:** no schema migration, no RLS change (DB already permits it), no change to default non-admin behavior, no new import/seed pipeline. `autoAdvanceFromIntake` already demonstrates the codebase pattern of bypassing `updateCaseStatus` validation, so an admin override is consistent with existing practice.

**Smaller alternative** if no role-gating is wanted: relax the constants directly (make `CASE_STATUS_TRANSITIONS` allow any→any, drop the invoice prerequisite, neuter `assertCaseNotClosed`). Lower effort but removes the guardrails for *all* users and is harder to walk back — not recommended given a role system already exists.
