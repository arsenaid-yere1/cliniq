# Research: Epic 3, Story 3.1 — Initial Visit Note Generation

**Date**: 2026-03-09T14:00:00-07:00
**Researcher**: Claude
**Git Commit**: a5cf091923151d6459703679b5f16c9dbebfc4ab
**Branch**: main
**Repository**: cliniq

## Research Question

What is the best design and implementation approach for Story 3.1 — Generate Initial Visit Draft? The feature should produce a clinical Initial Visit note with sections for patient info, chief complaint, history of present illness, imaging review, prior treatment summary, physical exam template, assessment, and treatment plan. The provider must be able to edit text before finalizing.

## Summary

The recommended design is a **two-stage approach**: (1) AI generates a structured Initial Visit note from existing case data (case summary, extractions, patient demographics, clinic/provider info), stored in a new `initial_visit_notes` table, and (2) the provider edits the note in a **section-based rich text editing interface** before finalizing. This builds directly on the established AI generation → provider review pattern from Story 2.3 (case summaries) but differs in two key ways:

1. **The output is a clinical document** (not a structured data card) — each section contains narrative prose, not structured JSONB fields
2. **The editing experience is richer** — providers need to edit paragraph-level clinical text, not just tweak individual fields in a form

The approach uses a **section-based Textarea editor** (no rich text library needed for MVP) where each note section is an independently editable `<Textarea>` within a structured form. This avoids the complexity of introducing a rich text editor while still giving providers full control over every section's content.

## Detailed Findings

### 1. Data Sources Already Available

The Initial Visit note can be generated almost entirely from data that already exists in the system. Here's the mapping from acceptance criteria sections to data sources:

| Note Section | Data Source | Status |
|---|---|---|
| **Patient info** | `patients` table (name, DOB, gender) + `cases` table (accident date/type, case number) | Available |
| **Chief complaint** | `case_summaries.chief_complaint` (AI-synthesized) or `cases.accident_description` | Available (if summary generated) |
| **History of present illness** | `case_summaries.symptoms_timeline` + `cases.accident_description` + `cases.accident_date` | Available (if summary generated) |
| **Imaging review** | `case_summaries.imaging_findings` or raw `mri_extractions` data | Available |
| **Prior treatment summary** | `case_summaries.prior_treatment` or raw `chiro_extractions` data | Available |
| **Physical exam template** | **No source data** — this is a blank template populated from provider defaults | New (template-based) |
| **Assessment** | `case_summaries.suggested_diagnoses` | Available (if summary generated) |
| **Treatment plan** | **No source data** — provider fills this in manually | New (template-based) |

**Key insight**: 6 of 8 sections can be pre-populated from existing data. Physical exam and treatment plan are template sections that the provider fills in during the visit.

### 2. How This Differs from Case Summary (Story 2.3)

| Aspect | Case Summary (2.3) | Initial Visit Note (3.1) |
|---|---|---|
| **Output format** | Structured JSONB fields (arrays, objects) | Narrative prose per section |
| **Display** | Data card with badges, lists, grids | Document-style flowing text |
| **Editing** | Structured form with field arrays | Free-text editing per section |
| **Storage** | JSONB columns per field | Text columns per section |
| **AI role** | Synthesize structured data | Generate clinical narrative prose |
| **Template sections** | None — all AI-generated | Physical exam + treatment plan are templates |
| **Finalization** | Approve/reject | Finalize (locks for signing) |
| **Provider signature** | Not included | Should be included on final document |

- Both accessible via `settings.ts` server actions

### 4. What's Missing from the Schema

The following **do not exist** and would need to be created:

1. **`initial_visit_notes` table** — to store the generated note with per-section text columns
2. **Physical exam template data** — either hardcoded in the prompt or stored as a clinic-level setting
3. **Treatment plan template data** — same as above

## Recommended Design

### Option Analysis

| Option | Description | Pros | Cons |
|---|---|---|---|
| **A. AI generates full narrative → single rich text editor** | One big document, edit with Tiptap/Quill | Most natural document editing | Requires new dependency (Tiptap), complex to implement, harder to regenerate individual sections |
| **B. AI generates per-section narrative → section-based Textarea form** | Each section is a Textarea in a form, displayed as a document when finalized | Fits existing patterns, no new dependencies, section-level regeneration possible | Slightly less "document-like" editing feel |
| **C. AI generates structured data → template rendering** | Store structured data, render through HTML/PDF template | Clean separation of data and presentation | Double the work (structured schema + template), over-engineered for MVP |

### Recommendation: Option B — Section-Based Textarea Form

This is the best fit for MVP because:

1. **No new dependencies** — uses existing Textarea + react-hook-form + Zod stack
2. **Section-level control** — provider can edit any section independently; can regenerate a single section if needed
3. **Matches existing patterns** — the form architecture is identical to what's used throughout the app
4. **Progressive enhancement path** — can upgrade individual sections to rich text (Tiptap) in the future without redesigning the data model
5. **Physical exam template** fits naturally as a pre-populated Textarea section

### Architecture

#### 1. New Database Table: `initial_visit_notes`

```sql
CREATE TABLE public.initial_visit_notes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  case_id UUID NOT NULL REFERENCES public.cases(id),

  -- Section text fields (narrative prose)
  patient_info TEXT,
  chief_complaint TEXT,
  history_of_present_illness TEXT,
  imaging_review TEXT,
  prior_treatment_summary TEXT,
  physical_exam TEXT,
  assessment TEXT,
  treatment_plan TEXT,

  -- AI metadata
  ai_model TEXT,
  raw_ai_response JSONB,

  -- Note status
  status TEXT NOT NULL DEFAULT 'draft'
    CHECK (status IN ('generating', 'draft', 'finalized', 'failed')),
  generation_error TEXT,
  source_data_hash TEXT,

  -- Provider finalization
  finalized_by_user_id UUID REFERENCES public.users(id),
  finalized_at TIMESTAMPTZ,

  -- Standard audit fields
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_by_user_id UUID REFERENCES public.users(id),
  updated_by_user_id UUID REFERENCES public.users(id),
  deleted_at TIMESTAMPTZ
);

-- One active note per case
CREATE UNIQUE INDEX idx_initial_visit_notes_case_active
  ON public.initial_visit_notes(case_id) WHERE deleted_at IS NULL;
```

**Key differences from `case_summaries`**:
- **Text columns instead of JSONB** — each section is narrative prose, not structured data
- **Simpler status model** — `generating` → `draft` → `finalized` (no `pending_review`/`approved`/`edited`/`rejected`)
- **No `provider_overrides`** — the provider edits the text columns directly (the note IS the editable artifact)
- **`finalized_by/at`** — tracks who locked the note and when

#### 2. Claude Integration: `src/lib/claude/generate-initial-visit.ts`

The AI generates narrative prose for each section using the same tool-use pattern:

- **Input**: Case summary data (or raw extractions if no summary), patient demographics, accident details, clinic/provider info
- **Output**: 8 text strings (one per section)
- **Physical exam**: AI generates a standard template based on the body regions identified in imaging (e.g., cervical spine exam template for cervical MRI findings)
- **Treatment plan**: AI generates a preliminary plan based on diagnoses and prior treatment, clearly marked as a draft for provider modification

The system prompt should instruct Claude to write in clinical note format appropriate for personal injury documentation, with precise medical terminology suitable for legal proceedings.

**Visit framing (important)**: The note must be framed as an **initial medical evaluation visit** — not a pain management visit. The introduction, patient info, and chief complaint sections should explicitly position the encounter as a comprehensive medical evaluation of injuries sustained from the accident (diagnostic assessment, examination, and care planning). This framing matters because:

- "Pain management" implies an ongoing treatment modality (often interventional) with different clinical and legal connotations
- Personal injury documentation requires an evaluative posture: establishing causation, documenting injuries, and forming a diagnostic impression
- The system prompt must include explicit instructions to use "medical evaluation" / "initial evaluation" language and avoid "pain management" phrasing in the introduction and chief complaint

Example prompt language: *"This is an initial medical evaluation visit. Frame the note as a comprehensive medical evaluation of the patient's injuries from the accident. Do NOT characterize this as a pain management visit or use pain management terminology in the introduction or chief complaint."*

### 7. Prerequisite Strategy

The note generation should be flexible about prerequisites:
- **Ideal**: Case summary exists and is approved → best quality note
- **Acceptable**: At least one approved MRI or chiro extraction → generate from raw extraction data
- **Minimum**: Just case demographics and accident details → generate a skeleton with template sections filled in and data sections marked "[To be completed after imaging review]"

This progressive approach means providers can generate a note at any point in the case lifecycle, not just after all extractions are complete.

## Architecture Insights

1. **Text columns over JSONB**: Unlike case summaries which store structured data for display in cards/badges, the Initial Visit note stores narrative prose. Plain TEXT columns are the right choice — they're simpler to query, edit, and don't need structured parsing.

2. **No `provider_overrides` needed**: The case summary pattern uses `provider_overrides` to preserve the original AI output while allowing edits. For the Initial Visit note, the provider edits the text directly. The `raw_ai_response` JSONB column preserves the original AI output for audit purposes.

3. **Dedicated page, not a card**: The case summary fits as a card on the overview page because it's compact structured data. The Initial Visit note is a multi-section clinical document that needs its own page with space for reading and editing.

4. **`finalized` replaces `approved`**: The note uses `finalized` instead of `approved` because finalization implies the note is locked and ready for the patient record — a stronger commitment than approval.

5. **Progressive prerequisite model**: Rather than blocking generation until all data is available, the AI should gracefully handle partial data and produce the best possible note. This matches clinical workflow where providers may need to start documentation before all results are in.

6. **No PDF export in Story 3.1**: The acceptance criteria say "Provider can edit text before finalizing" — this is about the editing workflow, not PDF generation. PDF/print export can be a follow-up story.

## Open Questions

1. **Should the note auto-save as the provider types?** Debounced auto-save (like Google Docs) vs. explicit "Save Draft" button. Recommendation: Explicit save for MVP (matches existing form patterns), auto-save as a future enhancement.

2. **Should there be a "Regenerate Section" button per section?** This would allow regenerating just the imaging review if new MRI data is added, without losing edits to other sections. Recommendation: Yes, include per-section regenerate — it's a significant UX win and not complex to implement.

3. **Physical exam template content**: Should the template vary by case type (auto accident vs. slip and fall vs. workplace)? Recommendation: Yes — the AI should generate a body-region-specific exam template based on the imaging findings. An auto accident with cervical MRI findings gets a cervical spine exam template; a slip and fall with lumbar findings gets a lumbar template.

4. **Should the finalized note be stored as a document in the `documents` table?** This would make it appear in the document list with type `'generated'`. Recommendation: Yes — after finalization, create a `documents` row with `document_type: 'generated'` that links back to the note. This keeps the document repository complete.

5. **Signature on finalized note**: Should the provider's signature image be embedded when the note is finalized? Recommendation: Yes — display in the finalized view. The signature image is already stored in Supabase storage (`provider_profiles.signature_storage_path`).

6. **Visit framing — medical evaluation vs. pain management**: Resolved. The note is framed as an **initial medical evaluation visit**. The system prompt must explicitly instruct Claude to use medical evaluation language and avoid pain management terminology (see Section 2 above). This is a hard requirement, not a stylistic preference, because of the legal and clinical implications in personal injury documentation.
