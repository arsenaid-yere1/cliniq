## Research Question
Is there any codebase check or guard that requires (or is tied to) an invoice being generated before a case can be closed? Document the case-close flow, the invoice-generation flow, and any linkage between them.

## Summary

Case-close prerequisite is **a medical invoice** (row in `invoices` with `invoice_type = 'visit'` for the case, not soft-deleted). Applies to transitions into `pending_settlement` and `closed`.

Prior behavior required a finalized discharge note; that check was swapped out for the medical-invoice check on 2026-04-22.

The reverse-direction relationship is unchanged: once a case enters a locked status (`closed` or `archived`), invoice writes (create/update/delete) are blocked by the shared `assertCaseNotClosed` guard.

- Close prerequisite: `invoices` row with `invoice_type='visit'` for the case — [src/actions/case-status.ts:55-68](src/actions/case-status.ts#L55-L68)
- Close wrapper: `closeCase()` → `updateCaseStatus(caseId, 'closed')` — [src/actions/case-status.ts:136-138](src/actions/case-status.ts#L136-L138)
- Transition map: allowed `from → to` status moves — [src/lib/constants/case-status.ts:17-23](src/lib/constants/case-status.ts#L17-L23)
- Reverse-direction guard: invoice writes fail on closed cases — [src/actions/billing.ts:367-368](src/actions/billing.ts#L367-L368), [src/actions/billing.ts:436-437](src/actions/billing.ts#L436-L437), [src/actions/billing.ts:507-508](src/actions/billing.ts#L507-L508)

## Detailed Findings

### Case status model

Allowed statuses and transitions are defined in [src/lib/constants/case-status.ts](src/lib/constants/case-status.ts):

```typescript
export const CASE_STATUSES = ['intake', 'active', 'pending_settlement', 'closed', 'archived'] as const

export const CASE_STATUS_TRANSITIONS: Record<CaseStatus, CaseStatus[]> = {
  intake:             ['active', 'closed'],
  active:             ['pending_settlement', 'closed'],
  pending_settlement: ['closed', 'active'],
  closed:             ['active', 'archived'],
  archived:           ['closed'],
}

export const LOCKED_STATUSES: CaseStatus[] = ['closed', 'archived']
```

- `closed` is reachable from `intake`, `active`, `pending_settlement`, and `archived`.
- `LOCKED_STATUSES` powers the reverse-direction guard that blocks invoice/other writes after closure.

### Case-close path

File: [src/actions/case-status.ts](src/actions/case-status.ts)

The unified entry point is `updateCaseStatus(caseId, newStatus, notes?)`. `closeCase()` is a thin wrapper that calls it with `'closed'`:

```typescript
// src/actions/case-status.ts:136-138
export async function closeCase(caseId: string) {
  return updateCaseStatus(caseId, 'closed')
}
```

Inside `updateCaseStatus` the checks run in this order:
1. Auth — user must be logged in.
2. Case must exist and not be soft-deleted.
3. No-op guard: `currentStatus === newStatus` returns an error.
4. Transition must be in `CASE_STATUS_TRANSITIONS[currentStatus]`.
5. **Prerequisite gate** (lines 55-68):

   ```typescript
   if (newStatus === 'pending_settlement' || newStatus === 'closed') {
     const { data: medicalInvoice } = await supabase
       .from('invoices')
       .select('id')
       .eq('case_id', caseId)
       .eq('invoice_type', 'visit')
       .is('deleted_at', null)
       .maybeSingle()

     if (!medicalInvoice) {
       return { error: 'A medical invoice is required before changing to this status.' }
     }
   }
   ```

6. Build payload: sets `case_status`, `updated_by_user_id`, and sets `case_close_date` to today when moving to `closed`/`archived` (or nulls it otherwise).

"Medical invoice" maps to `invoices.invoice_type = 'visit'` per the enum in [016_invoices / 017_invoice_enhancements migrations](supabase/migrations/017_invoice_enhancements.sql) and the Zod schema in [src/lib/validations/invoice.ts:16](src/lib/validations/invoice.ts#L16) (`z.enum(['visit', 'facility'])`). The alternative type `'facility'` does not satisfy the gate.

### Shared guard `assertCaseNotClosed`

File: [src/actions/case-status.ts:9-24](src/actions/case-status.ts#L9-L24)

```typescript
export async function assertCaseNotClosed(
  supabase: Awaited<ReturnType<typeof createClient>>,
  caseId: string,
): Promise<{ error: string | null }> {
  const { data } = await supabase
    .from('cases')
    .select('case_status')
    .eq('id', caseId)
    .is('deleted_at', null)
    .single()

  if (data?.case_status && LOCKED_STATUSES.includes(data.case_status as CaseStatus)) {
    return { error: 'This case is closed. No modifications are allowed until it is reopened.' }
  }
  return { error: null }
}
```

Returns the error message `"This case is closed. No modifications are allowed until it is reopened."` when the case's `case_status` is in `LOCKED_STATUSES` (`['closed', 'archived']`).

## Architecture Documentation

- **Case status is a linear-ish state machine** defined in code (`CASE_STATUS_TRANSITIONS`), validated server-side in `updateCaseStatus` before any DB write.
- **Close prerequisite is billing-centric.** Closing (or moving to `pending_settlement`) requires a medical invoice (`invoice_type='visit'`) for the case. The gate is one-directional at the prereq stage, but the system still blocks invoice writes after closure via `assertCaseNotClosed`, so both directions are guarded — just on different artifacts.
- **One shared guard (`assertCaseNotClosed`)** is reused at the top of every mutating billing action. The invariant is: locked cases are read-only. This is enforced in application code only — no DB trigger on `invoices` was found in the migrations referenced during research.
- **`closeCase`/`reopenCase` are thin wrappers** over `updateCaseStatus`. The whole rule-set (transitions + prerequisites + `case_close_date` bookkeeping) lives in one function.
- **`case_close_date`** is set to today when moving to `closed` or `archived`, and nulled on any other transition.
