# Research: Story 2.2 — Chiropractor Report Extraction Design & Implementation

**Date**: 2026-03-06T18:00:00-07:00

## Research Question
Design the best implementation approach for extracting structured data from chiropractor/conservative care reports, building on the existing MRI extraction architecture.

## Summary

Story 2.2 requires extracting five categories from chiropractic PI reports: treatment dates, diagnosis, treatment modalities, functional outcomes, and plateau statements. The existing MRI extraction pipeline (Story 2.1) provides a proven, reusable pattern. The recommended approach mirrors the MRI architecture exactly — new DB table, new Claude prompt/tool, new server action, new review UI — while accounting for chiropractic reports being structurally more complex (multiple visits, SOAP notes, outcome scores, MMI determinations).

## Detailed Findings

### 1. Existing Architecture to Follow

The MRI extraction pipeline is the template. It follows this flow:

```
Upload (document_type='chiro_report')
  → saveDocumentMetadata()
  → extractChiroReport(documentId)     [new server action]
    → soft-delete old chiro_extractions
    → insert chiro_extractions (status='processing')
    → download PDF from case-documents bucket
    → check for /Encrypt marker
    → base64 encode → Claude API call
    → validate with Zod → store results
    → revalidatePath()
```

- [extract-mri.ts](src/lib/claude/extract-mri.ts) → `extract-chiro.ts`
- [mri-extraction.ts](src/lib/validations/mri-extraction.ts) → `chiro-extraction.ts`

### 2. Chiropractic Report Complexity vs MRI

| Aspect | MRI Report | Chiro Report |
|--------|-----------|--------------|
| Document length | 2-10 pages | 5-50+ pages |
| Structure | Single study, standardized | Multiple visit types, variable |
| Data model | Fixed fields + findings array | Treatment timeline + visits + outcomes |
| Key challenge | Per-level findings | Multi-visit aggregation |
| Report subtypes | None (always same format) | Initial eval, SOAP notes, re-eval, discharge |

**Implication**: The chiro extraction schema must be richer and handle the possibility of a compiled chart (multiple visits in one PDF).

### 3. Recommended Extraction Schema

Based on the acceptance criteria and PI domain research, here is the recommended structured output schema:

```typescript
// What Claude extracts
const chiroExtractionResultSchema = z.object({
  // Report metadata
  report_type: z.enum([
    'initial_evaluation',
    'soap_note',
    're_evaluation',
    'discharge_summary',
    'other'
  ]),
  report_date: z.string().nullable(),  // ISO 8601

  // Treatment dates (Acceptance Criteria #1)
  treatment_dates: z.object({
    first_visit: z.string().nullable(),     // ISO 8601
    last_visit: z.string().nullable(),      // ISO 8601
    total_visits: z.number().nullable(),
    visit_dates: z.array(z.string()),       // All dates found
    treatment_gaps: z.array(z.object({      // Gaps > 14 days
      from: z.string(),
      to: z.string(),
      days: z.number(),
    })),
  }),

  // Diagnosis (Acceptance Criteria #2)
  diagnoses: z.array(z.object({
    icd10_code: z.string().nullable(),      // e.g., "S13.4XXA"
    description: z.string(),                 // e.g., "Cervical sprain"
    region: z.enum([
      'cervical', 'thoracic', 'lumbar', 'sacral',
      'upper_extremity', 'lower_extremity', 'other'
    ]).nullable(),
    is_primary: z.boolean(),
  })),

  // Treatment modalities (Acceptance Criteria #3)
  treatment_modalities: z.array(z.object({
    modality: z.string(),                    // e.g., "Spinal manipulation"
    cpt_code: z.string().nullable(),         // e.g., "98941"
    regions_treated: z.array(z.string()),    // e.g., ["cervical", "thoracic"]
    frequency: z.string().nullable(),        // e.g., "3x/week"
  })),

  // Functional outcomes (Acceptance Criteria #4)
  functional_outcomes: z.object({
    pain_levels: z.array(z.object({
      date: z.string().nullable(),
      scale: z.string(),                     // "NRS" or "VAS"
      score: z.number(),
      max_score: z.number(),
      context: z.string().nullable(),        // "initial", "discharge", etc.
    })),
    disability_scores: z.array(z.object({
      date: z.string().nullable(),
      instrument: z.string(),                // "NDI", "Oswestry", "DASH", etc.
      score: z.number(),
      max_score: z.number(),
      percent_disability: z.number().nullable(),
      interpretation: z.string().nullable(), // "mild", "moderate", "severe"
    })),
    progress_status: z.enum([
      'improving', 'stable', 'plateauing', 'worsening'
    ]).nullable(),
  }),

  // Plateau statement (Acceptance Criteria #5)
  plateau_statement: z.object({
    present: z.boolean(),
    mmi_reached: z.boolean().nullable(),
    date: z.string().nullable(),             // Date MMI was determined
    verbatim_statement: z.string().nullable(), // Exact wording — critical for legal
    residual_complaints: z.array(z.string()),
    permanent_restrictions: z.array(z.string()),
    impairment_rating_percent: z.number().nullable(),
    future_care_recommended: z.boolean().nullable(),
  }),

  // AI metadata
  confidence: z.enum(['high', 'medium', 'low']),
  extraction_notes: z.string().nullable(),
})
```

### 4. Database Table Design

```sql
create table chiro_extractions (
  id uuid primary key default gen_random_uuid(),
  document_id uuid not null references documents(id),
  case_id uuid not null references cases(id),
  schema_version integer not null default 1,

  -- Report metadata
  report_type text check (report_type in (
    'initial_evaluation', 'soap_note', 're_evaluation',
    'discharge_summary', 'other'
  )),
  report_date date,

  -- Structured extraction (stored as JSONB for flexibility)
  treatment_dates jsonb not null default '{}',
  diagnoses jsonb not null default '[]',
  treatment_modalities jsonb not null default '[]',
  functional_outcomes jsonb not null default '{}',
  plateau_statement jsonb not null default '{}',

  -- AI metadata
  ai_model text,
  ai_confidence text check (ai_confidence in ('high', 'medium', 'low')),
  extraction_notes text,
  raw_ai_response jsonb,

  -- Pipeline state
  extraction_status text not null default 'pending'
    check (extraction_status in ('pending', 'processing', 'completed', 'failed')),
  extraction_error text,
  extraction_attempts integer not null default 0,
  extracted_at timestamptz,

  -- Review workflow
  review_status text not null default 'pending_review'
    check (review_status in ('pending_review', 'approved', 'edited', 'rejected')),
  reviewed_by_user_id uuid references auth.users(id),
  reviewed_at timestamptz,
  provider_overrides jsonb not null default '{}',

  -- Audit
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  deleted_at timestamptz,
  created_by_user_id uuid not null references auth.users(id),
  updated_by_user_id uuid not null references auth.users(id)
);

-- Indexes
create index idx_chiro_extractions_case_id on chiro_extractions(case_id) where deleted_at is null;
create index idx_chiro_extractions_document_id on chiro_extractions(document_id) where deleted_at is null;
create index idx_chiro_extractions_review_status on chiro_extractions(review_status) where deleted_at is null;
create index idx_chiro_extractions_diagnoses on chiro_extractions using gin(diagnoses);
create index idx_chiro_extractions_treatment_modalities on chiro_extractions using gin(treatment_modalities);

-- RLS
alter table chiro_extractions enable row level security;
create policy "Authenticated users can manage chiro extractions"
  on chiro_extractions for all to authenticated using (true) with check (true);

-- Updated_at trigger
create trigger set_chiro_extractions_updated_at
  before update on chiro_extractions
  for each row execute function update_updated_at();
```

**Design decisions:**
- JSONB columns for the five extraction categories (not individual columns) because chiro data is inherently variable — different reports have different subsets of data
- GIN indexes on `diagnoses` and `treatment_modalities` for querying across cases
- Same soft-delete, audit, and review patterns as `mri_extractions`

### 5. Claude System Prompt Design

```
You are a medical data extraction assistant for a personal injury clinic.
You are extracting structured data from a chiropractor / conservative care report.

RULES:
1. Identify the report type first (initial evaluation, SOAP note, re-evaluation, or discharge summary).
2. Extract ALL diagnosis codes exactly as written, including the ICD-10 7th character.
3. Extract ALL treatment modalities mentioned, with CPT codes if present.
4. For functional outcomes, extract exact numeric values (pain scores, disability percentages). Do not estimate or calculate values not explicitly stated.
5. For the plateau/MMI statement, extract the VERBATIM wording. Attorneys review exact language — do not paraphrase.
6. For treatment dates, list every individual visit date found in this report. Flag any gaps exceeding 14 days.
7. Return null for any field that cannot be determined from the document.
8. Set confidence to "low" for poor quality scans, partial documents, or ambiguous content.
9. Add extraction_notes for anything ambiguous or noteworthy.
```

### 6. Review UI Design

The review form is more complex than MRI because of the nested data. Recommended layout:

**Split-pane** (same as MRI review):
- Left: PDF viewer
- Right: Tabbed form with sections:
  - **Overview** tab: Report type, report date, treatment date range, total visits, progress status
  - **Diagnoses** tab: Editable list of diagnosis entries (code + description + region)
  - **Treatment** tab: Editable list of modalities (modality + CPT + regions)
  - **Outcomes** tab: Pain levels, disability scores (each as sub-tables)
  - **Plateau/MMI** tab: Boolean toggle, date, verbatim text (textarea), residual complaints list, permanent restrictions list

Each tab uses `useFieldArray` for the array fields. The same approve/edit/reject workflow applies.

### 8. Key Differences from MRI Implementation

| Aspect | MRI | Chiro |
|--------|-----|-------|
| Tool schema | ~45 lines | ~150 lines (much more complex) |
| DB columns | Fixed columns (body_region, mri_date, etc.) | JSONB columns (more flexible) |
| Review form | Single flat form | Tabbed form with multiple sections |
| Max tokens | 2048 | 4096 (chiro reports are longer, more data to extract) |
| System prompt | 12 lines | 20+ lines (more complex extraction rules) |
| `findings` field | Array of {level, description, severity} | Replaced by 5 distinct JSONB categories |

### 9. Risk Considerations

1. **Long documents**: Chiro compiled charts can be 50+ pages. Claude handles up to 100 pages, but extraction quality may degrade. Consider a `max_tokens: 4096` setting and monitor.
2. **Variable report formats**: Unlike MRI reports (very standardized), chiro reports vary wildly by practice management software. The prompt must be robust to format variation.
3. **Compiled charts deferred**: Multi-visit PDFs (compiled charts) are deferred from MVP. Each uploaded document is treated as a single report producing one extraction. This simplifies both the prompt and the review UI.
4. **Treatment gaps calculation**: Could be done AI-side or app-side. Recommend AI-side for MVP (Claude can identify gaps in visit dates), with potential to move to app-side calculation later for accuracy.

## Architecture Insights
- The MRI extraction pattern is highly reusable — same pipeline shape, just different schema/prompt
- JSONB storage for chiro data (vs fixed columns for MRI) is the right call given the variable nature of chiro reports
- The provider override pattern (`overrides[field] ?? ai_extraction[field]`) works for JSONB fields too
- `schema_version` on the table future-proofs prompt/schema evolution
- Auto-extract on upload (fire-and-forget `.then()`) is the proven pattern

## Decisions Made
1. **No compiled chart support for MVP** — one report = one extraction. Compiled charts deferred to a future story.
2. **No ROM measurements for MVP** — pain levels and disability scores (NDI, Oswestry, etc.) cover functional outcomes. ROM extraction deferred to reduce schema/UI complexity.
3. **Tabs for MRI vs Chiro on clinical page** — two tabs ("MRI Reports" and "Chiro Reports") to separate extraction types cleanly.
4. **Max tokens: 4096** — 2x the MRI setting, sufficient for single-report extraction. Can bump to 8192 later if truncation is observed.
