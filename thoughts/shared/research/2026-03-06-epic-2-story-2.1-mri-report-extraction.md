## Research Question

Find the best design and implementation approach for Story 2.1 (MRI Report Extraction) -- extracting structured clinical data from uploaded MRI reports so providers can review findings.

## Summary

The existing ClinIQ codebase already has a fully functional document upload pipeline (TUS resumable uploads to Supabase Storage, document metadata tracking, PDF preview). Story 2.1 builds on this foundation by adding an AI extraction layer using Claude API and a provider review workflow.

**Recommended approach:**
1. Send uploaded PDFs directly to Claude API (base64) -- no intermediate text extraction needed
2. Use Claude's structured outputs (GA) to guarantee valid JSON extraction
3. Store extractions in a new `mri_extractions` table (fixed columns for queryable fields + JSONB for variable findings)
4. Provider reviews in a split-view UI: PDF on left, editable extracted data on right
5. Provider overrides stored separately to preserve original AI output for audit

## Detailed Findings

### 1. PDF Processing: Send Directly to Claude

**Recommendation: Skip text extraction libraries. Send the PDF as base64 to Claude API.**

Claude natively processes PDFs using vision + text extraction simultaneously. Libraries like `pdf-parse` or `pdfjs-dist` are unnecessary for an AI extraction workflow and add complexity without benefit.

**Approach -- Base64 in API route:**
```typescript
// app/api/extract-mri/route.ts (or server action)
import Anthropic from "@anthropic-ai/sdk";

const anthropic = new Anthropic(); // uses ANTHROPIC_API_KEY env var

const response = await anthropic.messages.create({
  model: "claude-sonnet-4-6",
  max_tokens: 2048,
  messages: [{
    role: "user",
    content: [
      {
        type: "document",
        source: { type: "base64", media_type: "application/pdf", data: pdfBase64 },
      },
      { type: "text", text: EXTRACTION_PROMPT },
    ],
  }],
});
```

**Claude PDF limits:** 32 MB per request, 100 pages, no encrypted/password-protected PDFs. MRI reports are typically 2-10 pages, well within limits.

**Alternative -- Claude Files API (beta):** Upload once, get a `file_id`, reference on re-extraction. Not recommended for MVP because it's still beta and not ZDR-eligible (relevant for future HIPAA production).

### 2. Structured Outputs for Guaranteed JSON

**Recommendation: Use Claude structured outputs (GA on Sonnet 4.6/Opus 4.6) to eliminate JSON parsing failures.**

Structured outputs compile the JSON schema into a grammar that constrains token generation, guaranteeing valid JSON output.

**Extraction schema:**
```typescript
const mriExtractionSchema = {
  type: "object",
  properties: {
    body_region: {
      type: "string",
      description: "e.g. 'Lumbar Spine', 'Cervical Spine', 'Right Knee'"
    },
    mri_date: {
      type: ["string", "null"],
      description: "ISO 8601 date (YYYY-MM-DD) or null if not found"
    },
    findings: {
      type: "array",
      items: {
        type: "object",
        properties: {
          level: { type: "string" },        // e.g. "L4-L5", "C5-C6"
          description: { type: "string" },  // e.g. "Broad-based disc herniation"
          severity: {
            type: ["string", "null"],
            enum: ["mild", "moderate", "severe", null]
          }
        },
        required: ["level", "description", "severity"]
      }
    },
    impression_summary: {
      type: ["string", "null"],
      description: "Radiologist's impression/conclusion section"
    },
    confidence: {
      type: "string",
      enum: ["high", "medium", "low"]
    },
    extraction_notes: {
      type: ["string", "null"],
      description: "Ambiguities, missing data, or quality issues"
    }
  },
  required: ["body_region", "mri_date", "findings", "impression_summary", "confidence", "extraction_notes"]
};
```

**Extraction prompt:**
```
You are a medical data extraction assistant for a personal injury clinic.
Extract structured information from the attached MRI radiology report.

<instructions>
- Extract the body region scanned (e.g., "Lumbar Spine", "Cervical Spine")
- Extract the MRI study date if present
- Extract each disc level or anatomical finding individually -- do NOT combine
- For each finding, identify the spinal level (e.g., L4-L5) or anatomical location
- Include the radiologist's Impression section verbatim if present
- If a field cannot be determined, return null -- do NOT guess
- Set confidence to "low" if document quality is poor or report is incomplete
- Add extraction_notes for anything ambiguous or missing
</instructions>

Extract the structured data from this MRI report now.
```

**Key prompt rules (from Anthropic docs):**
- Place the PDF document BEFORE instructions in the message (improves quality ~30%)
- Use `cache_control: { type: "ephemeral" }` if making multiple calls against the same PDF

### 3. Database Schema

**Recommendation: New `mri_extractions` table with fixed queryable columns + JSONB for findings array.**

```sql
CREATE TABLE mri_extractions (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  document_id       UUID NOT NULL REFERENCES documents(id),
  case_id           UUID NOT NULL REFERENCES cases(id),

  -- Schema version for migration
  schema_version    INTEGER NOT NULL DEFAULT 1,

  -- Fixed columns (queryable)
  body_region       TEXT,
  mri_date          DATE,
  impression_summary TEXT,

  -- Variable findings (JSONB)
  findings          JSONB NOT NULL DEFAULT '[]',

  -- AI metadata
  ai_model          TEXT NOT NULL,
  ai_confidence     TEXT CHECK (ai_confidence IN ('high', 'medium', 'low')),
  extraction_notes  TEXT,
  raw_ai_response   JSONB,

  -- Extraction pipeline
  extraction_status TEXT NOT NULL DEFAULT 'pending'
    CHECK (extraction_status IN ('pending', 'processing', 'completed', 'failed')),
  extraction_error  TEXT,
  extraction_attempts INTEGER DEFAULT 0,
  extracted_at      TIMESTAMPTZ,

  -- Provider review workflow
  review_status     TEXT NOT NULL DEFAULT 'pending_review'
    CHECK (review_status IN ('pending_review', 'approved', 'edited', 'rejected')),
  reviewed_by_user_id UUID REFERENCES users(id),
  reviewed_at       TIMESTAMPTZ,
  provider_overrides JSONB DEFAULT '{}',

  -- Audit (matching project convention)
  created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  created_by_user_id UUID REFERENCES users(id),
  updated_by_user_id UUID REFERENCES users(id)
);

CREATE INDEX idx_mri_extractions_case_id ON mri_extractions(case_id);
CREATE INDEX idx_mri_extractions_document_id ON mri_extractions(document_id);
CREATE INDEX idx_mri_extractions_review_status ON mri_extractions(review_status);
CREATE INDEX idx_mri_extractions_findings ON mri_extractions USING GIN(findings);
```

**Design decisions:**
- `provider_overrides` stores only fields the provider changed -- canonical value is `overrides[field] ?? extraction[field]`, preserving original AI output for audit
- `raw_ai_response` stores the full Claude response for debugging (can be omitted/truncated in production)
- `schema_version` enables handling extraction schema changes without data migration
- `findings` as JSONB rather than a separate child table -- MRI findings are naturally variable and Postgres has excellent JSONB indexing

### 4. Extraction Pipeline Flow

**Recommended pipeline -- triggered after document upload:**

```
1. User uploads MRI PDF (existing TUS flow)
   |
2. After saveDocumentMetadata(), call extractMriReport() server action
   |
3. Server action:
   a. Download file from Supabase Storage (server-side signed URL)
   b. Convert to base64
   c. Send to Claude API with structured output schema
   d. Insert result into mri_extractions table
   e. Update document status to 'reviewed' or mark extraction as 'failed'
   |
4. Client receives extraction result (or polls/subscribes for async)
   |
5. Provider reviews in split-view UI
```

**Sync vs Async:**
- MRI reports are 2-10 pages; Claude extraction should take 5-15 seconds
- **Recommendation for MVP: Synchronous.** Call the extraction in the server action and await the result. Show a loading spinner in the UI. This avoids the complexity of polling/realtime subscriptions.
- If extraction time becomes a problem, upgrade to async with Supabase Realtime subscription later.

**Retry pattern:**
```typescript
async function extractWithRetry(pdfBase64: string, maxAttempts = 2) {
  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return await callClaudeForExtraction(pdfBase64);
    } catch (error) {
      if (!isRetryableError(error) || attempt === maxAttempts) throw error;
      await new Promise(r => setTimeout(r, 2000 * attempt));
    }
  }
}
```

### 5. Integration with Existing Upload Flow

**Option A -- Auto-extract on upload (recommended for MVP):**
When a user uploads a document with `document_type = 'mri_report'`, automatically trigger extraction after the upload completes. The upload sheet already knows the document type per file.

Modify the upload flow in `upload-sheet.tsx`:
1. After `saveDocumentMetadata()` succeeds
2. If `documentType === 'mri_report'`, call `extractMriReport(documentId)`
3. Show extraction progress in the same upload sheet or redirect to the clinical data view

**Option B -- Manual trigger from document card:**
Add an "Extract Findings" button to `DocumentCard` for MRI-type documents. Provider clicks to trigger extraction on demand. Better for control but adds friction.

**Recommendation: Option A** for automatic extraction with a manual "Re-extract" button as fallback.

**Review workflow:**

- **Save & Approve** -- writes edits to `provider_overrides`, sets `review_status = 'edited'`

**Confidence indicator:**
- High = green badge
- Medium = yellow badge with note "Review recommended"
- Low = red badge with warning "Extraction confidence is low. Please verify all fields."

### 7. Error Handling

| Failure | Cause | Handling |
|---------|-------|----------|
| Upload failure | Network/size | Existing TUS retry handles this |
| Encrypted PDF | Password-protected | Check for `/Encrypt` in first 8KB before sending to Claude; show user-friendly error |
| Claude API timeout | Rate limit/overload | Retry 2x with exponential backoff |
| Low-quality extraction | Poor scan/handwriting | Claude self-reports `confidence: low`; surface warning in UI |
| Extraction failure | Corrupted PDF / API error | Mark `extraction_status = 'failed'`, show "Enter manually" option |

**Fallback to manual entry:** When extraction fails, render the same review form with empty fields. On save, create `mri_extractions` record with `ai_model = null`, `review_status = 'edited'`.

## Architecture Insights

1. **Builds on existing patterns:** The extraction feature follows the same patterns as the rest of the app -- server actions with Zod validation, shadcn/ui components, Supabase queries. No new architectural patterns needed.

2. **Claude API as a server-only dependency:** The Anthropic SDK runs only in server actions/API routes. PDFs are downloaded server-side from Supabase Storage, processed, and results stored in the database. No PHI leaves the server except to Claude's API.

3. **Separation of extraction and review:** The `mri_extractions` table is separate from `documents`. A document can have zero or one extraction. This allows re-extraction (delete old, create new) and preserves the document lifecycle independently.

4. **Provider override pattern:** Storing edits in `provider_overrides` JSONB rather than overwriting the AI columns preserves the original extraction for audit. The canonical display value is `overrides[field] ?? ai_extraction[field]`.

5. **Schema versioning:** The `schema_version` integer on `mri_extractions` enables evolving the extraction prompt/schema without breaking existing data.

## Resolved Decisions

1. **Model choice:** Use **Sonnet 4.6**. MRI reports contain dense medical terminology where extraction accuracy matters (missed findings affect legal cases). Cost difference is negligible at clinic volume. Haiku is better suited for simple classification, not structured medical extraction.

2. **Auto-extract scope:** Auto-extract on upload for `mri_report` only in Story 2.1. Build the trigger generically (check document type -> call appropriate extractor) so adding `chiro_report` extraction in a later story is just a new schema + prompt. No manual trigger needed -- auto-extract by default with a "Re-extract" button as fallback.

3. **Sync vs async:** **Synchronous for MVP.** MRI reports are 2-10 pages; Claude processes in 5-15 seconds. Acceptable for a loading spinner. User is already in a "waiting" mindset from the upload itself. Revisit only if extraction consistently exceeds 20 seconds.

4. **HIPAA data flow:** **Synthetic data only for now.** Matches existing project convention (no real PHI in dev). Claude API call sends PDF content to Anthropic's servers -- production requires Anthropic's HIPAA Enterprise plan with BAA. Document this in a production readiness checklist.

5. **Re-extraction:** **Overwrite (soft-delete old, create new).** Soft-delete the existing `mri_extractions` record (`deleted_at`), create a fresh one. Preserves audit history in the DB without complicating the UI with version management. One active extraction per document at any time.
