# Research: Application Architecture Review & Improvement Recommendations

**Date**: 2026-04-16T20:09:52Z
**Researcher**: arsenaid
**Git Commit**: 7b8ed068939e2f57163780790a1da369df387d38
**Branch**: main
**Repository**: cliniq

## Research Question

Document the current application architecture and recommend improvements.

## Summary

ClinIQ is a Next.js 16 App Router + Supabase + Anthropic SDK app for a personal-injury chiropractic clinic. The architecture is **consistent and minimalist** — a hallmark of a small team that has stayed disciplined: one client type per surface (browser/server/middleware), one validation library (Zod), one forms stack (react-hook-form + shadcn), one PDF engine (@react-pdf/renderer), all server logic in Server Actions (no API routes), all tables RLS-gated, all Claude calls via tool-use. Data flows top-down through RSC pages → server actions → Supabase, with thin client leaves for interactivity. Tests exist only for validations, constants, and a handful of actions (26 files total, 0 component tests, no CI).

Given that baseline, the most impactful improvements cluster into seven themes:

1. **RLS that actually protects data** — today every authenticated user can read/write every row on every table, which is a PHI leak waiting to happen.
2. **A shared auth/validation/error wrapper for server actions** — the same 10-line preamble is duplicated 70+ times.
3. **Observability** — no Sentry, no logging library, no CI; errors vanish silently.
4. **Anthropic prompt caching + retry handling** — the same multi-KB system prompts are sent on every call, extractions silently retry once then persist `failed`.
5. **Background jobs for long-running work** — extractions, PDF renders, and Claude generation run inside Server Actions blocking the request path.
6. **RSC fetch discipline** — several pages fetch 5+ things in parallel via server actions that each open a new Supabase client; small repository layer or DataLoader-style batching would help.
7. **CI + component tests + Playwright smoke** — the app has no automated gate before deploy.

The recommendations below are ordered by **risk-weighted value** (impact × likelihood of harm if ignored), not by implementation ease. Section headers identify what to do; bodies cite the specific files and give a concrete landing shape.

## Current Architecture (as-is)

### Stack
- **Framework**: Next.js 16.1.6 (App Router), React 19.2.3
- **DB/Auth/Storage**: Supabase (`@supabase/ssr` 0.9, `@supabase/supabase-js` 2.98)
- **AI**: `@anthropic-ai/sdk` 0.78 (tool use, `claude-sonnet-4-6` for extractions, `claude-opus-4-6` with extended thinking for summaries)
- **PDF**: `@react-pdf/renderer` 4.3, `sharp` for image conversion
- **Forms**: `react-hook-form` 7.71 + `@hookform/resolvers` + `zod` 4
- **UI**: shadcn wrapping Radix primitives, Tailwind v4, `next-themes`, `sonner`
- **Tables**: `@tanstack/react-table` 8.21
- **Uploads**: `tus-js-client` 4.3 (resumable uploads direct to Supabase Storage)
- **Testing**: `vitest` 4.1 (node env), no jsdom, no coverage config, no CI

### Routing
- Two route groups: [`(auth)`](src/app/(auth)/) (login only) and [`(dashboard)`](src/app/(dashboard)/) (everything else).
- Nested layout at [`(dashboard)/patients/[caseId]/layout.tsx`](src/app/(dashboard)/patients/%5BcaseId%5D/layout.tsx) fetches the case and hangs a `CaseStatusContext` around per-case tabs (overview, initial-visit, clinical, timeline, discharge, documents, procedures, billing).
- **No `api/` route handlers anywhere** — all server logic is Server Actions.
- **No `loading.tsx`, `error.tsx`, or `not-found.tsx`** files exist; no `export const dynamic|revalidate|runtime` segment config is set.
- Single middleware at [src/middleware.ts](src/middleware.ts) that delegates to `updateSession` in [src/lib/supabase/middleware.ts:4-51](src/lib/supabase/middleware.ts#L4-L51).

### Data layer
- Three Supabase client factories: [client.ts](src/lib/supabase/client.ts) (browser), [server.ts](src/lib/supabase/server.ts) (RSC/Server Actions, anon key + cookies), [middleware.ts](src/lib/supabase/middleware.ts) (request/response cookie propagation).
- **No service-role client exists**. Grep for `SUPABASE_SERVICE_ROLE`, `service_role`, etc. returns zero hits. All server-side queries run under the caller's JWT.
- 28 tables in [src/types/database.ts](src/types/database.ts) (~2,730 lines, generated by `supabase gen types`).
- ~35 migrations in [supabase/migrations/](supabase/migrations/); RLS is enabled on every table but every policy is the single expression `for all using (auth.role() = 'authenticated')` (see Detailed Findings §1).
- No repository/DAO layer — each server action opens `createClient()` and builds its own query chain inline.

### Server logic
- 31 server action modules in [src/actions/](src/actions/) covering attorneys, patients, cases, documents, 6 extraction types, 4 note types, invoices, payments, service catalog, settings, auth.
- Every write action repeats the preamble `getUser` → null check → `assertCaseNotClosed` → Zod `safeParse` → Supabase mutation → `revalidatePath`. See [patients.ts:29-101](src/actions/patients.ts#L29-L101), [mri-extractions.ts:11-116](src/actions/mri-extractions.ts#L11-L116), [billing.ts:295-351](src/actions/billing.ts#L295-L351).
- Return shape is consistent: `{ data, error }` discriminated object; errors are never thrown to the RSC boundary.
- `assertCaseNotClosed` at [case-status.ts:9-24](src/actions/case-status.ts#L9-L24) is the sole business-logic authz primitive and is imported into every case-scoped write.

### AI
- 11 modules in [src/lib/claude/](src/lib/claude/): 6 extraction, 5 generation. Each uses `tool_choice: { type: 'tool', name: ... }` to force structured JSON.
- Claude response is re-validated with Zod before persisting (e.g., [extract-mri.ts:130-133](src/lib/claude/extract-mri.ts#L130-L133), [generate-summary.ts:335-339](src/lib/claude/generate-summary.ts#L335-L339)).
- Extended thinking (`budget_tokens: 10000`) only on case summary; no prompt caching anywhere.
- `new Anthropic()` is instantiated at module top-level in each file.

### PDF
- 8 document types (chiropractic order, imaging orders, lien, procedure consent, procedure note, initial visit, discharge note, invoice), each with a `-template.tsx` component and a `render-*.ts` renderer in [src/lib/pdf/](src/lib/pdf/).
- Renderers fetch their own data from Supabase and convert logos/signatures to PNG with `sharp`.
- Invoke path: server action dynamically imports renderer, returns base64 buffer, client triggers download — see [billing.ts:459-468](src/actions/billing.ts#L459-L468).

### Frontend
- `components/ui/` — 25 shadcn wrappers. Everything else is domain-grouped flat folders: `attorneys/`, `billing/`, `cases/`, `clinical/`, `discharge/`, `documents/`, `layout/`, `patients/`, `procedures/`, `providers/`, `settings/`, `timeline/`.
- 63 files with `"use client"`. Data fetching is RSC-only (no TanStack Query / SWR). State is local `useState` + a single `CaseStatusContext` at [case-status-context.tsx:1-23](src/components/patients/case-status-context.tsx#L1-L23).
- Forms: `useForm({ resolver: zodResolver(schema), mode: 'onBlur' })`, errors via `<FormMessage />`, submit via bespoke `isSubmitting` useState + `toast.error` — **no `useActionState` / `useFormStatus` usage, no optimistic updates**.
- Uploads: Server action `getUploadSession` returns a signed path → client uses `tus-js-client` to upload directly to Supabase Storage → client calls `saveDocumentMetadata` → then fires extractions via `setTimeout(..., 0)` (see [upload-sheet.tsx:114, 198](src/components/documents/upload-sheet.tsx#L114)).

### Testing
- `vitest.config.ts`: node env, globals on, `@` alias, no setup file, no coverage config.
- 26 tests total: 16 for validations, 6 for actions, 2 for constants, 1 for filenames, 1 for procedures. **Zero component tests, zero e2e, zero Anthropic SDK mocking, no CI workflow, no Playwright**.
- Mock Supabase helper exists at [src/test-utils/supabase-mock.ts](src/test-utils/supabase-mock.ts) and is well-designed (chainable builder + `then` handler).

## Recommendations (Prioritized)

### 1. ⚠️ Fix RLS so authenticated ≠ "read all PHI"

**Problem.** Every RLS policy produced by the migrations is functionally `for all using (auth.role() = 'authenticated')`. That means any logged-in user of the clinic (today, anyone with `auth.users` access) can `SELECT`/`UPDATE`/`DELETE` any row on any table, including other patients' PHI. Since there is also no service-role client anywhere, the application layer is not silently elevating — the DB itself is wide open to every signed-in session.

For a personal-injury clinic with medical records, lien agreements, and attorney PII, this is the single highest-severity issue in the repo. It is invisible today because there's only one clinic and a few providers, but:
- Multi-tenant is a near-term direction (the schema already has `clinic_settings` and `provider_profiles`).
- HIPAA audit exposure: even in single-tenant, RLS is the evidence that only authorized users can see patient rows.
- A compromised low-privilege login today gets everything.

**Landing shape.**

1. Add a `clinic_id` (or `organization_id`) column to every root table (`patients`, `cases`, `attorneys`, `documents`, `invoices`, `provider_profiles`, `clinic_settings`, …). For derived tables (extractions, notes, line items), inherit via `case_id` → `cases.clinic_id`.
2. Add a `user_clinic_memberships` table linking `auth.users.id` → `clinic_id` → `role` (`owner` | `provider` | `staff` | `billing` | `read_only`).
3. Replace every `for all using (auth.role() = 'authenticated')` with role+clinic-scoped policies:
   ```sql
   create policy "patients_select_clinic_member" on patients
     for select using (
       clinic_id in (
         select clinic_id from user_clinic_memberships
         where user_id = auth.uid()
       )
     );
   ```
4. Split read from write policies and scope writes by role.
5. Add a `security definer` helper `is_clinic_member(clinic_id uuid)` to keep policies readable.
6. Write pgTAP tests in a new migration that assert: user A cannot select patient of clinic B, staff role cannot delete invoices, etc. These are the cheapest PHI regression tests you can have.

**Why this comes first.** Every other recommendation below is about code hygiene or performance. This one is about not leaking records.

### 2. Introduce a server-action framework (`createAction` wrapper)

**Problem.** Every write action begins with the same 10–15 lines:

```ts
const supabase = await createClient();
const { data: { user } } = await supabase.auth.getUser();
if (!user) return { error: 'Not authenticated' };

const parsed = schema.safeParse(input);
if (!parsed.success) return { error: parsed.error.flatten().fieldErrors };

const closed = await assertCaseNotClosed(caseId);
if (closed.error) return { error: closed.error };

// … the actual 3 lines of work …

revalidatePath(`/patients/${caseId}`);
return { data };
```

Across 31 action files this boilerplate appears 70+ times (see [patients.ts:29-45](src/actions/patients.ts#L29-L45), [mri-extractions.ts:11-40](src/actions/mri-extractions.ts#L11-L40), [billing.ts:295-310](src/actions/billing.ts#L295-L310), …). Mistakes are inevitable: some read actions skip the auth check entirely and rely on RLS (which, per §1, doesn't actually filter today).

**Landing shape.** Introduce a single helper at `src/lib/actions/create-action.ts`:

```ts
export function createAction<TInput, TOutput>(opts: {
  input: z.ZodSchema<TInput>;
  requireAuth?: boolean;           // default true
  requireOpenCase?: (i: TInput) => string | null;  // returns caseId or null
  revalidate?: (i: TInput) => string[];
  handler: (ctx: {
    input: TInput;
    supabase: SupabaseClient<Database>;
    user: User;
    logger: Logger;                // see §3
  }) => Promise<TOutput>;
}): (raw: unknown) => Promise<{ data?: TOutput; error?: string | FieldErrors }>;
```

Every action becomes the rule it used to repeat:

```ts
export const createPatientCase = createAction({
  input: createPatientCaseSchema,
  revalidate: () => ['/patients'],
  handler: async ({ input, supabase, user }) => {
    const { data: patient } = await supabase.from('patients').insert(...).select().single();
    // …
    return patient;
  },
});
```

**Secondary wins.**
- Centralizes the one place that decides how errors are shaped for clients — easier to evolve.
- Lets you attach telemetry (§3) in one spot.
- Lets you switch from `{ data, error }` to React 19's `useActionState` return shape later without rewriting 31 files.
- Enables per-action rate limiting (§4).

### 3. Observability: logger + Sentry + a logs table

**Problem.** Grep confirms no Sentry, no PostHog, no pino/winston, no logging library. The only instrumentation in the repo is a handful of `console.log` / `console.error` calls in PDF renderers and the Claude summary validator. Action errors return a string to the UI and then evaporate — you have no record of how often extractions fail, which prompts break Zod validation, or which user hit the invoice PDF crash last Tuesday.

Symptoms in the current code:
- [mri-extractions.ts:86-99](src/actions/mri-extractions.ts#L86-L99) silently retries once and persists `extraction_status: 'failed'` with no trace of what Claude returned.
- [generate-summary.ts:337](src/lib/claude/generate-summary.ts#L337) logs the Zod error only via `console.error`.
- [invoice-status.ts:59](src/actions/invoice-status.ts#L59) has one lonely `console.error`.
- The `audit_logs` table exists in migration 001 but no code writes to it today.

**Landing shape.**

1. Install `@sentry/nextjs`, wire it into `instrumentation.ts`, `app/global-error.tsx`, and the middleware.
2. Add a lightweight `Logger` type (structured logging: `logger.info({ action, caseId, userId, duration }, 'message')`) — pino with `transport: 'pino-pretty'` in dev, silent in prod, Sentry breadcrumb integration.
3. Start writing to `audit_logs` from inside `createAction` (§2): `action_name`, `user_id`, `entity`, `entity_id`, `outcome`, `duration_ms`. This gives you clinical/HIPAA-useful audit trails for free.
4. Add an error boundary: [src/app/global-error.tsx](src/app/global-error.tsx) + at least one `error.tsx` inside `(dashboard)` so Next.js doesn't fall back to a blank page.

### 4. Claude: prompt caching, module-level client, and resilient extraction

**Problem.**
- Every Claude module instantiates `new Anthropic()` at module top-level (fine) but sends the full system prompt on every request with no `cache_control`. Prompts for extraction tools are multi-KB schemas — caching would typically cut 80–95% of the input tokens on the warm path.
- [extract-mri.ts](src/lib/claude/extract-mri.ts) and siblings retry exactly once inside the action ([mri-extractions.ts:86-99](src/actions/mri-extractions.ts#L86-L99)) then give up. No backoff, no jitter, no distinction between retryable (529, 429, 5xx) and non-retryable (400, schema violation) errors.
- Extractions happen inline inside a Server Action triggered by `setTimeout(..., 0)` on the client after upload ([upload-sheet.tsx:198](src/components/documents/upload-sheet.tsx#L198)). If the user closes the tab, the action still completes — but the user has no way to see status, cancel, or resume.

**Landing shape.**

1. Add `cache_control: { type: 'ephemeral' }` to the `system` block and the `tools` block in every Claude module. For long-running tools (summary, extractions), this is the single biggest latency + cost win available.
2. Extract a `src/lib/claude/client.ts` that owns the `Anthropic` singleton + a `callClaudeWithRetry(params, { maxAttempts, backoffMs })` helper using full-jitter exponential backoff and proper retryable-status detection. Today `extract-mri.ts`, `extract-chiro.ts`, etc. each open-code their own SDK call.
3. Validate Claude tool responses at one layer (the retry helper), not scattered across the file. Today [generate-summary.ts:335-339](src/lib/claude/generate-summary.ts#L335-L339) and each `extract-*.ts` each repeat the safeParse dance.
4. Consider `claude-haiku-4-5` for the cheaper/short extractions (chiro summary pages, CT-scan single-page) once you measure — the repo uses `sonnet-4-6` uniformly.
5. Move extractions into a background job (§5) so the client sees real progress.

### 5. Background job runner for extractions, generation, and PDF rendering

**Problem.** Three expensive operations live inside Server Actions today:
- **Claude extractions** — 10–60s per document, retry inline, no user-visible progress.
- **Claude generation** (`generate-summary.ts` uses extended thinking with `budget_tokens: 10000` — can run 30–120s).
- **PDF rendering** — server-side `@react-pdf/renderer` + `sharp` image conversions (see [render-invoice-pdf.ts:76-87](src/lib/pdf/render-invoice-pdf.ts#L76-L87)).

Server Actions run in the request/response cycle. If Vercel times out (default 10s on hobby, 60–300s on pro) you lose the work. The current mitigation — client-side `setTimeout(..., 0)` at [upload-sheet.tsx:198](src/components/documents/upload-sheet.tsx#L198) — is a workaround, not a design.

**Landing shape.** Use Supabase's built-in queue or a managed worker (Inngest, Trigger.dev, QStash). Concretely:

1. Create a `jobs` table with `id, kind, payload jsonb, status, attempts, last_error, result jsonb, created_at, started_at, finished_at`.
2. Create a job runner (a Next.js route handler triggered by a cron, or a Supabase Edge Function) that pulls `status = 'pending'` and dispatches by `kind`.
3. Server actions enqueue instead of inline-running: `enqueueJob({ kind: 'extract_mri', documentId })` returns immediately, UI subscribes to job status via Supabase Realtime or polling.
4. User-facing benefits: real progress, cancel/retry, parallelism control, visibility into failures.
5. This is also where SMS/email notifications and bulk PDF generation naturally land when those ship.

### 6. Next.js error/loading boundaries + route segment config

**Problem.** The app router has no `error.tsx`, `loading.tsx`, `not-found.tsx`, or `export const dynamic|revalidate|runtime` anywhere. That means:
- Any thrown error in an RSC page bubbles to the framework default (a blank dev overlay, a plain production error page).
- No streaming: the dashboard at [(dashboard)/patients/[caseId]/page.tsx](src/app/(dashboard)/patients/%5BcaseId%5D/page.tsx) does `Promise.all` of 5 queries and the user sees nothing until the slowest finishes.
- No per-route opt-in for `dynamic = 'force-dynamic'` — cache behavior is implicit.

**Landing shape.**

1. Add `src/app/global-error.tsx` (catches root) and `src/app/(dashboard)/error.tsx` (catches dashboard) — both reset-capable, both reporting to Sentry.
2. Add `loading.tsx` at `(dashboard)/patients/[caseId]/` with skeleton cards for the overview tabs. Use `<Suspense>` with section-scoped fallbacks inside the page so the fastest queries paint first.
3. Explicitly declare `export const dynamic = 'force-dynamic'` on routes that must never be cached (anything showing PHI) and `export const revalidate = 60` on low-change pages (settings list). Implicit behavior is a footgun as Next.js versions change.
4. Add `not-found.tsx` so case-not-found renders cleanly.

### 7. Thin repository layer + batch/co-locate queries

**Problem.** Every Server Action and several RSCs call `createClient()` and then build the query inline. The case overview page ([(dashboard)/patients/[caseId]/page.tsx](src/app/(dashboard)/patients/%5BcaseId%5D/page.tsx)) runs 5 parallel server actions, each of which opens its own Supabase client and its own `.from(...).select(...)` — no batching, no shared context, and (because of the anon key pattern) the request headers are re-read each time.

Small footprint problem today; becomes large as table count grows past 28 and new features keep adding tabs.

**Landing shape.**

1. Introduce `src/lib/db/` with one file per aggregate root (`patients.ts`, `cases.ts`, `documents.ts`, `invoices.ts`, `extractions.ts`). Each exports typed read functions: `getCaseOverview(caseId)`, `listDocumentsByCase(caseId, filters)`, etc. These consume a `SupabaseClient` injected by the caller (so the same functions work from Server Actions, RSCs, and tests).
2. Hoist shared `SELECT` fragments into constants per aggregate so the projection is reused (e.g., `CASE_OVERVIEW_COLUMNS`). Prevents the "everyone selects `*`" drift that's already starting in [patients.ts:14-19](src/actions/patients.ts#L14-L19).
3. Use Postgres views for common join shapes (e.g., `case_overview_v`) — cleaner RLS and cleaner types.
4. Page-level RSCs become: `const { case, documents, procedures } = await getCaseOverview(caseId, supabase);`.

**Explicitly not needed.** A full ORM (Prisma, Drizzle). The Supabase client + generated types is already good enough — this is just a thin DAO layer, not a framework swap.

### 8. Forms: adopt `useActionState` + `useFormStatus`, add optimistic updates

**Problem.** React 19 is already installed; Server Actions already return `{ data, error }`. Every form in the repo, however, keeps a manual `const [isSubmitting, setIsSubmitting] = useState(false)` and handles errors by awaiting the action in a bespoke submit handler. Field-level server errors are not re-mapped onto react-hook-form's error state — they're shown via `toast.error`.

**Landing shape.**
1. Replace manual submit handlers with `useActionState(action, initialState)`. The hook handles pending automatically and is the future-proof pattern.
2. Use `form.setError(field, { message })` to surface per-field server errors returned in `error: fieldErrors` — today users just see a toast.
3. Add `useOptimistic` for list mutations (e.g., updating case status, toggling lien on a document) so the UI reflects state before the revalidation round-trip. Worst case on failure: revert + toast.

### 9. Client-side data fetching: adopt TanStack Query for targeted cases

**Problem.** The app is 100% RSC for reads. For most pages this is great. It starts to hurt at:
- **Document extraction status** — currently users refresh the page to see if Claude has finished ([upload-sheet.tsx:198](src/components/documents/upload-sheet.tsx#L198) fires-and-forgets).
- **Filters on long lists** — every filter change today is a full server round-trip (see patient list, document list).
- **Dialog form lookups** — comboboxes like [cpt-code-combobox.tsx](src/components/billing/cpt-code-combobox.tsx) and [diagnosis-combobox.tsx](src/components/procedures/diagnosis-combobox.tsx) would benefit from cached client-side query + dedupe.

**Landing shape.** Don't adopt TanStack Query everywhere — only the five-or-so screens where client-side caching + realtime is the right shape. Pair it with Supabase Realtime channels for extraction-status updates. The rest of the app stays RSC.

### 10. Testing + CI gate

**Problem.** 26 tests, all unit-scoped. Zero component tests, zero e2e, no CI workflow file anywhere. The download-filename utility has a test; the invoice PDF renderer that produces the output the attorney actually sees does not.

**Landing shape.**

1. Add `.github/workflows/ci.yml` that runs `npm ci && npm run lint && npm run build && npm test` on PR.
2. Flip `vitest.config.ts` env to `jsdom` (or keep node for actions and add a second project config for components), add `@testing-library/react`, and write smoke tests for 5 high-risk components: `PatientWizard`, `UploadSheet`, `CreateInvoiceDialog`, `CaseOverview`, `InitialVisitEditor`.
3. Add Playwright with 3 smoke paths: login → create patient → upload → see extraction; login → create invoice → download PDF; login → close case → verify locks.
4. Add `@vitest/coverage-v8` config and a 60% threshold on `src/lib/` and `src/actions/`.
5. Mock the Anthropic SDK — there is currently no pattern for this. Add `src/test-utils/anthropic-mock.ts` alongside [supabase-mock.ts](src/test-utils/supabase-mock.ts).

### 11. Environment & secrets hygiene

**Problem.** No central `src/lib/env.ts` — env vars are pulled directly where used (`process.env.ANTHROPIC_API_KEY` in each Claude module, `NEXT_PUBLIC_SUPABASE_URL` in each client factory). No runtime validation. A missing key fails deep inside an action, not at boot.

**Landing shape.** Add a single `src/lib/env.ts` using Zod:

```ts
export const env = z.object({
  NEXT_PUBLIC_SUPABASE_URL: z.string().url(),
  NEXT_PUBLIC_SUPABASE_ANON_KEY: z.string().min(1),
  SUPABASE_SERVICE_ROLE_KEY: z.string().min(1).optional(),   // when §1 needs it
  ANTHROPIC_API_KEY: z.string().min(1),
  SENTRY_DSN: z.string().url().optional(),
}).parse(process.env);
```

Imported from `instrumentation.ts` so boot fails loud if something's missing.

### 12. Minor cleanups worth batching

- **Dead file**: [src/hooks/use-mobile.ts](src/hooks/use-mobile.ts) is the only file in `hooks/`. Either build out the folder or inline the hook into the one component that uses it.
- **No `"server-only"` imports**: Server-only modules (Claude, PDF renderers, service actions) should import `'server-only'` at the top to fail the build if they ever get pulled into a client bundle. Today nothing prevents a refactor from leaking `ANTHROPIC_API_KEY` to the browser.
- **`revalidatePath` strings are duplicated** across actions with typos waiting to happen — centralize in `src/lib/routes.ts`: `routes.caseOverview(caseId)`, `routes.caseBilling(caseId)`, …
- **`<Form>` wrapper types**: every form re-declares the `z.infer<typeof schema>` generic. A `useZodForm(schema)` helper at `src/lib/forms/use-zod-form.ts` would remove this.
- **Case closed check lives in SQL too**: `assertCaseNotClosed` is app-level only ([case-status.ts:9-24](src/actions/case-status.ts#L9-L24)). Add a CHECK/trigger at the DB layer so a malicious client can't bypass by calling Supabase directly (especially relevant once multi-tenant RLS lands, see §1).

## Architecture Documentation

The current architecture can be summarized as **"thin, disciplined, and not yet hardened for multi-user HIPAA-grade operation"**:

- Uniform request pipeline: middleware refreshes session → RSC page calls server action → action opens anon-key Supabase client → returns `{ data, error }` → RSC renders → client leaves handle interactivity.
- Single validation surface (Zod), single forms stack (react-hook-form + shadcn), single PDF engine (@react-pdf/renderer), single AI surface (Anthropic tool-use).
- All expensive operations (Claude, PDF, extraction pipelines) currently run inline in the request/response cycle.
- RLS is the nominal authorization surface but is not scoped — all the actual authorization is the application-layer `getUser()` + `assertCaseNotClosed` primitive.
- Observability surface is empty.
