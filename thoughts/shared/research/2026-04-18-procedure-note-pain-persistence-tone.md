# Research: Procedure reporting tone on repeat procedures — where pain persistence vs. improvement/intermittent phrasing is decided

**Date**: 2026-04-18T12:50:57-0700
**Researcher**: arsenaid
**Git Commit**: 6813cb8fb5d7a75456a3ddf229f2774dc616590b
**Branch**: main
**Repository**: arsenaid-yere1/cliniq

## Research Question
Check procedure reporting tone when a procedure is done multiple times. Pain persistence is commonly mentioned in generated notes instead of reporting improvements as "intermittent" or "improved."

## Summary

The PRP Procedure Note is generated by a single Claude tool call whose system prompt (in [generate-procedure-note.ts:102-203](src/lib/claude/generate-procedure-note.ts#L102-L203)) is the only place where the tone of the `subjective` section is controlled. The prompt instructs the model, for every procedure number (first, second, third…), to open with a fixed "returns for scheduled PRP injection" sentence and then write a clinical narrative consisting of **"persistent symptoms, functional limitations, current pain"**. That phrase is hard-coded — there is no conditional branch that softens the language when the patient's pain has improved between visits.

The only data passed to the model that reflects interval change is the prior procedure's `pain_score_max`, and the prompt restricts the comparison to a single templated phrase: `"compared to [prior_pain_score_max]/10 at [his/her] last visit"` ([generate-procedure-note.ts:123](src/lib/claude/generate-procedure-note.ts#L123)). The model is never told "if current pain < prior pain, describe the patient as improved/intermittent." The one reference example in the same prompt uses the wording "mild improvement" — so this phrasing is shown as an example, not as a conditional rule.

`procedure_number` is passed through in the input data and is only referenced by the prompt for section 16 (`procedure_followup`) to talk about the number of remaining injections in the series. Nothing in the prompt keys `subjective` tone off `procedure_number`. Other sections that describe current state (`review_of_systems`, `objective_physical_exam`, `assessment_summary`) also contain no conditional improvement/persistence logic.

By contrast:
- The **Discharge Note** prompt ([generate-discharge-note.ts:104-181](src/lib/claude/generate-discharge-note.ts#L104-L181)) explicitly instructs a tone of **completion, improvement, and sustained recovery** across every section (subjective, objective exam, assessment, prognosis).
- The **Initial Visit / Pain Evaluation Visit** prompt ([generate-initial-visit.ts:78-216](src/lib/claude/generate-initial-visit.ts)) hard-codes "Persistent" in the Chief Complaint reference ("• Neck pain: Persistent, rated 7-8/10.") and asks for "persistent symptoms despite conservative care" in Medical Necessity.
- Only the chiropractic extractor has a structured improvement axis — an enum `['improving', 'stable', 'plateauing', 'worsening', 'null']` at [extract-chiro.ts:118](src/lib/claude/extract-chiro.ts#L118) — but this is not wired into procedure-note generation.

In short: the procedure-note prompt gives Claude one narrative stance ("persistent symptoms") plus a single numeric delta (`prior pain_score_max` vs. current pain). It does not express the range of clinically valid descriptions (improved, intermittent, stable, worsened) as conditional outputs.

## Detailed Findings

### 1. PRP Procedure Note — generator and prompt

**File**: [src/lib/claude/generate-procedure-note.ts](src/lib/claude/generate-procedure-note.ts)

The `SYSTEM_PROMPT` constant ([lines 102-203](src/lib/claude/generate-procedure-note.ts#L102-L203)) is the single prompt used for every PRP procedure note regardless of whether it is the 1st, 2nd, or Nth injection in the series. The tool definition ([lines 205-255](src/lib/claude/generate-procedure-note.ts#L205-L255)) lists 20 structured sections; none are conditionally enabled based on procedure count.

#### The `subjective` section — where tone is set
[generate-procedure-note.ts:122-124](src/lib/claude/generate-procedure-note.ts#L122-L124):

> `1. subjective (~1 paragraph):`
> `Open with a one-sentence patient identification: "[Patient Name] is a [age]-year-old [gender] who returns for [his/her] scheduled PRP injection to the [site]." ... Then continue with the clinical narrative: persistent symptoms, functional limitations, current pain. Pain is captured as a MIN/MAX range on vitalSigns.pain_score_min / pain_score_max. Render as a range when both are present and differ ... If priorProcedure exists AND priorProcedure.pain_score_max is not null, compare to the prior visit using the prior pain_score_max as the reference ("compared to [prior_pain_score_max]/10 at [his/her] last visit"). If first injection or no prior pain recorded: no comparison.`
> `Reference: "Mr. Vardanyan is a 45-year-old male who returns today for his scheduled follow-up visit, 14 days after receiving his first PRP injection to the lumbosacral region. He reports mild improvement in his low back pain and function following the initial injection... Pain is now rated 3-4/10, compared to 6/10 at his last visit."`

Observations visible in the prompt:

1. The directive **"persistent symptoms, functional limitations, current pain"** is an unconditional list of narrative beats. There is no "if improved, write X; if worse, write Y" branch.
2. The comparison to the prior visit uses one templated form: `compared to [prior_pain_score_max]/10 at [his/her] last visit`. It does not instruct the model to explicitly state improvement, plateau, or worsening.
3. The reference example contains the phrase "mild improvement," but this is shown only as a reference for narrative style — the instruction line itself frames the narrative as "persistent symptoms."
4. The opening sentence is `"[Patient Name] ... returns for [his/her] scheduled PRP injection to the [site]."` It does not vary with procedure number. The reference example, by contrast, uses "returns today for his scheduled follow-up visit, 14 days after receiving his first PRP injection" — richer wording that references the prior procedure's timing. The instruction text does not require this richer framing.

#### Other sections that describe current state
[generate-procedure-note.ts:142-148](src/lib/claude/generate-procedure-note.ts#L142-L148) — `review_of_systems` and `objective_vitals`: reference examples use language like "Ongoing low back pain with bilateral sciatica exacerbation" and "Continued headaches on and off." The instruction text contains no improvement/persistence branch.

[generate-procedure-note.ts:154-156](src/lib/claude/generate-procedure-note.ts#L154-L156) — `assessment_summary`: reference is "necessitating further pain management intervention" — a phrase that presupposes continued need rather than improvement.

[generate-procedure-note.ts:196](src/lib/claude/generate-procedure-note.ts#L196) — `prognosis`: reference is "Due to the chronic nature of the injury, the prognosis is guarded." Again a persistence-leaning reference.

[generate-procedure-note.ts:182-184](src/lib/claude/generate-procedure-note.ts#L182-L184) — `procedure_followup`: is the only section whose text references `procedure_number`, and only to size the future series ("Additional PRP injections may be considered based on his progress. Patient was reminded of the potential need for 1-2 additional PRP injections, depending on the degree of symptom improvement.").

### 2. Data that reaches the prompt for repeat procedures

**File**: [src/actions/procedure-notes.ts](src/actions/procedure-notes.ts)

`gatherProcedureNoteSourceData` ([lines 28-237](src/actions/procedure-notes.ts#L28-L237)) assembles the input for `generateProcedureNoteFromData`. The specific interval-aware fields it produces are:

- `procedureRecord.procedure_number` ([line 168](src/actions/procedure-notes.ts#L168)) — the 1-based index of this procedure in the case.
- `priorProcedure` ([lines 190-197](src/actions/procedure-notes.ts#L190-L197)) — populated from the most recent non-current procedure on the case ([lines 85-94](src/actions/procedure-notes.ts#L85-L94)): `procedure_date`, `procedure_number`, and the prior procedure's `pain_score_min` / `pain_score_max` from `vital_signs` ([lines 109-123](src/actions/procedure-notes.ts#L109-L123)).
- `vitalSigns.pain_score_min` / `pain_score_max` — current-visit pain ([line 51](src/actions/procedure-notes.ts#L51)).

`ProcedureNoteInputData` ([generate-procedure-note.ts:15-100](src/lib/claude/generate-procedure-note.ts#L15-L100)) only exposes **one** prior procedure. It is always the single most recent; earlier injections in a 3- or 4-procedure series are not represented. There is no trajectory array. The prompt therefore cannot compare across more than two time points even if instructed to.

Query used for the prior procedure ([procedure-notes.ts:85-94](src/actions/procedure-notes.ts#L85-L94)):

```ts
supabase
  .from('procedures')
  .select('id, procedure_date, procedure_number')
  .eq('case_id', caseId)
  .neq('id', procedureId)
  .is('deleted_at', null)
  .order('procedure_date', { ascending: false })
  .limit(1)
  .maybeSingle()
```

### 3. Discharge Note — a different tone explicitly required

**File**: [src/lib/claude/generate-discharge-note.ts](src/lib/claude/generate-discharge-note.ts)

The Discharge Note's system prompt ([lines 104-181](src/lib/claude/generate-discharge-note.ts#L104-L181)) embeds a `=== CONTEXT ===` section that explicitly shifts tone:

[generate-discharge-note.ts:124](src/lib/claude/generate-discharge-note.ts#L124):
> `This is a DISCHARGE note — the patient has COMPLETED their PRP treatment series and is being evaluated for discharge from active interventional pain management care. The tone should reflect completion, improvement, and forward-looking recommendations. Summarize the entire treatment course and outcomes.`

Every section of the discharge prompt is written to presuppose improvement:
- `subjective` para 1: "Report sustained and progressive improvement in pain severity, functional capacity, and quality of life." ([line 130](src/lib/claude/generate-discharge-note.ts#L130))
- `subjective` para 2: "Region-by-region symptom status — current pain ratings, **quality of remaining pain (mild stiffness vs sharp), improvement in mobility, resolution of radicular symptoms**. Compare to pre-treatment baseline." ([line 131](src/lib/claude/generate-discharge-note.ts#L131))
- `objective_general`: "demonstrates improved posture and ease of movement compared to prior visits." ([line 141](src/lib/claude/generate-discharge-note.ts#L141))
- `objective_cervical` / `objective_lumbar`: "Emphasize improvement from baseline." ([lines 144, 148](src/lib/claude/generate-discharge-note.ts#L144))
- `assessment`: "sustained clinical improvement following completion of a PRP treatment" ([line 161](src/lib/claude/generate-discharge-note.ts#L161))
- `prognosis`: "Favorable prognosis. Meaningful and sustained improvement in pain control, mobility, and functional capacity." ([lines 174-175](src/lib/claude/generate-discharge-note.ts#L174-L175))

Pain data passed in uses every procedure's `pain_score_max` as a trendline across the whole series ([line 136](src/lib/claude/generate-discharge-note.ts#L136)):
> `When describing pain progression across visits elsewhere in the note, use each procedure's pain_score_max as the trendline value.`

The discharge prompt therefore has both the data (full array of procedures with their pain scores on [generate-discharge-note.ts:29-38](src/lib/claude/generate-discharge-note.ts#L29-L38)) and the instruction to narrate a trajectory. The procedure note prompt has neither.

### 4. Initial Visit / Pain Evaluation Visit — where "persistent" and "intermittent" live

**File**: [src/lib/claude/generate-initial-visit.ts](src/lib/claude/generate-initial-visit.ts)

The Initial Visit prompt controls the use of "Persistent / Intermittent" in the Chief Complaint bullets ([line 95-96](src/lib/claude/generate-initial-visit.ts#L95-L96)):

> `4. CHIEF COMPLAINT (~1 intro sentence + bullet list):`
> `Brief intro sentence, then "• " bullet per complaint with: region, persistent/intermittent, pain rating X–X/10, radiation status, aggravating factors, alleviating factors. ...`
> `Reference: "• Neck pain: Persistent, rated 7–8/10. ..."`

This `persistent/intermittent` axis is described as a model-choice, not a data-derived flag. The source of the choice comes from `providerIntake.chief_complaints` rather than a structured `pain_persistence` field (see `ProviderIntake` typing at [generate-initial-visit.ts:451-457](src/lib/claude/generate-initial-visit.ts#L451-L457)).

The Pain Evaluation Visit variant (triggered when imaging findings exist, [lines 24-33](src/lib/claude/generate-initial-visit.ts#L24-L33)) adds a `=== PRIOR VISIT REFERENCE (READ-ONLY) ===` block ([lines 222-236](src/lib/claude/generate-initial-visit.ts#L222-L236)) that allows comparative language:

> `3. Physical Examination: ... You MAY add one brief comparative sentence at the end of each region: "Compared to the initial evaluation, cervical ROM has [improved/worsened/remained unchanged]." Use priorVisitData.rom_data and priorVisitData.physical_exam for the comparison basis only.`

So this prompt *does* expose the explicit improved/worsened/unchanged vocabulary — but only for Pain Evaluation Visit notes, not for Procedure Notes.

### 5. Chiropractic extraction — a structured trajectory enum

**File**: [src/lib/claude/extract-chiro.ts:118](src/lib/claude/extract-chiro.ts#L118)

```ts
enum: ['improving', 'stable', 'plateauing', 'worsening', 'null']
```

This is the only place in the AI layer where improvement trajectory is represented as a constrained enum. It is produced by the chiropractic extraction pipeline and is not currently fed into the procedure-note generator's input data ([ProcedureNoteInputData](src/lib/claude/generate-procedure-note.ts#L15-L100) has no `chiroExtraction` field).

**Files**:

Fields like `pain_range`, `pain_intensity`, `pain_persistence`, and `pain_score_min/max` appear in these extraction schemas. `pain_score_min/max` is the only one connected to procedure-note generation via `vital_signs` rows keyed to `procedure_id` ([procedure-notes.ts:49-55](src/actions/procedure-notes.ts#L49-L55)).

### 7. Per-section regeneration uses the same prompt

`regenerateProcedureNoteSection` ([generate-procedure-note.ts:302-331](src/lib/claude/generate-procedure-note.ts#L302-L331)) and `regenerateProcedureNoteSectionAction` ([procedure-notes.ts:665-712](src/actions/procedure-notes.ts#L665-L712)) re-use the same `SYSTEM_PROMPT` constant plus a small suffix ("You are regenerating ONLY the '{sectionLabel}' section …"). Regenerating the `subjective` section therefore re-runs the same "persistent symptoms, functional limitations, current pain" instruction.

### 8. PDF template does not mutate tone

[src/lib/pdf/procedure-note-template.tsx](src/lib/pdf/procedure-note-template.tsx) renders the `procedure_notes` row's fields verbatim — the only transformation around repeat visits is an ordinal label:
- [procedure-note-template.tsx:199](src/lib/pdf/procedure-note-template.tsx#L199): `<Text>{ordinal(data.procedureNumber)} Injection</Text>`

No PDF-side logic adjusts the subjective narrative or interpolates pain-change language.

## Architecture Documentation

### Generator layout
Each clinical document type has its own generator file under [src/lib/claude/](src/lib/claude/):
- `generate-procedure-note.ts` — per-procedure report
- `generate-discharge-note.ts` — end-of-series summary
- `generate-initial-visit.ts` — initial/pain evaluation visit

Each file owns a single `SYSTEM_PROMPT` string (or builder) and a tool definition enumerating structured output fields. A corresponding server action in [src/actions/](src/actions/) gathers source data, computes a `source_data_hash`, and persists the result in a matching `*_notes` table.

### Input-data pipeline for procedure notes
`gatherProcedureNoteSourceData` runs 8 parallel Supabase queries ([procedure-notes.ts:42-100](src/actions/procedure-notes.ts#L42-L100)) covering the procedure, vitals, case, pain-management extraction, MRI extractions, finalized initial visit note, the most-recent prior procedure, and clinic settings. Prior-procedure vitals require a separate follow-up query because `vital_signs` is joined by `procedure_id`, not by case.

### Tone control location
The entire tone/voice of a procedure note is expressed in the prompt string in [generate-procedure-note.ts:102-203](src/lib/claude/generate-procedure-note.ts#L102-L203). There is no per-visit-type branching, no improvement-detection code path, and no templating layer that conditionally swaps adjectives based on the delta between `priorProcedure.pain_score_max` and the current `vitalSigns.pain_score_max`. The phrase **"persistent symptoms"** appears verbatim in the instruction for the `subjective` section for every repeat visit.
