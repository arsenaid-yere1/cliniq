# Research: Clinical Case Summary Generation — Best Design Recommendation

**Date**: 2026-03-08T12:00:00-07:00

## Research Question

What is the best design for Story 2.3 — Clinical Case Summary Generation? The feature should produce a summarized clinical case view containing chief complaint, imaging findings, prior treatment, symptoms timeline, and suggested diagnoses, with provider-editable fields.

## Summary

The recommended design is a **server-generated aggregate summary stored in a new `case_summaries` table**, rendered as a multi-section Card on the case overview page with inline-edit capability. Summary generation uses the existing Anthropic Claude integration (tool-use pattern) to synthesize data from MRI extractions, chiro extractions, case details, and procedures into a structured summary. The provider review/edit flow mirrors the established extraction review pattern but is lighter-weight — no split-pane PDF viewer needed since the summary is synthesized, not extracted from a single document.

## Recommended Design

### Option Analysis

| Option | Description | Pros | Cons |
|---|---|---|---|
| **A. Real-time aggregation (no storage)** | Query all related tables on page load, render raw data | No new table, always fresh | No AI synthesis, no "suggested diagnoses", no editable summary, slow page loads |
| **B. AI-generated summary stored in DB** | Claude synthesizes a structured summary from all clinical data, stored in `case_summaries` table | Matches existing AI pattern, provider can edit, fast reads, supports suggested diagnoses | Requires regeneration trigger, stale data possible |
| **C. Hybrid: stored summary + live data fallback** | Store AI summary but also show live data sections alongside | Best of both, never stale for raw data | More complex UI, confusing UX (which is the "truth"?) |

### Recommendation: Option B — AI-Generated Summary Stored in DB

This is the best fit because:
1. **Suggested diagnoses** requires AI — raw aggregation can't do this
2. **Matches the established extraction pattern** almost exactly (Claude tool-use → structured output → Zod validate → DB → provider review)
3. **Provider editability** maps directly to the `provider_overrides` JSONB pattern used by MRI and chiro extractions
4. **Fast reads** — the overview page already loads 3 server actions in parallel; adding a 4th for a pre-computed summary is trivial

### Architecture

#### 1. New Database Table: `case_summaries`

```sql
CREATE TABLE case_summaries (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  case_id UUID NOT NULL REFERENCES cases(id),

  -- Structured summary fields
  chief_complaint TEXT,
  imaging_findings JSONB,        -- [{body_region, summary, key_findings[], severity}]
  prior_treatment JSONB,         -- {modalities[], total_visits, treatment_period, gaps[]}
  symptoms_timeline JSONB,       -- {onset, progression[], current_status, pain_levels[]}
  suggested_diagnoses JSONB,     -- [{diagnosis, icd10_code, confidence, supporting_evidence}]

  -- AI metadata (same pattern as extractions)
  ai_model TEXT,
  ai_confidence TEXT CHECK (ai_confidence IN ('high', 'medium', 'low')),
  raw_ai_response JSONB,
  extraction_notes TEXT,

  -- Provider review (same pattern as extractions)
  review_status TEXT NOT NULL DEFAULT 'pending_review'
    CHECK (review_status IN ('pending_review', 'approved', 'edited', 'rejected')),
  provider_overrides JSONB,
  reviewed_by_user_id UUID REFERENCES users(id),
  reviewed_at TIMESTAMPTZ,

  -- Generation tracking
  generation_status TEXT NOT NULL DEFAULT 'pending'
    CHECK (generation_status IN ('pending', 'processing', 'completed', 'failed')),
  generation_attempts INTEGER NOT NULL DEFAULT 0,
  source_data_hash TEXT,         -- hash of input data to detect staleness

  -- Standard audit fields
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_by_user_id UUID REFERENCES users(id),
  updated_by_user_id UUID REFERENCES users(id),
  deleted_at TIMESTAMPTZ,

  UNIQUE (case_id, deleted_at)   -- one active summary per case
);
```

Key design decisions:
- **One active summary per case** (enforced by unique constraint with soft-delete)
- **`source_data_hash`**: SHA-256 hash of the concatenated input data; when source data changes (new extraction approved, new procedure added), the hash won't match, signaling staleness
- **`provider_overrides`** JSONB: identical pattern to `mri_extractions.provider_overrides` — stores the full edited form values when a provider modifies the summary

#### 2. Claude Integration: `src/lib/claude/generate-summary.ts`

```
SYSTEM_PROMPT → clinical synthesis instructions
SUMMARY_TOOL → extract_case_summary tool definition matching the JSONB schemas above
generateCaseSummary(inputData) → structured summary
```

The tool schema will have 5 top-level fields matching the DB columns. Claude receives:
- Case details (accident type, date, description)
- All approved MRI extraction data
- All approved chiro extraction data
- Procedure history

This follows the exact same pattern as `extract-mri.ts` and `extract-chiro.ts`.

#### 3. Server Actions: `src/actions/case-summaries.ts`

| Action | Purpose |
|---|---|
| `generateCaseSummary(caseId)` | Orchestrate: gather input data → call Claude → write to DB |
| `getCaseSummary(caseId)` | Read the active summary for a case |
| `saveCaseSummaryEdits(caseId, formValues)` | Save provider edits to `provider_overrides` |
| `approveCaseSummary(caseId)` | Set `review_status: 'approved'` |
| `checkSummaryStaleness(caseId)` | Compare current source hash vs stored hash |

The orchestration flow mirrors the extraction actions:
1. Auth user
2. Soft-delete existing summary (if any)
3. Insert with `generation_status: 'processing'`
4. Gather all source data (approved extractions, case details, procedures)
5. Hash the source data
6. Call `generateCaseSummary()` from the Claude utility
7. Write results + hash to DB
8. `revalidatePath()`

#### 5. Staleness Detection

When source data changes (new extraction approved, existing extraction re-extracted, procedure added/removed), the summary becomes potentially stale. Two approaches:

**Recommended: Lazy staleness check**
- On page load, `checkSummaryStaleness()` computes the current source data hash and compares to stored `source_data_hash`
- If mismatched, render "Outdated" badge with "Regenerate" button
- No background jobs, no real-time subscriptions needed
- Simple, fits the existing architecture (no cron, no webhooks)

**Not recommended for MVP**: Automatic regeneration triggers (too complex, could waste API calls on intermediate edits).

#### 6. Zod Validation: `src/lib/validations/case-summary.ts`

Two schemas (same pattern as extraction validations):
- `caseSummaryResultSchema` — validates Claude's tool output
- `caseSummaryEditSchema` — validates provider edit form

## Architecture Insights

1. **The codebase has a clear, repeatable pattern for AI features**: prompt + tool schema → Server Action orchestration → DB storage → provider review UI. Story 2.3 should follow this exactly.
2. **No streaming needed**: The existing extraction features use standard request-response with no streaming. Summary generation will be similar in latency (the input data is structured text, not a PDF, so it may even be faster).
3. **`provider_overrides` JSONB is the established edit pattern**: Rather than modifying the AI-generated fields directly, edits are stored separately. This preserves the original AI output for auditing while letting providers customize.
4. **The overview page is the right home**: The case overview already aggregates stats, case details, and recent activity. A clinical summary card fits naturally as another section, positioned prominently since it's the most clinically valuable view.
5. **Lazy staleness > automatic regeneration**: The project has no background job infrastructure. A simple hash comparison on page load is consistent with the existing architecture and avoids unnecessary complexity.

## Open Questions

1. **Should the summary be auto-generated when all extractions are approved?** Or only on explicit provider request? (Recommendation: explicit "Generate" button for MVP — avoids surprise API costs)
2. **Should the summary tab/section be visible before any extractions exist?** (Recommendation: show the card with disabled state + message "Upload and approve clinical documents to generate a summary")
3. **How much of the symptoms timeline should be AI-synthesized vs. pulled directly from structured chiro data?** (Recommendation: AI synthesizes a narrative from the structured data, with the raw data available on the clinical tab for reference)
4. **Should suggested diagnoses show a confidence indicator?** (Recommendation: yes — the chiro extraction already stores `is_primary`, and Claude can assess confidence based on supporting evidence strength)
