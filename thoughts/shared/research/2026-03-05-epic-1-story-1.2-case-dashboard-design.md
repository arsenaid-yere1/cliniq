# Research: Epic 1 — Story 1.2: View Case Dashboard Design

**Date**: 2026-03-05
**Researcher**: Claude
**Git Commit**: a4b10cb

## Research Question

Design research for Story 1.2 (View Case Dashboard): how should we structure the dashboard to show uploaded MRI reports, chiropractor reports, generated documents, procedures, and billing history in one place, with document search and a timeline view of visits and procedures?

## Summary

Story 1.1 built a case dashboard shell with a sidebar, overview tab, and disabled "Coming Soon" navigation links for Documents, Clinical Data, Procedures, and Invoices. Story 1.2 fills these in. Research across clinical EHR design, personal injury case management tools (Clio, ChiroTouch, CaseFleet), and shadcn/ui component availability points to a **tabbed layout with a persistent case header**, where each tab uses the appropriate UI pattern for its content type (cards for documents, tables for procedures/billing, vertical timeline for encounters).

## Current State (from Story 1.1)

### What Exists

- **Case dashboard layout**: `src/app/(dashboard)/patients/[caseId]/layout.tsx` — fetches case data, renders `CaseSidebar` + children
- **Case sidebar**: `src/components/patients/case-sidebar.tsx` — patient name, case number (with copy), DOB, accident date, status badge, navigation links (Overview active; Documents, Clinical Data, Procedures, Invoices disabled)
- **Case overview**: `src/components/patients/case-overview.tsx` — case summary card (demographics, case details, attorney) + activity summary card (hardcoded zeros) + quick actions (all disabled)
- **Database tables**: `patients`, `cases`, `attorneys`, `case_status_history`, `audit_logs`, `users`
- **No document, procedure, invoice, or billing tables exist yet**

### What's Missing

- Database tables: `documents`, `procedures`, `procedure_types`, `invoices`, `invoice_line_items`, `payments`, `appointments`/`encounters`
- Document upload/storage infrastructure
- All tab content beyond Overview
- Timeline component
- Search/filter functionality within case tabs

## Detailed Findings

### 1. Dashboard Architecture: Tabbed vs. Unified

**Recommendation: Tabbed layout with persistent sidebar (extend existing architecture)**

EHR usability research shows clinicians average 1.4 task switches per minute. Combining all content types into a single scrolling page causes cognitive overload. The established clinical pattern uses categorized tabs with a persistent case header.

The existing Story 1.1 sidebar already has navigation links for each tab. Story 1.2 should activate these links, routing to dedicated sub-pages under `[caseId]/`.

**Route structure:**
```
/patients/[caseId]/                → Overview (existing, enhance with stat cards)
/patients/[caseId]/documents       → Documents tab
/patients/[caseId]/procedures      → Procedures tab
/patients/[caseId]/billing         → Billing (was "Invoices" in sidebar)
/patients/[caseId]/timeline        → Timeline view
```

Each route gets a shareable, bookmarkable URL — important for clinic staff sharing specific case sections.

### 2. Content Type → Layout Matrix

| Tab | Layout Pattern | Rationale |
|---|---|---|
| **Overview** | Stat cards + summary fields + recent activity preview | At-a-glance dashboard; leads users to other tabs |
| **Documents** | Search bar + filter chips + card grid/list | Documents need visual context (type icon, category, date) |
| **Procedures** | TanStack Table | Tabular comparative data: dates, CPT codes, amounts |
| **Billing** | TanStack Table + summary stat card | Line-item financial data; header shows totals |
| **Timeline** | Vertical timeline with event-type icons | Sequential narrative; not a comparison task |

### 3. Overview Tab (Enhanced)

Extend the existing `case-overview.tsx` to add:

**Stat cards row** (4 cards, full width):
- Total Documents uploaded (count)
- Total Procedures recorded (count)
- Total Billed / Balance Due (financial summary)
- Last Visit/Encounter date

**Recent Activity** — 3-5 item timeline preview with "View All →" linking to Timeline tab

**Quick Actions** — Enable buttons: Upload Document, Record Procedure, Create Invoice (currently disabled with "Coming Soon" tooltips)

**Case Summary** — Keep existing demographics/case/attorney display

### 4. Documents Tab

**Search and filter pattern:**
1. **Keyword search input** at top (searches filename, document type, notes) — debounced 300ms
2. **Filter chips** below search:
   - Document Type: MRI Report | Chiro Report | Generated | Other
   - Date Range: Last 30 days | Last 6 months | All | Custom
   - Status: Reviewed | Pending Review | All
3. **Active filter badges** with dismiss buttons
4. **Result count**: "Showing 4 of 12 documents"

**Document display**: Card-based list (not table) — each card shows:
- File type icon (PDF, image, etc.)
- Document name / filename
- Document type badge (MRI, Chiro Report, Generated, etc.)
- Upload date
- Status badge (Reviewed / Pending)
- Actions: Download, Preview (Sheet slide-over), Delete

**Search implementation**: For MVP (small document volume per case), client-side filtering is sufficient. Use `use-debounce` for the search input. For multi-filter state, consider `nuqs` for URL-based state management.

```bash
npm install use-debounce
# Optional for complex multi-filter:
npm install nuqs
```

### 5. Procedures Tab

**TanStack Table with columns:**

| Column | Notes |
|---|---|
| Date | Date of procedure, sortable |
| Procedure Name | Name or description |
| CPT Code | Standard billing code |
| Provider | Who performed it |
| Notes | Truncated, expandable |
| Actions | Edit, Delete |

Include a "Record Procedure" button in the toolbar. Use a `Sheet` (slide-over) or `Dialog` for the add/edit form.

### 6. Billing Tab

**TanStack Table with columns:**

| Column | Notes |
|---|---|
| Date | Invoice or service date, sortable |
| Description | Procedure name or line item |
| Provider | Rendering provider |
| Amount Billed | Currency formatted |
| Amount Paid | Currency formatted |
| Balance | Computed: billed minus paid |
| Status | Badge: Paid / Pending / Partial / Denied / Overdue |
| Actions | View invoice, Record payment |

**Status badge colors:**
- Paid: green
- Pending: amber/yellow
- Partial: blue
- Denied: red
- Overdue: red (bold/outlined)
- Draft: gray

**Summary card above table:**
- Total Billed
- Total Paid
- Balance Due

These values already exist as denormalized fields on the `cases` table (`total_billed`, `total_paid`, `balance_due`).

### 7. Timeline Tab

**Recommendation: Vertical timeline with color-coded event types**

**Community component: `timDeHof/shadcn-timeline`**
- Built on shadcn/ui + Framer Motion
- Props: `status`, `icon`, `iconColor`, `size`, `loading`, `error`
- SSR compatible, accessible with ARIA
- Install by copying component source into `src/components/ui/timeline.tsx`

**Event types for PI clinic timeline:**

| Event Type | Icon | Color |
|---|---|---|
| Encounter/Visit | `Stethoscope` or `Calendar` | Blue (primary) |
| Procedure | `Activity` or `ClipboardList` | Green (secondary) |
| Document Uploaded | `FileText` | Gray (muted) |
| Status Change | `Flag` | Yellow (accent) |
| Invoice Created | `Receipt` | Purple (info) |

**Timeline items show:** event type badge, date, provider name, description. Clicking expands inline (accordion) for details — no modal.

**Alternative: shadcn official blocks**
- `tables-activity-log` — audit-style timestamped entries
- `tables-timeline` — date-grouped sections with event badges

The vertical connector-line style (`timDeHof/shadcn-timeline`) is more recognizable as a "timeline" to users and better suited for a medical encounter history.

### 8. Database Schema Requirements (New Tables)

Story 1.2 requires new database tables. Based on the research data model from Story 1.1:

```sql
-- Documents table
create table public.documents (
  id uuid primary key default uuid_generate_v4(),
  case_id uuid not null references public.cases(id),
  document_type text not null check (document_type in ('mri_report', 'chiro_report', 'generated', 'other')),
  file_name text not null,
  file_path text not null,            -- Supabase Storage path
  file_size_bytes bigint,
  mime_type text,
  status text not null default 'pending_review' check (status in ('pending_review', 'reviewed')),
  notes text,
  uploaded_by_user_id uuid references public.users(id),
  reviewed_by_user_id uuid references public.users(id),
  reviewed_at timestamptz,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  deleted_at timestamptz,
  created_by_user_id uuid references public.users(id),
  updated_by_user_id uuid references public.users(id)
);

-- Procedures table
create table public.procedures (
  id uuid primary key default uuid_generate_v4(),
  case_id uuid not null references public.cases(id),
  procedure_date date not null,
  procedure_name text not null,
  cpt_code text,
  provider_id uuid references public.users(id),
  notes text,
  charge_amount numeric(10,2),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  deleted_at timestamptz,
  created_by_user_id uuid references public.users(id),
  updated_by_user_id uuid references public.users(id)
);

-- Invoices table
create table public.invoices (
  id uuid primary key default uuid_generate_v4(),
  case_id uuid not null references public.cases(id),
  invoice_number text not null unique,
  invoice_date date not null default current_date,
  due_date date,
  status text not null default 'draft' check (status in ('draft', 'pending', 'paid', 'partial', 'denied', 'overdue')),
  total_amount numeric(10,2) not null default 0,
  paid_amount numeric(10,2) not null default 0,
  notes text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  deleted_at timestamptz,
  created_by_user_id uuid references public.users(id),
  updated_by_user_id uuid references public.users(id)
);

-- Invoice line items
create table public.invoice_line_items (
  id uuid primary key default uuid_generate_v4(),
  invoice_id uuid not null references public.invoices(id),
  procedure_id uuid references public.procedures(id),
  description text not null,
  cpt_code text,
  quantity integer not null default 1,
  unit_price numeric(10,2) not null,
  total_price numeric(10,2) not null,
  created_at timestamptz not null default now()
);

-- Payments table
create table public.payments (
  id uuid primary key default uuid_generate_v4(),
  invoice_id uuid not null references public.invoices(id),
  payment_date date not null default current_date,
  amount numeric(10,2) not null,
  payment_method text,
  reference_number text,
  notes text,
  created_at timestamptz not null default now(),
  created_by_user_id uuid references public.users(id)
);

-- Indexes
create index idx_documents_case_id on public.documents(case_id);
create index idx_procedures_case_id on public.procedures(case_id);
create index idx_invoices_case_id on public.invoices(case_id);
create index idx_invoice_line_items_invoice_id on public.invoice_line_items(invoice_id);
create index idx_payments_invoice_id on public.payments(invoice_id);

-- Updated_at triggers
create trigger set_updated_at before update on public.documents for each row execute function update_updated_at();
create trigger set_updated_at before update on public.procedures for each row execute function update_updated_at();
create trigger set_updated_at before update on public.invoices for each row execute function update_updated_at();

-- RLS
alter table public.documents enable row level security;
alter table public.procedures enable row level security;
alter table public.invoices enable row level security;
alter table public.invoice_line_items enable row level security;
alter table public.payments enable row level security;

create policy "Authenticated users full access" on public.documents for all using (auth.role() = 'authenticated');
create policy "Authenticated users full access" on public.procedures for all using (auth.role() = 'authenticated');
create policy "Authenticated users full access" on public.invoices for all using (auth.role() = 'authenticated');
create policy "Authenticated users full access" on public.invoice_line_items for all using (auth.role() = 'authenticated');
create policy "Authenticated users full access" on public.payments for all using (auth.role() = 'authenticated');
```

## Architecture Insights

1. **Tabbed with URL routing** > client-side tabs — gives shareable URLs for each dashboard section, important for clinic staff sharing links
2. **Cards for documents, tables for structured data** — documents need visual context (type icon, status); procedures/billing are comparative tabular data
3. **Vertical timeline** for encounter history — not a table; the narrative sequential pattern matches how providers think about case history
4. **Client-side filtering for MVP** — per-case data volumes (5-30 documents, 10-50 procedures) don't warrant server-side pagination initially
5. **Existing sidebar navigation already has the tab structure** — just enable the disabled links and add route handlers
6. **Supabase Storage** for document files — DB stores metadata + storage path only, consistent with Story 1.1 research decisions

## Anti-Patterns to Avoid

- **Infinite scroll with mixed types** — never combine documents, procedures, and billing in one unfiltered list
- **Modal-only document preview** — prefer a `Sheet` slide-over to keep context
- **Color-only status indicators** — always pair with text labels (WCAG 2.1 AA)
- **Date fields without presets** — always offer quick presets (Last 30 days, Last 6 months)
- **Global search in tab context** — keep search scoped to the current tab's content type

## Resolved Questions

1. **Document upload destination** — Story 1.2 focuses on **display capability only**. The dashboard will show documents that exist in the database, but actual file upload (Supabase Storage integration) is deferred to a later story (Epic 2). For now, the Documents tab renders document metadata from the `documents` table; seed/test data can be inserted manually.
2. **Rename "Invoices" to "Billing"** — Yes, rename to "Billing". The acceptance criteria says "Billing history" and the tab will show both invoices and payments, not just invoices. The sidebar nav link, route path (`/billing`), and component names should all use "Billing".
3. **Clinical Data tab** — Drop from the sidebar in Story 1.2. It's not in the acceptance criteria and has no backing feature yet. "Clinical Data" refers to AI-extracted structured data from documents (Epic 2 scope). Re-add when extraction is implemented — at that point decide whether it's a separate tab or a sub-view within Documents.
4. **Timeline data sources** — Build from existing data: merge `case_status_history` + `documents` + `procedures` + `invoices` into a chronological feed. No `encounters`/`appointments` table needed yet. Event types: "Status Change", "Document Added", "Procedure", "Invoice Created". An encounters table can be added in a later story when appointment scheduling ships — timeline will automatically include those events.
