## Research Question

Explore the best design and implementation options for Story 1.3: Patient Document Repository — accepting PDF, DOCX, and image uploads with document type labeling, secure storage, and file versioning.

## Summary

The existing codebase already has substantial document infrastructure (DB schema, types, server actions, list/card UI with "Coming Soon" stubs). The key missing pieces are: Supabase Storage bucket setup, upload flow, versioning schema, file preview, and download. The recommended approach is a **two-phase signed URL upload** (server action generates URL, client uploads directly to Supabase) with a **separate `document_versions` table** for versioning, **react-dropzone** for drag-and-drop UI, and **react-pdf** for PDF preview.

### 2. Upload Architecture: Three Options

#### Option A: Two-Phase Signed URL Upload (Recommended)

**Flow**: Server action validates access + generates signed upload URL → Client uploads directly to Supabase Storage → Server action saves metadata to DB.

**Pros**:
- Bypasses Next.js server action 1MB body limit entirely
- File never passes through the Next.js server (lower latency, lower bandwidth cost)
- RLS checked at URL generation time
- Works for any file size

**Cons**:
- Two server action calls per upload (URL generation + metadata save)
- Slightly more complex client code

**Implementation sketch**:

```ts
// Server action 1: Generate signed upload URL
export async function getSignedUploadUrl(input: { caseId: string; fileName: string; fileType: string }) {
  const supabase = await createClient()
  // Verify case access
  const storagePath = `cases/${input.caseId}/${Date.now()}-${input.fileName}`
  const { data, error } = await supabase.storage
    .from('case-documents')
    .createSignedUploadUrl(storagePath)
  return { signedUrl: data.signedUrl, token: data.token, path: storagePath }
}

// Client: Upload directly to Supabase
const { error } = await supabase.storage
  .from('case-documents')
  .uploadToSignedUrl(result.path, result.token, file)

// Server action 2: Save metadata
export async function saveDocumentMetadata(input: DocumentMetadata) { ... }
```

#### Option B: Direct Server Action Upload (Simpler, Limited)

**Flow**: File sent as FormData to server action → Server action uploads to Supabase Storage + saves metadata.

**Pros**: Single action call, simpler client code.
**Cons**: Requires raising `serverActions.bodySizeLimit` in next.config.ts; file routes through Next.js server; limited to ~10MB practically.

**When to use**: Only if all documents will be under 5MB (unlikely for MRI reports).

#### Option C: TUS Resumable Upload (For Large Files)

**Flow**: Client uses `tus-js-client` to upload directly to Supabase's TUS endpoint with real progress callbacks.

**Pros**: Real upload progress, resumable on failure, handles any file size.
**Cons**: Additional dependency (`tus-js-client`), more complex setup, requires session access token management.

**When to use**: If MRI reports or images routinely exceed 6MB and real progress bars are required.

### Recommendation

**Use Option C** (TUS resumable upload) for real upload progress. This gives genuine `onProgress` callbacks, resumability on network failure, and handles any file size. The `tus-js-client` dependency is lightweight (~15KB gzipped). Option A's signed URL approach lacks real progress events. Option B is not recommended due to the body size limitation.

The flow becomes:
1. Server action validates file metadata + verifies case access
2. Client uploads directly to Supabase's TUS endpoint using the user's session token
3. `onProgress` callback drives shadcn `<Progress>` bars per file
4. On completion, server action saves metadata to the `documents` table

### 3. File Versioning: Minimal "Replace File" Approach

**Decision**: Minimal versioning — "replace file" only, no version history UI.

Supabase Storage has **no native versioning**. Given the minimal scope, the simplest approach is to use the existing `documents` table as-is:

#### Implementation

When a user replaces a document:
1. Upload the new file to a new storage path (timestamp ensures uniqueness)
2. Update the existing `documents` row with the new `file_path`, `file_name`, `file_size_bytes`, `mime_type`
3. Optionally soft-delete the old storage object (or keep it — storage is cheap, and HIPAA requires retention)

```ts
export async function replaceDocument(documentId: string, newFileMeta: FileMeta) {
  const supabase = await createClient()
  const { data, error } = await supabase
    .from('documents')
    .update({
      file_path: newFileMeta.storagePath,
      file_name: newFileMeta.fileName,
      file_size_bytes: newFileMeta.fileSize,
      mime_type: newFileMeta.mimeType,
      updated_by_user_id: userId,
    })
    .eq('id', documentId)
    .select()
    .single()
  // Old file remains in storage (HIPAA retention)
  return { data, error: error?.message }
}
```

**No schema changes needed.** The existing `documents` table already has all the required columns. The old file stays in Supabase Storage (HIPAA 6-year retention) even though the DB row now points to the new file. Full version history UI can be added later if needed by introducing a `document_versions` table.

### 4. Supabase Storage Setup

#### Bucket Configuration

```sql
INSERT INTO storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
VALUES (
  'case-documents',
  'case-documents',
  false,  -- private bucket
  52428800,  -- 50 MB limit
  ARRAY[
    'application/pdf',
    'image/jpeg',
    'image/png',
    'image/webp',
    'application/vnd.openxmlformats-officedocument.wordprocessingml.document'
  ]
);
```

#### Storage Path Convention

```
case-documents/cases/{case_id}/{timestamp}-{filename}
```

Timestamp prefix prevents naming collisions and provides natural ordering.

#### RLS Policies on `storage.objects`

```sql
-- Upload: authenticated users can upload to any case
CREATE POLICY "Authenticated users can upload documents"
ON storage.objects FOR INSERT TO authenticated
WITH CHECK (bucket_id = 'case-documents');

-- Read: authenticated users can read any document
CREATE POLICY "Authenticated users can read documents"
ON storage.objects FOR SELECT TO authenticated
USING (bucket_id = 'case-documents');

-- Delete: authenticated users can delete documents
CREATE POLICY "Authenticated users can delete documents"
ON storage.objects FOR DELETE TO authenticated
USING (bucket_id = 'case-documents');
```

Note: This matches the existing RLS pattern on the `documents` table (all authenticated users have full access). More granular case-scoped policies can be added later when RBAC is implemented.

#### Upload Dialog/Sheet Component

**Recommended pattern**: shadcn `Sheet` (slides from right) containing:
1. **Dropzone area** (react-dropzone + shadcn Card styling)
   - Accepts: PDF, DOCX, JPG, PNG, WEBP
   - Max size: 50MB
   - Shows drag-active state
2. **Staged file list** with per-file:
   - File icon + name + size
   - Document type `Select` (MRI Report, Chiropractor Report, Other)
   - Remove button (before upload starts)
   - Real upload progress bar via `tus-js-client` `onProgress`
   - Cancel button during upload (`upload.abort()`)
3. **Upload all button** at bottom

**Integration with react-hook-form**: Use `Controller` wrapping `useDropzone`'s `onDrop` callback to manage file state. Validate file type/size with Zod schema.

#### Zod Validation Schema

```ts
const documentUploadSchema = z.object({
  files: z
    .array(z.instanceof(File))
    .min(1, 'At least one file required')
    .refine(
      files => files.every(f => f.size <= 50 * 1024 * 1024),
      'Each file must be under 50MB'
    )
    .refine(
      files => files.every(f =>
        ['application/pdf', 'image/jpeg', 'image/png', 'image/webp',
         'application/vnd.openxmlformats-officedocument.wordprocessingml.document'
        ].includes(f.type)
      ),
      'Unsupported file type'
    ),
  documentType: z.enum(['mri_report', 'chiro_report', 'other']),
})
```

#### Download Flow

Server action generates a signed URL with short expiry (1 hour), returns to client for `<a>` tag or `window.open()`.

```ts
export async function getDocumentDownloadUrl(storagePath: string) {
  const supabase = await createClient()
  const { data, error } = await supabase.storage
    .from('case-documents')
    .createSignedUrl(storagePath, 3600, { download: true })
  return { url: data?.signedUrl, error: error?.message }
}
```

### 6. Security Considerations

| Concern | Mitigation |
|---|---|
| File never exposed publicly | Private bucket + signed URLs with 1-hour expiry |
| MIME type validation | Validated client-side (Zod) + server-side (bucket `allowed_mime_types`) |
| File size limits | Enforced at bucket level (50MB) + client-side validation |
| Access control | RLS on `storage.objects` + case access check in server actions |
| Signed URL leakage | Short expiry (3600s), RLS checked at generation, never logged |
| HIPAA audit trail | Existing `audit_logs` table + document access logging in server actions |
| Service key exposure | All storage operations via server-side Supabase client only |

**HIPAA Note**: Full HIPAA compliance requires Supabase Team/Enterprise plan with signed BAA. The architecture is HIPAA-ready; the plan upgrade is a business decision.

## Architecture Insights

- **All data flows through server actions** — no API routes exist. The upload flow should follow this pattern (server actions for URL generation and metadata, client-side only for the actual binary upload).
- **Case-centric storage paths** (`cases/{case_id}/...`) align with the existing case-centric architecture.
- **Existing document UI components** have filter/search/badge infrastructure that only needs upload/preview/download wiring.
- **Soft delete pattern** already exists on the `documents` table — document "deletion" should update `deleted_at`, not remove from storage (HIPAA retention).
- **Error handling pattern**: All actions return `{ error: string }` on failure, never throw. Upload actions should follow this.

## Decisions Made

1. **Versioning scope**: Minimal — "replace file" only, no version history UI. Old files retained in storage for HIPAA. No schema changes needed.
2. **Multi-file upload**: Yes — upload dialog supports multiple files at once with per-file document type selection.
3. **Upload progress**: Real progress via `tus-js-client` resumable uploads to Supabase's TUS endpoint.
4. **HIPAA plan**: Free tier for dev, production will be HIPAA-compliant (Team/Enterprise with BAA). Architecture is HIPAA-ready now; no dev-time compromises needed.
5. **Document type picker**: Upload picker shows only MRI Report, Chiropractor Report, Other (matching story AC). `generated` is reserved for system-created documents in future stories and excluded from the manual upload UI.
6. **Review workflow**: All uploads default to `pending_review`. Review toggle deferred to a future story.
